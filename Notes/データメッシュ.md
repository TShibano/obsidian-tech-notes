## 概要

データメッシュ（Data Mesh）は、Zhamak Dehghani が 2019年に提唱した分散型データアーキテクチャパターン。従来の中央集権型データプラットフォームの課題を解消するため、データの所有権・管理責任をドメイン（事業部門）に分散させ、各ドメインが「データをプロダクトとして提供」する考え方に基づく。技術的なアーキテクチャであると同時に、組織的・運用的な変革を伴う概念である。

## 詳細

### 誕生の背景：中央集権型の限界

従来のデータプラットフォーム（中央集権型[[データレイク]]・[[データウェアハウス]]）には以下の課題があった：

- **ボトルネック**: データエンジニアリングチームが全ドメインのデータパイプラインを一手に担い、要求に追いつけない
- **所有権の分離**: データを生み出す部門とデータを管理する部門が異なるため、品質・文脈の理解が不十分
- **スケーラビリティの欠如**: 組織規模が大きくなるほど中央チームの負荷が増大し、俊敏性が失われる

### 4つの原則

データメッシュは以下の4つの原則で構成される：

#### 1. ドメイン指向の分散型データ所有権（Domain-Oriented Decentralized Data Ownership）

データを生み出す各ドメイン（営業・マーケティング・物流など）が、そのデータの所有権と管理責任を持つ。クロスファンクショナルなドメインチーム（ドメインオーナー・データエンジニア・インフラエンジニア）が自律的にデータを管理する。

#### 2. プロダクトとしてのデータ（Data as a Product）

各ドメインは自分のデータを「プロダクト」として扱い、内部利用だけでなく他ドメインへの提供を前提に品質・インターフェース・SLA を整備する。

- **発見可能（Discoverable）**: [[データカタログ]]等で検索・発見できる
- **理解可能（Understandable）**: スキーマ・意味・品質が明確に文書化されている
- **信頼できる（Trustworthy）**: データ品質 SLA が定義されている
- **自己記述的（Self-describing）**: メタデータが豊富
- **相互運用可能（Interoperable）**: 共通フォーマット・プロトコルに準拠
- **安全（Secure）**: アクセス制御が適切に設定されている

#### 3. セルフサービス型データインフラプラットフォーム（Self-Serve Data Infrastructure as a Platform）

各ドメインが中央のデータエンジニアリングチームに依存せず、自律的にデータパイプラインを構築・運用できるプラットフォームを提供する。ツール・テンプレート・ベストプラクティスを共通基盤として整備することで、ドメインチームの生産性を高める。

#### 4. 連邦型計算ガバナンス（Federated Computational Governance）

全社共通のルール（データ品質基準・セキュリティポリシー・プライバシー規制準拠）と、各ドメインの自律性を両立させる。コンピューテーション（自動化ポリシー適用・データカタログ連携など）によりガバナンスを機械的に実施する。

### アーキテクチャ構成

```
┌─────────────────────────────────────────────────────┐
│           セルフサービスデータプラットフォーム             │
│  (インフラ・ツール・テンプレートの共通基盤)                │
└───────────────────┬─────────────────────────────────┘
                    │
    ┌───────────────┼───────────────┐
    ▼               ▼               ▼
┌───────┐      ┌───────┐      ┌───────┐
│ドメインA│      │ドメインB│      │ドメインC│
│ データ │  ──▶ │ データ │  ──▶ │ データ │
│プロダクト│      │プロダクト│      │プロダクト│
└───────┘      └───────┘      └───────┘
    │               │               │
    └───────────────┴───────────────┘
                    │
            連邦型ガバナンス
         (共通ポリシー・標準の適用)
```

### データメッシュ vs データファブリック

| 観点 | データメッシュ | [[データファブリック]] |
|------|-------------|-------------|
| **アプローチ** | 組織・運用モデルの変革 | 技術的な統合レイヤーの構築 |
| **所有権** | ドメインに分散 | 中央集権的（または仮想統合） |
| **主な課題解決** | スケーラビリティ・俊敏性 | データサイロの統合 |
| **適合組織** | 大規模・複数ドメインを持つ組織 | 既存システムが複雑で分散している組織 |
| **実装の複雑さ** | 組織変革を伴うため難易度が高い | 技術的な統合が主であり導入しやすい場合も |

### クラウドプロバイダの対応

| プロバイダ | ソリューション |
|-----------|-------------|
| **AWS** | AWS Lake Formation + Glue Data Catalog + Event Bridge でドメイン間データ共有 |
| **Azure** | Azure Purview（データガバナンス）+ Synapse Analytics でメッシュ構成 |
| **GCP** | Dataplex（データメッシュ専用サービス）でドメインデータの分散管理を支援 |
| **Databricks** | Unity Catalog によるマルチワークスペースのデータ共有 |

### 実装上の課題

- **組織変革の難しさ**: 技術より人・プロセスの変革のほうが困難
- **ドメイン境界の定義**: 適切なドメイン分割には深いビジネス理解が必要
- **データ品質の分散**: 各ドメインが品質責任を持つが、全社品質の担保が難しい
- **ガバナンスの実装**: 連邦型ガバナンスを自動化・計算可能にするツール整備が必要

## ポイント

- 中央集権型データプラットフォームのスケーラビリティ問題を、ドメイン分散で解決するアーキテクチャパターン
- 4原則：ドメインオーナーシップ・データ・アズ・ア・プロダクト・セルフサービス基盤・連邦ガバナンス
- 技術アーキテクチャより「組織変革」の側面が強く、大規模組織に適している
- [[データファブリック]]が「技術的な統合」を重視するのに対し、データメッシュは「組織論的なアプローチ」
- GCP Dataplex・Azure Purview・AWS Lake Formation など主要クラウドがネイティブ対応
