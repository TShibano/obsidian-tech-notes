## 概要

データリネージ（Data Lineage）とは、データがどのソースから生成され、どのような変換を経て、どの下流システムやレポートに到達したかを追跡・記録する仕組み。データの「血統・系譜」を可視化することで、問題の根本原因特定・影響範囲の把握・コンプライアンス対応を支援する。

## 詳細

### データリネージが必要な理由

- **問題の根本原因特定**: レポートの数値がおかしい場合、どのパイプラインのどのステップで問題が発生したかを追跡できる
- **影響範囲の把握**: 上流のデータソースを変更したとき、どの下流テーブル・ダッシュボードが影響を受けるかを事前に把握できる
- **コンプライアンス**: GDPR等の規制対応で「このPIIデータがどこに流れているか」を証明できる
- **信頼性向上**: データの来歴が明確になることで、分析者がデータを信頼して使えるようになる

### リネージの種類

| 種別 | 説明 |
|------|------|
| **テーブルレベルリネージ** | テーブルAがテーブルBの元となっている、という粒度の追跡 |
| **カラムレベルリネージ** | テーブルAのカラムXがテーブルBのカラムYに変換された、という細かい粒度 |
| **ジョブリネージ** | どのジョブ（Spark ジョブ、Airflow タスク等）がデータを処理したかの記録 |

### OpenLineage

2020年に Astronomer（Airflow のホスティング企業）が提唱し、現在業界標準となったオープンなメタデータ・リネージ収集仕様。

**コアコンセプト:**
- **Job**: データを処理するタスク（Spark ジョブ、dbt モデル、Airflow タスク等）
- **Run**: Job の1回の実行インスタンス
- **Dataset**: Job の入力・出力データセット

これら3者の関係を JSON イベントとして記録することで、ツール横断のリネージを統一的に把握できる。

```json
{
  "eventType": "COMPLETE",
  "job": { "name": "transform_orders", "namespace": "my_pipeline" },
  "run": { "runId": "uuid" },
  "inputs": [{ "name": "raw_orders", "namespace": "my_db" }],
  "outputs": [{ "name": "orders_summary", "namespace": "my_db" }]
}
```

**対応ツール:**
- Apache Airflow（2.7+でネイティブ対応）
- Apache Spark（OpenLineage Spark インテグレーション）
- dbt（dbt-ol プラグイン）
- Flink, dbt, Debezium なども対応中

### Marquez

OpenLineage のリファレンス実装となるOSSバックエンド。OpenLineage イベントを受け取り、保存・検索・可視化する。Airflow と組み合わせて使うことが多い。

### 商用ツール

| ツール | 特徴 |
|--------|------|
| **Atlan** | データカタログとリネージを統合したプラットフォーム |
| **DataHub（OSS）** | LinkedIn が開発したメタデータプラットフォーム。リネージ対応 |
| **Apache Atlas** | Hadoop エコシステムのメタデータ・ガバナンスツール |
| **Monte Carlo** | データオブザーバビリティに強みを持つ商用ツール |

### dbt とリネージ

dbt は `ref()` 関数による依存関係の宣言から自動的にリネージグラフを生成する。`dbt docs generate` で視覚的なリネージを確認できるため、SQL変換のリネージを無料・自動で得られる手段として広く使われる。

### データリネージの実装パターン

```
[ソースDB] → [Airbyte: Extract & Load] → [DWH: raw層]
                    ↓ (OpenLineage イベント)
[Marquez / DataHub] ← [Airflow: オーケストレーション]
                    ← [dbt: 変換] → [DWH: mart層]
                    ← [Spark: 集計]
```

## ポイント

- **自動収集 vs 手動記述**: OpenLineage を使えばツールが自動的にリネージを記録する。手動でメタデータを記述するアプローチは維持コストが高くなる
- **カラムレベルリネージ**: テーブルレベルより詳細で有用だが、実装コストが高い。dbt や一部のDWHツールがサポート
- **データカタログとの統合**: リネージはデータカタログと組み合わせることで価値が倍増する
- **インパクト分析**: 上流データの変更前に影響を受ける下流を事前に把握できる
