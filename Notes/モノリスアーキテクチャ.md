## 概要

モノリスアーキテクチャ（Monolithic Architecture）は、アプリケーションのすべての機能を単一のデプロイ可能なユニットとして構築するソフトウェアアーキテクチャパターン。UI・ビジネスロジック・データアクセスが1つのコードベースに統合され、1つのプロセスとして実行される。

## 詳細

### 構造

モノリスアプリケーションは一般的に以下のレイヤーで構成される:

```
┌─────────────────────────────────┐
│        プレゼンテーション層        │  UI / API エンドポイント
├─────────────────────────────────┤
│          ビジネスロジック層        │  ドメインロジック
├─────────────────────────────────┤
│         データアクセス層          │  DB クエリ / ORM
├─────────────────────────────────┤
│         単一データベース          │
└─────────────────────────────────┘
```

すべての機能が同じプロセス内で実行され、同一のデータベースを共有する。デプロイは常にアプリケーション全体が対象となる。

### メリット

- **開発の単純さ**: 単一コードベースのため、プロジェクト初期のセットアップが容易。IDE のリファクタリング機能もフルに活用可能
- **デプロイの容易さ**: 1つの成果物（JAR、バイナリ、コンテナイメージ等）をデプロイするだけで済む
- **テストの容易さ**: エンドツーエンドテストが単一プロセス内で完結し、サービス間通信のモッキングが不要
- **デバッグの容易さ**: スタックトレースが1プロセス内で完結する
- **パフォーマンス**: プロセス内関数呼び出しのため、ネットワーク通信のオーバーヘッドがない
- **トランザクション管理**: 単一データベースで ACID トランザクションを容易に実現可能

### デメリット

- **スケーラビリティの制約**: 特定の機能だけをスケールアウトできず、アプリケーション全体を複製する必要がある
- **デプロイリスク**: 小さな変更でもアプリケーション全体の再デプロイが必要。障害がシステム全体に波及する
- **技術的負債の蓄積**: コードベースの肥大化により、ビルド時間の増加・理解困難・変更の影響範囲が拡大
- **技術選択の硬直性**: フレームワーク・言語の変更がアプリケーション全体に影響する
- **チームのスケーリング困難**: 大規模チームでは同一コードベースでのコンフリクトが頻発する

### モジュラーモノリス

モノリスの利点を活かしつつ、内部構造を明確なモジュール境界で分割するアプローチ。[[マイクロサービスアーキテクチャ]]への移行前の中間段階としても有効。

```
┌──────────────────────────────────────┐
│           モジュラーモノリス            │
│  ┌──────┐  ┌──────┐  ┌──────┐       │
│  │ 注文  │  │ 在庫  │  │ 決済  │      │
│  │モジュール│ │モジュール│ │モジュール│  │
│  └──┬───┘  └──┬───┘  └──┬───┘       │
│     │  公開API経由の通信  │             │
│  ┌──┴──────────┴────────┴──┐         │
│  │       共有データベース      │        │
│  └─────────────────────────┘         │
└──────────────────────────────────────┘
```

**モジュラーモノリスのルール:**
- 各モジュールは公開インターフェースのみを通じて通信（直接のDB参照禁止）
- モジュール間の依存方向を制限（循環依存を防止）
- 将来的にモジュール単位でサービスとして切り出し可能な構造を維持

### マイクロサービスへの移行パターン

モノリスが限界に達した場合、段階的に[[マイクロサービスアーキテクチャ]]へ移行する方法がある:

1. **Strangler Fig パターン**: ルーティング層を前段に配置し、新機能を新サービスで実装しつつ、既存機能を段階的に移行。60% 以上の企業がこのパターンを採用
2. **モジュラーモノリス経由**: まずモノリスをモジュール分割し、境界を実証してからサービスとして切り出す
3. **Branch by Abstraction**: インターフェースを挟んで実装を段階的に置き換える

### モノリスが適する場面

- プロジェクト初期段階・MVP 開発（ドメイン境界が不明確な時期）
- 小規模チーム（1〜2チーム程度）
- トランザクション整合性が重要なシステム
- 分散システムの運用コストが見合わない場合

## ポイント

- モノリスは「悪」ではなく、多くの場面で合理的な選択肢
- 初期段階では「モノリスファースト」アプローチが推奨される（Martin Fowler）
- モジュラーモノリスは[[疎結合]]の原則をモノリス内部に適用した現代的なアプローチ
- マイクロサービスへの移行は Strangler Fig パターンによる段階的移行が主流
- スケーラビリティ・チーム規模・デプロイ頻度の要件に応じて判断する
