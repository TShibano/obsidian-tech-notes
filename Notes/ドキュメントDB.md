## 概要

ドキュメントDB（Document Database）は，データを JSON / BSON / XML などの半構造化ドキュメント形式で格納する NoSQL データベースである．各ドキュメントが独立したスキーマを持てるため，スキーマレスで柔軟なデータ構造の管理に適しており，Web アプリケーションやコンテンツ管理に広く利用される．

## 詳細

### RDBMS との比較

| 概念 | RDBMS | ドキュメントDB |
|------|-------|-------------|
| データ単位 | 行（Row） | ドキュメント（JSON 等） |
| スキーマ | 固定（DDL で定義） | 柔軟（ドキュメントごとに異なる） |
| 格納形式 | テーブル（行×列） | コレクション（ドキュメントの集合） |
| 結合 | JOIN で複数テーブルを結合 | ネスト（埋め込み）または参照 |
| スケール | 垂直スケール中心 | 水平スケール（シャーディング） |
| ACID | 完全対応 | コレクション単位（トランザクションは制限あり） |

### ドキュメント構造の例

```json
{
  "_id": "order_001",
  "customer": {
    "name": "田中太郎",
    "email": "tanaka@example.com"
  },
  "items": [
    { "product": "ノートPC", "price": 120000, "qty": 1 },
    { "product": "マウス", "price": 3000, "qty": 2 }
  ],
  "total": 126000,
  "status": "shipped",
  "created_at": "2026-05-05T09:00:00Z"
}
```

関連データをネストで埋め込むことで，1回のクエリで完全な情報を取得できる（JOIN 不要）．

### 主要なドキュメントDB

| 製品 | 特徴 |
|------|------|
| **MongoDB** | 最も普及したドキュメント DB．豊富なエコシステム・BSON 形式 |
| **Amazon DocumentDB** | MongoDB 互換のフルマネージドサービス（AWS） |
| **Firestore** | Google Cloud のサーバーレス ドキュメント DB．リアルタイム同期 |
| **CouchDB** | HTTP API ベース．オフライン同期（PouchDB との連携）に強い |
| **[[ArangoDB]]** | ドキュメント＋グラフ＋KV のマルチモデル |
| **[[SurrealDB]]** | Rust 製マルチモデル．ドキュメント＋グラフ＋KV＋ACID |

### ユースケース

- **商品カタログ**: 商品カテゴリごとに属性が異なるデータをスキーマレスで管理
- **ユーザープロファイル**: ユーザーごとに持つ属性が異なる可変データ
- **コンテンツ管理（CMS）**: 記事・ページなど構造が多様なコンテンツ
- **IoT データ収集**: デバイスごとに送信フィールドが異なるセンサーデータ
- **ログ蓄積**: アプリケーションログのフレキシブルな蓄積・検索

### 注意点

- JOIN がないため，「複数コレクションにまたがる複雑な集計」は苦手
- スキーマがないため，データ品質の担保はアプリケーション側の責任
- MongoDB 4.0 以降でマルチドキュメントトランザクションを提供（ただし性能コストあり）

## ポイント

- 「スキーマの柔軟性」と「ネスト構造によるデータ局所性」がドキュメントDBの最大の強み
- スキーマが安定しているトランザクションデータには RDBMS の方が適する
- [[ポリグロット永続化]] の典型的な使い分け：RDBMS（注文・決済）＋ドキュメントDB（商品カタログ・プロファイル）
- MongoDB が事実上の標準だが，クラウドマネージドサービス（DocumentDB，Firestore）の採用も増加
