## 概要

データボルト（Data Vault）は Dan Linstedt が 1990 年代に開発し 2000 年に発表したデータウェアハウスのモデリング手法．**Hub**（ビジネスキー）・**Link**（関係）・**Satellite**（属性・履歴）の3要素で構成され，[[Inmonのトップダウン]] の整合性と [[Kimballのボトムアップ]] のアジャイルな拡張性を組み合わせた設計を目指す．監査証跡・履歴管理・変化への対応に優れ，2026 年現在 Snowflake・Databricks・Azure Synapse 上での採用が増加しつつあるが，まだ広く普及していない．デメリットとしては，複雑さ・データ重複によるコストの増加・クエリパフォーマンスの低下などが挙げられる．

## 詳細

### 3コンポーネント
#### Hub（ハブ）

ビジネスキー（業務的に一意な識別子）を管理する．顧客ID・注文番号・商品コードなどが例．重複なく，変化しないビジネスの核心概念を表す．

#### Link（リンク）

Hub 間の関係（アソシエーション）を管理する．「顧客 × 商品」「注文 × 配送」のような多対多の関係を表現する．

#### Satellite（サテライト）

Hub または Link の属性・履歴データを管理する．変更のたびに新しいレコードが挿入され（INSERT ONLY），更新・削除を行わないため**完全な変更履歴**が保持される．

```
Hub（ビジネスキー）
├─ Hash Key（PK）
├─ Business Key（例: 顧客ID）
├─ Load Date
└─ Record Source

Link（関係）
├─ Hash Key（PK）
├─ Hub A の Hash Key（FK）
├─ Hub B の Hash Key（FK）
├─ Load Date
└─ Record Source

Satellite（属性・履歴）
├─ Hub or Link の Hash Key（FK）
├─ Load Date（PK の一部）
├─ Hash Diff（変更検出）
├─ 属性カラム（名前・住所・ステータス等）
└─ Record Source
```


### Data Vault 2.0

Dan Linstedt が改訂した最新バージョン（2013年〜）．クラウド DWH・MPP アーキテクチャへの対応が強化された．

| Data Vault 1.0 → 2.0 の主要改善点            |
| -------------------------------------- |
| ハッシュキー採用（自然キーの代わりに SHA-256 等を使用）       |
| パラレルロード対応（MPP でハブ・リンク・サテライトを並列 INSERT） |
| Business Vault（ビジネスルールを適用した派生レイヤー）の追加  |
| Raw Vault + Business Vault の2層設計       |

### アーキテクチャ全体像

```
ソースシステム
     ↓ ETL/ELT
Raw Vault（Staging）
     ↓
Data Vault（Hub / Link / Satellite）
     ↓
Business Vault（ビジネスルール適用）
     ↓
Information Mart（スタースキーマ等 - レポーティング用）
```

### Kimball / Inmon との比較

| 観点 | データボルト | [[Kimballのボトムアップ]] | [[Inmonのトップダウン]] |
|------|------------|----------------------|----------------------|
| 設計構成 | Hub/Link/Sat | スタースキーマ | 3NF |
| 変更への対応 | 高い（追加のみ） | 中程度 | 低い（EDW 再設計が波及）|
| 履歴管理 | INSERT ONLY で完全 | SCD（緩やかに変化するディメンション）| 設計次第 |
| クエリ性能 | 低め（ジョイン多） | 高い | 中程度 |
| 監査対応 | 優れる（全履歴保持）| 中程度 | 優れる（3NF の整合性）|
| 学習コスト | 高い | 低〜中 | 高い |
| 2026 年採用 | 増加中（Snowflake/Databricks 上）| 主流 | エンタープライズ |

### 2026 年の動向

- Snowflake・Databricks・Azure Synapse でのクラウドネイティブな Data Vault 実装が増加
- Data Vault 2.0 のハッシュキー採用により，MPP クラスタでの並列ロードが大幅効率化
- dbt と組み合わせた ELT ベースの Data Vault パイプライン構築が普及
- 新規ソース統合時間を従来手法比 30〜40% 短縮できる事例が報告されている

## ポイント

- Hub（何）・Link（関係）・Satellite（属性・履歴）の3構造でデータを分解・整理する
- INSERT ONLY で完全な変更履歴を保持するため，監査証跡・コンプライアンス対応に強い
- ソースシステムの追加・変更が既存モデルに影響しにくく，拡張性が高い
- ハッシュキー採用（Data Vault 2.0）により MPP での並列ロードが可能
- クエリ性能は低め（ジョイン多）のため，レポーティング用に Information Mart（スタースキーマ）を別途作成する
