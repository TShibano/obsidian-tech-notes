## 概要

メダリオンアーキテクチャ（Medallion Architecture）は、データレイクハウスにおけるデータ設計パターンで、Bronze（生データ）→ Silver（検証済み）→ Gold（ビジネス活用）の3層でデータ品質を段階的に向上させる。Databricks が提唱し、Delta Lake と組み合わせて広く採用されている。

## 詳細

### 3層構造

#### Bronze 層（Raw）

- データソースから取り込んだ生データをそのまま保持する層
- 最小限のバリデーションのみ実施し、スキーマ変更への耐性のためフィールドを string / VARIANT / binary で保存することが推奨される
- 追記型（append-only）で増加し続け、全履歴を保持
- 監査・再処理のための Single Source of Truth として機能

#### Silver 層（Validated）

- データのクリーニング・検証を行う層
- null 値の除去、不正レコードの隔離（quarantine）、型変換、重複排除を実施
- 異なるソースからのデータを統合し、エンタープライズビューを構築
- アドホック分析・高度な分析・ML 向けのセルフサービス分析を実現
- Databricks は Bronze/Silver テーブルからの読み取り→Silver テーブルへの書き込みを推奨し、インジェストから直接 Silver への書き込みは非推奨

#### Gold 層（Enriched）

- ビジネスレベルの集約、スタースキーマ、特徴量テーブルを生成する層
- BI ダッシュボード、ML 学習・推論、SLA 付きエンドユーザークエリに最適化
- Delta Lake の Liquid Clustering によるデータ配置最適化で低レイテンシを実現

### Delta Lake との関係

Delta Lake はメダリオンアーキテクチャの基盤となるオープンテーブルフォーマット:

- ACID トランザクションによる信頼性の高いパイプライン構築
- スキーマ進化・タイムトラベル・MERGE 操作のサポート
- 小規模〜大規模データセットの両方に対応
- Managed テーブルの使用が推奨（Bronze/Silver/Gold すべて）

### 批判と課題

- **遅延**: 段階的精製プロセスにより、有用なインサイトへのアクセスが遅れる
- **運用負荷**: 3層フルセットアップは運用コストが予想以上に膨らむ場合がある
- **Pull 型の制約**: 下流のデータ消費者が ETL ジョブを作成してデータを引き込む必要があり、受動的な立場に置かれる
- **コスト**: データの重複保持（Bronze + Silver + Gold）によるストレージコスト増

### 代替・補完アプローチ

| アプローチ | 特徴 |
|---|---|
| **Kappa Architecture** | バッチ層を廃しリアルタイムストリーム単一パイプラインで処理。シンプルだがリプレイに課題 |
| **Lambda Architecture** | バッチ＋リアルタイムの2系統。網羅性と即時性の両立だが運用が複雑 |
| **Data Vault** | Hub/Link/Satellite 構造で変更履歴に強い。監査要件の厳しい領域向け |
| **[[データメッシュ]]** | ドメイン所有権と連邦型ガバナンス。メダリオンとの組み合わせ（ハイブリッド）も有効 |

2026年現在、重要なビジネスプロセスや AI ワークロードにはメダリオンを、ドメイン所有権や連邦ガバナンスにはデータメッシュを適用するハイブリッドアプローチが注目されている。

## ポイント

- Bronze → Silver → Gold の3層でデータ品質を段階的に向上させるシンプルで理解しやすいパターン
- Databricks / Delta Lake エコシステムとの親和性が高く、Managed テーブル + Liquid Clustering が推奨構成
- Bronze 層は監査・再処理の観点から全履歴を保持し、Single Source of Truth として機能
- 3層フルセットアップのコスト・遅延・運用負荷は事前に検討が必要
- データメッシュや Kappa Architecture との組み合わせで弱点を補完できる
