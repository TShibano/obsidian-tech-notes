## 概要

リレーショナルデータウェアハウス（Relational Data Warehouse）とは，リレーショナルデータベース技術を基盤とし，構造化データをテーブル形式で格納・管理する従来型のデータウェアハウスアーキテクチャ．SQL によるクエリと OLAP（オンライン分析処理）に最適化されており，ディメンショナルモデリング（スタースキーマ・スノーフレークスキーマ）を用いたデータ構造が特徴．リレーショナルデータベースのメリットを得ることができる一方，非構造化データ等に対応できないため，データレイクと組み合わせた[[モダンデータウェアハウス]]や[[データファブリック]]等で活用される．

## 詳細

### 開発方針
トップダウンアプローチで，開発を進めるのが良いとされる．これはレポートを作成する前の準備を事前にすべて行うことである．
手順
1. データで知りたい仮説を立案する
2. ビジネス要件を定義する
3. データウェアハウスのアーキテクチャを設計する
4. データモデルを開発する
5. データアーキテクチャを構築する
6. [[ETL]]を開発する
7. BIツールやアプリケーションの開発と導入を行う
8. データウェアハウスをテストし，改良する
9. データウェアハウスを運用・拡張する

### 採用するメリット

- 本番システムの負荷を軽減する
	- 本番DBからデータをコピーして扱うため，本番システムへの影響を最小限にできる．
- 読み出しアクセスを最適化する
	- 通常のRDBは書き込みに最適化されている．データ分析には，読み出し処理が重要．
- 複数のデータソースを統合する
	- 一元的にデータを集約でき，より価値の高いレポートを容易に作成できる
- 正確な履歴レポートを発行する
	- 変更履歴も把握できるため，いつ実行しても正確なレポートを作成できる
- テーブルを再構築し，わかりやすい名前に変更する
	- アプリケーションDBではわかりにくい名前(列名など)が使われていることも多い
- アプリケーションのアップグレードを守る
	- 元のDBが変更された場合，ETLだけ変更すれば良い
- セキュリティ上の問題を緩和する
	- RDBはアクセス権の付与などを厳格にできる
- 履歴データを保持する
	- オペレーショナルDBでは，保存期間が決められていることが多いが，RDWは通常過去すべてのデータを保持する
- マスターデータ管理
	- MDMによりデータを正しくクレンジングすることができる
	- 詳細はこちら[[MDM]]
- ソースシステムのデータの穴を修正し，データ品質を向上させる
	- ETL時にデータの検証を行うことができ，誤ったデータによる誤った分析結果を出すことを防げる
- レポート作成においてIT部門の関与を不要にする
	- データがRDBで一元管理されているため，ユーザが簡単にレポートを作成できる

### 制約と課題
- 複雑さ
	- 設計・構築・保守に専門的なスキルとリソースが必要になる
- 高コスト
	- ハードウェア・ソフトウェア・人材への投資が必要
- データ統合の問題
	- 複数のデータを統合する際に，データ形式・構造・品質などの違いに起因するデータの前処理に時間と労力がかかる
- 構造化データのみ
	- JSON，画像，ログなどの非構造化・半構造化データの扱いが苦手
- ETLのボトルネック・レイテンシの発生
	- 大量データの変換処理がデータ鮮度の低下を招く
- メンテナンスウィンドウ
	- データの読み込みや前処理に時間がかかるため，メンテナンス時間を長めに見積もる必要がある
- スキーマ変更の硬直性
	- 事前定義スキーマのため，ビジネス要件の変化への対応が遅い
- スケーラビリティ
	- オンプレミス環境ではストレージとコンピュートの独立スケーリングが困難


これらの課題を解決するために，[[モダンデータウェアハウス]] や [[データレイクハウス]] といったアーキテクチャが登場している．


### 基本アーキテクチャ

リレーショナルデータウェアハウスは RDBMS 上に構築され，以下の層で構成される：

| 層 | 役割 |
|----|------|
| **データソース層** | 業務システム（ERP，CRM 等）からデータを抽出 |
| **ステージング層** | 抽出データの一時保管・クレンジング |
| **DWH 層** | 統合・変換されたデータの格納．Inmon/Kimball アプローチで構築 |
| **データマート層** | 部門・用途別の集計済みデータ |
| **プレゼンテーション層** | BI ツール・レポートへのデータ提供 |

データの流れは ETL（Extract → Transform → Load）が基本で，ソースシステムから抽出したデータを変換してから DWH にロードする．
データの抽出頻度は，エンドユーザの用途とデータサイズで決定する．
抽出方法には，フル抽出(毎回前データを取得)と増分抽出(前回からの変更分だけ取得)がある．前回からの変更分を取得する方法として，タイムスタンプや変更データキャプチャ(CDC)，パーティション化(日付をキーとしてテーブルを分割すること)，データベーストリガー，MERGE文などがある．
また，オンライン抽出(データソースに直接接続してデータ取得)とオフライン抽出(データソースが別のところにファイルをはき出してそれを読み込む)の2種類ある．

### 設計手法: Inmon vs Kimball

リレーショナル DWH の設計には2つの代表的なアプローチがある：

#### Inmon アプローチ（トップダウン）

[[Inmonのトップダウン]]

Bill Inmon が提唱．正規化された統合的なエンタープライズ DWH を先に構築し，そこからデータマートを派生させる．

- **データモデル**: 第3正規形（3NF）で正規化
- **設計思想**: 企業全体のデータを統合した「Single Source of Truth」を構築
- **メリット**: データの一貫性・整合性が高い．冗長性が低くストレージ効率が良い
- **デメリット**: 初期構築に時間がかかる．ジョインが多く直接的なクエリ性能はやや低い

#### Kimball アプローチ（ボトムアップ）

[[Kimballのボトムアップ]]

Ralph Kimball が提唱．ビジネスプロセスごとにディメンショナルモデルのデータマートを構築し，共有ディメンション（Conformed Dimensions）で統合する．

- **データモデル**: スタースキーマ（非正規化）
- **設計思想**: ビジネスユーザーが理解しやすい分析モデルを優先
- **メリット**: クエリが高速．段階的な構築が可能．理解しやすい
- **デメリット**: データの冗長性が高い．全社統合に共有ディメンションの設計が必要

### ディメンショナルモデリング

リレーショナル DWH の中核的な設計手法で，データを「ファクト」と「ディメンション」に分離する．[[データウェアハウス]]のスタースキーマの項も参照．

#### スタースキーマ

```
          ┌──────────┐
          │ 日付 DIM  │
          └────┬─────┘
               │
┌──────────┐   │   ┌──────────┐
│ 製品 DIM  ├───┼───┤ 店舗 DIM  │
└──────────┘   │   └──────────┘
          ┌────┴─────┐
          │ 売上 FACT │
          └────┬─────┘
               │
          ┌────┴─────┐
          │ 顧客 DIM  │
          └──────────┘
```

- **ファクトテーブル**: ビジネスイベントの計測値（売上金額，数量等）と各ディメンションへの外部キーを持つ
- **ディメンションテーブル**: 分析軸の属性情報を非正規化して格納（日付，顧客，製品等）
- RDBMS 上で特に高いパフォーマンスを発揮する（スタージョイン最適化）

#### スノーフレークスキーマ

ディメンションテーブルをさらに正規化し，階層構造を別テーブルに分離した構造．

- **例**: 製品ディメンション → 製品テーブル + カテゴリテーブル + 部門テーブル
- **メリット**: ストレージ効率が高い．ディメンション更新時の整合性が保ちやすい
- **デメリット**: ジョインが増加し，クエリが複雑化・低速化する傾向

### OLAP 処理

リレーショナル DWH は OLAP（Online Analytical Processing）に最適化されており，以下の操作をサポートする：

| OLAP 操作 | 説明 |
|-----------|------|
| **スライス** | 1つのディメンションで値を固定して断面を取得 |
| **ダイス** | 複数ディメンションで部分集合を抽出 |
| **ドリルダウン/アップ** | 詳細レベルと集約レベルを行き来 |
| **ピボット** | 分析軸を回転させて異なる視点でデータを表示 |
| **ロールアップ** | データを上位の集計レベルにまとめる |

### 代表的な製品

#### オンプレミス

| 製品 | 特徴 |
|------|------|
| **Teradata** | 大規模エンタープライズ向け．MPP（超並列処理）アーキテクチャ |
| **Oracle Exadata** | Oracle DB ベース．ハイブリッドクラウド対応 |
| **IBM Db2 Warehouse** | AI 統合．マルチプラットフォーム対応 |
| **Microsoft SQL Server** | Analysis Services による OLAP 機能 |

#### クラウド

クラウドベースのリレーショナル DWH は [[モダンデータウェアハウス]] として進化しており，[[Snowflake]]，BigQuery，Amazon Redshift 等が代表的．

## ポイント

- RDBMS 上に構築され，SQL と OLAP に最適化された分析基盤
- Inmon（正規化・トップダウン）と Kimball（ディメンショナル・ボトムアップ）の2大設計手法がある
- スタースキーマが RDBMS 上で最も高いクエリ性能を発揮する
- 構造化データの分析には依然として強力だが，非構造化データや即時性の要求には限界がある
- クラウド化・レイクハウスとの融合により[[モダンデータウェアハウス]]へ進化している
