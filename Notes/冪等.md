## 概要

冪等（idempotent）とは，同じ操作を何度繰り返しても結果が変わらない性質．分散システムやネットワーク通信ではリトライが不可避なため，冪等な設計が信頼性の基盤となる．

## 詳細

### HTTP メソッドと冪等性

| メソッド | 冪等 | 安全（副作用なし） |
|---------|------|----------------|
| GET | YES | YES |
| HEAD | YES | YES |
| PUT | YES | NO |
| DELETE | YES | NO |
| POST | NO | NO |
| PATCH | NO（原則） | NO |

- **安全**（Safe）= 副作用がない（読み取りのみ）
- **冪等**（Idempotent）= 同じ操作を繰り返しても状態が変わらない

### 実現パターン

**1. 冪等キー（Idempotency Key）**
クライアントがリクエストに一意のキー（UUID 等）を付与し，サーバがキーに基づいて重複を排除する．Stripe の Payment API 等で採用．

**2. 自然な冪等設計**
「残高を 100 円にする」（PUT）は冪等だが，「100 円引く」（PATCH/POST）は冪等でない．状態の絶対値を指定する設計が冪等になりやすい．

**3. 条件付き更新**
`If-Match` ヘッダや楽観的ロックで，意図した状態の場合のみ更新を実行する．

### 分散システムでの重要性

- ネットワーク障害でリトライが発生した場合，非冪等な操作（注文作成等）が二重実行される
- [[Webhook]] でも同一イベントが複数回送られる可能性があるため，受信側で冪等処理が必要

## ポイント

- リトライを前提とする分散システム設計の必須概念
- [[REST API]] の GET, PUT, DELETE は冪等；POST は冪等でない
- 冪等でない操作を安全にリトライさせるには冪等キーを使う
