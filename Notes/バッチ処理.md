## 概要

バッチ処理（Batch Processing）は，一定期間に蓄積されたデータを決められたタイミングでまとめて処理する方式である．個々のイベントをその場で処理するリアルタイム処理とは対照的に，大量データを効率よく一括処理することが特徴で，データウェアハウスへの日次データ投入や帳票生成などに広く用いられる．

## 詳細

### リアルタイム処理との比較

| 観点 | バッチ処理 | リアルタイム処理（ストリーミング） |
|------|----------|---------------------------------|
| 処理タイミング | 定時（日次，時次など） | データ到着次第，即時 |
| レイテンシ | 分〜時間 | ミリ秒〜秒 |
| スループット | 高い（大量一括） | 低〜中（逐次処理） |
| 実装コスト | 低い | 高い |
| 向いているデータ | 大量履歴データ，集計 | センサー，ログ，金融取引 |
| 主なフレームワーク | Apache Spark，Hadoop | Apache Kafka，Flink |
| 障害復旧 | 容易（再実行） | 複雑（状態管理が必要） |

### 処理の種類

| 種類 | 処理間隔 | 説明 |
|------|---------|------|
| **オフラインバッチ** | 日次，週次，月次 | 夜間バッチ，月次集計，帳票 |
| **マイクロバッチ** | 数秒〜数分 | バッチとリアルタイムの中間．Spark Streaming が採用 |
| **バルク処理** | 不定期 | 初期ロード，データ移行，大規模集計 |

### バッチ処理の典型的なパターン

```
[データソース群]
        ↓ 抽出（Extract）
[ステージングエリア]
        ↓ 変換・クレンジング（Transform）
[DWH / データレイク]
        ↓ 集計・加工（dbt 等）
[データマート]
        ↓
[BI ツール / レポート]
```

### 主要フレームワーク・ツール

| ツール | 特徴 |
|-------|------|
| **Apache Spark** | 大規模分散バッチ処理の標準．インメモリ処理で高速 |
| **Apache Hadoop（MapReduce）** | 初期の大規模分散バッチ基盤．現在は Spark に移行が進む |
| **Apache Airflow** | バッチワークフローのオーケストレーションツール |
| **dbt** | DWH 上での SQL バッチ変換 |
| **AWS Glue / Azure Data Factory** | クラウドネイティブな ETL/バッチサービス |

### [[Lambdaアーキテクチャ]] での位置づけ

Lambda アーキテクチャではバッチ処理と[[リアルタイム処理]]を組み合わせる:
- **バッチ層**: 全履歴データを高精度に処理（遅いが正確）
- **スピード層（リアルタイム）**: 最新データを低遅延で処理（速いが近似）
- **サービング層**: 両者の結果をマージして提供

## ポイント

- 大量データの処理コスト効率はバッチ処理が優れる（スループット重視）
- 実装・運用の複雑さが低く，初期構築コストが抑えられる
- 日次 DWH 更新，月次財務集計，機械学習の定期バッチ学習などに適する
- リアルタイム性が求められる場面では [[リアルタイム処理]] と組み合わせる
- クラウドでは需要に応じたスケールアップ/ダウンが可能（コスト最適化）
