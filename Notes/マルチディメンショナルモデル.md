## 概要

マルチディメンショナルモデル（Multidimensional Model）は，データウェアハウスや OLAP システムにおけるデータ設計手法で，ビジネス指標（メジャー）と分析軸（ディメンション）を明確に分離してデータを表現する．Ralph Kimball が体系化したディメンショナルモデリングが代表的な実装手法であり，スタースキーマとスノーフレークスキーマが主要なデータモデルである．非正規化によりテーブルの結合回数が減ることで，クエリの実行速度を大幅に向上する．

## 詳細

### 基本概念

マルチディメンショナルモデルは「何を」（メジャー）「どの切り口で」（ディメンション）分析するかを分離するデータ設計思想である．

| 要素                     | 説明                    | 例              |
| ---------------------- | --------------------- | -------------- |
| **ファクト（Fact）**         | 計測可能なビジネス指標．データの値を表す． | 売上金額，販売数量，利益   |
| **ディメンション（Dimension）** | 分析の切り口・属性．データの特性を表す．  | 時間，顧客，製品，地域    |
| **粒度（Grain）**          | ファクトテーブルの1行が表す最小単位    | 1件の取引，1日の売上サマリ |

### スタースキーマ（Star Schema）

最もシンプルで広く採用される多次元モデルの実装形式．中央のファクトテーブルを複数のディメンションテーブル(DIM)が星型に取り囲む構造．

```
          ┌──────────┐
          │ 日付 DIM  │
          └────┬─────┘
               │
┌──────────┐   │   ┌──────────┐
│ 製品 DIM  ├───┼───┤ 地域 DIM  │
└──────────┘   │   └──────────┘
          ┌────┴─────┐
          │ 売上 FACT │  ← メジャー（金額，数量等）
          └────┬─────┘     + 各 DIM への外部キー
               │
          ┌────┴─────┐
          │ 顧客 DIM  │
          └──────────┘
```

**特性:**
- ディメンションテーブルは非正規化（冗長データあり）
- ジョイン数が少なく，クエリが高速
- BI ツール・分析ユーザーにとって理解しやすい構造
- [[OLAP]] の5大操作（スライス・ダイス・ドリルダウン等）と直接対応

### スノーフレークスキーマ（Snowflake Schema）

スタースキーマのディメンションテーブルをさらに正規化した構造．ディメンションテーブルが階層的に展開され，雪の結晶状になる．

```
          日付 DIM ─ 月 DIM ─ 年 DIM
              │
          売上 FACT ─ 製品 DIM ─ カテゴリ DIM
              │
          顧客 DIM ─ 都道府県 DIM ─ 地方 DIM
```

**スタースキーマとの比較:**

| 項目 | スタースキーマ | スノーフレークスキーマ |
|------|--------------|----------------------|
| 正規化 | 非正規化（冗長あり） | 正規化（冗長なし） |
| クエリ速度 | 高速（ジョイン少） | 低速（ジョイン多） |
| ストレージ | 多い | 少ない |
| 保守性 | 変更がしやすい | ディメンション更新が整合しやすい |
| 主な用途 | 大多数の分析ワークロード | ストレージ効率が重要な場面 |

### ファクトテーブルの種類

| 種類 | 説明 | 例 |
|------|------|----|
| **トランザクションファクト** | 1件のビジネスイベント | 1回の注文，1回の支払 |
| **周期スナップショットファクト** | 定期的な状態のスナップショット | 月末在庫，週次KPI |
| **累積スナップショットファクト** | プロセスの各マイルストーン | 受注→出荷→請求→回収の状態 |

### ディメンションの特殊パターン

- **SCD（Slowly Changing Dimension; 緩慢変化ディメンション）**: 時間の経過とともに変化するディメンション属性の管理
	- Type1: 上書き．古い値は完全に消える．
	- Type2: 履歴追加．レコード(行)を追加して保持する
	- Type3: 旧値保持．同一レコードにフィールド(列)を追加して保持する
- **退化ディメンション（Degenerate Dimension）**: ファクトテーブル内に直接置かれるディメンション（注文番号など）
- **ジャンクディメンション**: フラグ・区分コードなど細かい属性をまとめたディメンション
- **ロールプレイングディメンション**: 同一ディメンションを複数の役割で参照（受注日・出荷日・入金日はすべて日付ディメンション）

### Kimball のボトムアップアプローチとの関係

[[Kimballのボトムアップ]] では，マルチディメンショナルモデル（スタースキーマ）を基礎とし，ビジネスプロセスごとにデータマートを構築する．共有ディメンション（Conformed Dimensions）によって複数のマートを統合し，全社的な分析を可能にする．

## ポイント

- ファクト（何を計測するか）とディメンション（どの軸で切るか）の分離がマルチディメンショナルモデルの本質
- スタースキーマが最も一般的な実装．クエリ速度・可読性に優れる
- スノーフレークスキーマはストレージ効率は高いが，ジョイン増加でクエリが複雑化
- [[キューブ]]（OLAP キューブ）はマルチディメンショナルモデルを物理的に実装したデータ構造
- クラウド DWH（Snowflake，BigQuery 等）でも引き続きスタースキーマが推奨設計
