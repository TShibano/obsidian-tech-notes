## 概要

要求分析（Requirements Analysis）とは、システム開発においてステークホルダーのニーズ・期待・制約を収集・整理・分析し、曖昧な「要求」を実装可能な「要件」へと精緻化するプロセスである。要件定義の前段工程に位置し、「何を作るべきか」の共通認識を形成することを目的とする。

## 詳細

### 要求と要件の違い

| 用語 | 定義 | 例 |
|------|------|-----|
| **要求（Requirements）** | ユーザーが望む状態・ニーズ（曖昧・自然言語） | 「もっと早く注文できるようにしたい」 |
| **要件（Specification）** | システムが満たすべき具体的な条件（明確・検証可能） | 「注文完了まで3ステップ以内に絞る」「1秒以内にレスポンスを返す」 |

要求分析は要求→要件への変換プロセスである。

### 要求の種類

| 種類 | 概要 |
|------|------|
| **機能要求** | システムが行うべき処理・動作 |
| **非機能要求** | 性能・可用性・セキュリティ等の品質要求 |
| **制約** | 技術・法律・予算・スケジュールによる制限 |
| **ビジネスルール** | 業務上の規則・ポリシー |

### 要求収集の技法

#### ヒアリング・インタビュー

ステークホルダーへの直接インタビュー。オープン質問（「現在の課題は？」）とクローズド質問（「〇〇機能は必要ですか？」）を組み合わせる。

#### ワークショップ（共同要求定義）

関係者が集まってリアルタイムで要求を議論・合意する手法。JAD（Joint Application Development）セッションが代表例。

#### プロトタイプ法

早期に画面モックアップや動くプロトタイプを作成し、ユーザーの反応から真の要求を引き出す。要求の曖昧さを早期に解消できる。

#### ユースケース分析

アクター（利用者・外部システム）とシステムのインタラクションを「ユースケース」として記述。業務フローからユースケースを抽出する手法（RDRA 等）が体系化されている。

```
ユースケースの基本形：
「[アクター] は [目的] を達成するために [システム] を使用して [操作] を行う」

例：「購入者は商品を購入するためにECサイトを使用してカートに追加・決済を行う」
```

#### ユーザーストーリー（アジャイル）

アジャイル開発で使われる軽量な要求記述形式。

```
形式：「[ユーザーの種類] として、[行いたいこと] がしたい。なぜなら [理由] だから」
例：「購入者として、注文履歴を確認したい。なぜなら過去の購入を参照したいから」
```

ユーザーストーリーはそれ自体が仕様ではなく、詳細を議論するための「会話の起点」として機能する。

### ユースケースとユーザーストーリーの比較

| 観点 | ユースケース | ユーザーストーリー |
|------|------------|-----------------|
| 詳細レベル | 詳細（シナリオ・代替フロー含む） | 簡潔（1〜3文） |
| 視点 | システムとアクターの相互作用 | ユーザーが得る価値 |
| 適した手法 | ウォーターフォール | アジャイル（スクラム） |
| 合意のタイミング | 前もって詳細化 | 実装直前に詳細化 |

### 要求分析のプロセス

1. **要求収集**: ヒアリング・ワークショップ・既存資料調査
2. **要求の整理**: 重複排除・矛盾解消・優先度付け（MoSCoW 法等）
3. **要求のモデル化**: ユースケース図・ER 図・アクティビティ図等で可視化
4. **要求の検証**: 「完全性・一貫性・実現可能性・テスト可能性」を確認
5. **要求の合意・承認**: ステークホルダー全員の承認を得る

### MoSCoW 法（優先度付け）

| 分類 | 意味 |
|------|------|
| **Must have** | 必須機能（これがなければリリースできない） |
| **Should have** | 重要だが必須ではない |
| **Could have** | あれば良いが優先度低 |
| **Won't have** | 今回のスコープ外（将来的に検討） |

## ポイント

- 要求分析の目的は「曖昧な要求を検証可能な要件へ変換すること」
- ユースケースは「何を・誰が・どう使うか」を定義する要件仕様、ユーザーストーリーは「価値と会話の起点」であり目的が異なる
- アジャイルでは要求を一度に詳細化せず、スプリントごとに段階的に具体化（ジャストインタイム）
- 要求の曖昧さ・矛盾・漏れを早期に発見するためにプロトタイプが有効
- MoSCoW 法で優先度を明示し、スコープのコントロールを行う
