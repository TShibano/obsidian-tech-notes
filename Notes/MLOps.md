## 概要

MLOps（Machine Learning Operations）は、機械学習モデルのライフサイクル全体を効率的かつ信頼性高く管理するための一連の実践・プロセス・ツールの総称である。DevOps の原則をデータエンジニアリングと機械学習に適用し、データ収集からモデルのデプロイ・監視・再学習までを継続的に自動化する。

## 詳細

### 背景：なぜ MLOps が必要か

機械学習モデルの開発と運用には、従来のソフトウェア開発とは異なる固有の課題がある:

| 課題 | 説明 |
|------|------|
| **再現性** | データ・コード・パラメータ・環境の組み合わせにより結果が変わるため、実験の再現が困難 |
| **データドリフト** | 本番データの分布が時間とともに変化し、モデル精度が劣化する |
| **技術的負債** | ML システムはモデルコード以外に、データパイプライン、特徴量管理、監視など膨大な周辺インフラを必要とする |
| **チーム間の分断** | データサイエンティストが開発したモデルをエンジニアが本番化する際に、手動の引き渡しプロセスがボトルネックになる |
| **スケーラビリティ** | 実験環境で動作するモデルが、本番環境の規模やレイテンシ要件を満たさない |

### MLOps のライフサイクルと各工程

MLOps は大きく3つのフェーズに分かれ、8つの工程で構成される:

#### フェーズ1: 設計

ML で解決すべきビジネス課題を定義し、データの利用可能性やフィージビリティを評価する段階。

#### フェーズ2: 実験・開発

| 工程 | 役割 | 主なアクティビティ |
|------|------|-------------------|
| **1. データ抽出** | 各種データソースからデータを収集・統合する | DB、API、ストリーミングソースからのデータ取り込み。データソースの選定と統合 |
| **2. データ分析（EDA）** | データの特性を理解し、モデリング方針を決定する | スキーマ確認、統計的特性の把握、欠損値・外れ値の分析、特徴量の相関分析 |
| **3. データ準備** | モデル学習に適した形式にデータを加工する | クレンジング、特徴量エンジニアリング、正規化、学習/検証/テストデータの分割 |
| **4. モデル学習** | アルゴリズムを選定し、モデルを訓練する | アルゴリズム選定、ハイパーパラメータチューニング、学習の実行、計算リソースの管理 |
| **5. モデル評価** | テストデータでモデルの品質を検証する | 精度・再現率・F1 スコア等の指標で性能を測定。過学習の検出、公平性の確認 |

#### フェーズ3: 運用

| 工程 | 役割 | 主なアクティビティ |
|------|------|-------------------|
| **6. モデル検証** | 本番デプロイの準備が整っているか確認する | ベースラインモデルとの比較、インフラ互換性テスト、A/B テスト |
| **7. モデルデプロイ（サービング）** | モデルを本番環境に展開し、予測を提供する | コンテナ化（Docker）、マイクロサービス化、API エンドポイントの構築、バッチ推論/リアルタイム推論の選択 |
| **8. 監視** | 本番環境でのモデル性能を継続的に追跡する | データドリフト検知、モデル精度の劣化検出、レイテンシ・スループットの監視、アラート設定、再学習トリガーの判定 |

### 成熟度モデル

Google Cloud と Microsoft が提唱する MLOps 成熟度モデルは、組織の自動化レベルを段階的に定義する:

| レベル | 名称 | 特徴 |
|--------|------|------|
| **Level 0** | 手動プロセス | すべての工程が手動。Jupyter Notebook での実験。モデルの引き渡しは手作業。再学習は稀 |
| **Level 1** | ML パイプライン自動化 | データ検証・モデル検証を含むパイプラインが自動化。継続的学習（CT）の実現。特徴量ストアの導入 |
| **Level 2** | CI/CD パイプライン自動化 | ML パイプラインに加え、CI（コードとデータのテスト）・CD（モデルの自動デプロイ）が自動化。高速な実験サイクル |
| **Level 3** | 自動モデルデプロイ | A/B テストやカナリアデプロイの自動化。モデルガバナンスの統合 |
| **Level 4** | 完全自動化運用 | 監視→再学習→デプロイの全サイクルが自動化。自己修復型システム |

### 継続的プロセス

MLOps では DevOps の CI/CD に加え、ML 固有の継続的プロセスが定義される:

| プロセス | 説明 |
|----------|------|
| **CI（継続的インテグレーション）** | コードだけでなく、データスキーマやモデルの品質もテスト対象に含める |
| **CD（継続的デリバリー）** | モデル予測サービスの自動デプロイ。パイプライン全体の自動デプロイも含む |
| **CT（継続的トレーニング）** | データドリフトや性能劣化をトリガーに、モデルの再学習を自動実行する |
| **CM（継続的モニタリング）** | 本番データとモデル性能の継続的な監視。異常検知とアラート |

### 主要ツール・プラットフォーム

#### エンドツーエンドプラットフォーム

| ツール | 特徴 |
|--------|------|
| **AWS SageMaker** | AWS エコシステムとの統合。学習・デプロイ・監視の全工程をカバー |
| **Google Vertex AI** | AutoML と高度な研究ツールの統合。BigQuery・GCS とシームレス連携 |
| **Azure Machine Learning** | Azure エコシステムとの統合。責任ある AI 機能が充実 |
| **Databricks** | [[データレイクハウス]]上の統合 ML プラットフォーム。Unity Catalog によるガバナンス |
| **MLflow** | オープンソースの ML ライフサイクル管理。実験トラッキング、モデルレジストリ、デプロイ |

| 工程 | ツール例 |

### 最新動向（2026年）

- **AgentOps の台頭**: 自律型 AI エージェントの管理・デプロイ・監視に特化した AgentOps が MLOps の進化形として登場
- **Policy-as-Code**: 公平性、データリネージ、コンプライアンスルールを CI/CD パイプラインに自動組み込み
- **説明可能性（XAI）の本番化**: 説明可能な AI が研究段階から本番レベルの必須要件に
- **エッジ・分散デプロイ**: TinyML、エッジアーキテクチャ、フェデレーテッドラーニングへの対応
- **LLMOps**: LLM/RAG 特有のツール群（ベクトルストア、プロンプトバージョン管理、ハルシネーション診断）が MLOps スタックに統合
- **市場規模**: 2024年の17億ドルから2034年に1,290億ドルへ成長見込み（CAGR 43%）

## ポイント

- MLOps は DevOps の原則を ML に適用し、データ収集からモデル監視までのライフサイクル全体を自動化する
- 8つの工程（データ抽出→分析→準備→学習→評価→検証→デプロイ→監視）で構成される
- 成熟度は Level 0（手動）から Level 4（完全自動化）まで段階的に進化する
- CT（継続的トレーニング）は ML 固有のプロセスで、データドリフト検知に基づく自動再学習を実現する
- 2026年は AgentOps、LLMOps、エッジデプロイが主要トレンド
