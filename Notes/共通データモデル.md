## 概要

共通データモデル（CDM: Common Data Model）は Microsoft が主導して策定した，ビジネスエンティティの標準化されたスキーマ集．アカウント・連絡先・注文・製品・キャンペーンなどの共通ビジネス概念を統一されたスキーマで定義することで，異なるアプリケーション・システム・クラウドサービス間のデータ相互運用性（インターオペラビリティ）を実現する．Microsoft・Adobe・SAP が共同で推進する Open Data Initiative（ODI）の中核．

## 詳細

### 目的と価値

```
アプリ A（独自スキーマ）
アプリ B（独自スキーマ）  →  変換・マッピングが複雑
アプリ C（独自スキーマ）

           ↓ CDM 採用

アプリ A（CDM スキーマ）
アプリ B（CDM スキーマ）  →  共通言語で直接連携
アプリ C（CDM スキーマ）
```

### 主要エンティティ

CDM が定義する代表的な標準エンティティ:

| カテゴリ | エンティティ例 |
|---------|--------------|
| 顧客・取引先 | Account（取引先企業），Contact（連絡先個人）|
| 営業 | Lead（見込み客），Opportunity（商談），Order（受注）|
| マーケティング | Campaign（キャンペーン），Activity（活動）|
| 組織 | Organization（組織），Department（部門），Worker（従業員）|
| 製品 | Product（製品），PriceList（価格表）|

### Microsoft エコシステムとの統合

CDM は Microsoft の主要製品・サービスで採用されている:

| 製品・サービス | CDM との関係 |
|--------------|------------|
| **Microsoft Dataverse** | CDM スキーマのクラウドストレージ基盤 |
| **Dynamics 365** | CRM/ERP データが CDM エンティティに準拠 |
| **Power Platform** | Power Apps/Power BI が CDM データを直接利用 |
| **Azure Data Factory** | CDM フォーマットでのデータ統合 |
| **Azure Synapse Analytics** | CDM データを分析基盤に取り込み |

### Open Data Initiative (ODI)

Microsoft・Adobe・SAP の3社が共同で推進するデータ標準化イニシアチブ．各社の顧客データ・マーケティングデータ・ERP データを CDM という共通言語で統合し，顧客の360度ビューを実現することを目的とする．

### OMOP CDM との違い

医療分野では OHDSI が策定した「OMOP Common Data Model」が別途存在する．Microsoft CDM とは別物で，医療記録（患者・診断・薬剤等）の標準化に特化している．

| | Microsoft CDM | OMOP CDM |
|-|---------------|----------|
| 対象 | 汎用ビジネスエンティティ | 医療・ヘルスケア |
| 推進 | Microsoft/Adobe/SAP | OHDSI（学術・医療コミュニティ）|
| 用途 | CRM/ERP/BI 統合 | 医療データ分析・研究 |

### 採用のメリットと課題

**メリット:**
- アプリ間のデータ統合コストが低下（変換ロジックが不要）
- Microsoft エコシステム（Dynamics 365・Power Platform・Azure）との親和性
- 意味的に一貫したデータ（セマンティックコンシステンシー）

**課題:**
- Microsoft 中心のエコシステムへのベンダーロックイン懸念
- 標準スキーマが自社ビジネスに合わない場合の拡張コスト
- オープンな業界標準（FHIR・OWL 等）との住み分けが不明確な場合も

## ポイント

- ビジネスエンティティの標準スキーマを定義し，アプリ間のデータ統合を簡素化する
- Microsoft Dataverse・Dynamics 365・Power Platform・Azure との深い統合が最大の強み
- Open Data Initiative（Microsoft・Adobe・SAP）で顧客データの360度ビューを目指す
- 医療分野の OMOP CDM とは別物．名称が同じ「共通データモデル」でも対象領域が異なる
- Microsoft エコシステム外での採用は限定的で，ベンダーロックインのリスクを評価する必要がある
