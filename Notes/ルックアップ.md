## 概要

ルックアップ（Lookup）とは，キーに基づいてテーブルや辞書から対応する値を高速に取得する操作である．データエンジニアリングでは，コードから名称への変換，ディメンションテーブルとのJOIN，計算の事前計算テーブルへの参照など，広く使われるパターンである．

## 詳細

### ルックアップテーブル

あらかじめ計算済みの値や対応表を格納したテーブルで，実行時の計算コストを削減する．

**使用例: ショップIDからカテゴリへのマッピング**
```sql
SELECT t.transaction_id, l.store_category, SUM(t.amount)
FROM transactions t
JOIN store_lookup l ON t.shop_id = l.shop_id
GROUP BY t.transaction_id, l.store_category;
```

### 主な用途

| 用途 | 説明 | 例 |
|------|------|----|
| **コード変換** | コードや ID から人が読める名称へ変換 | 都道府県コード → 都道府県名 |
| **データエンリッチメント** | イベントデータに属性情報を付加 | ユーザーIDに年齢層・地域を付加 |
| **ディメンション結合** | ファクトテーブルにディメンション属性を結合 | スタースキーマでの DWH クエリ |
| **計算キャッシュ** | 重い計算の結果を事前格納 | 三角関数テーブル，ハッシュ値テーブル |
| **バリデーション** | 入力値が有効な値かを確認 | マスタテーブルとの突合 |

### ルックアップの実装パターン

**インメモリルックアップ**
- 小さいルックアップテーブルをメモリに展開し，O(1) でアクセスする
- ストリーム処理（Kafka Streams，Flink等）でよく使われる

**ブロードキャストJOIN**
- 分散クエリエンジン（Spark，BigQuery等）で小テーブルを全ノードにブロードキャストしてネットワーク転送を削減する

**サイドインプット**
- バッチ処理パイプライン（Apache Beam 等）で，メインデータの処理中に参照用データを別途読み込む

## ポイント

- ルックアップテーブルはサイズが小さいほど効果的（ブロードキャストJOINの恩恵が大きい）
- ルックアップ頻度が高い場合は[[キャッシュ]]（Redisなど）に乗せると低遅延になる
- ルックアップ対象のカラムにインデックスを貼ることで検索が高速化する
- ストリーミング処理では，ルックアップテーブルの更新タイミングに注意が必要（古い値を参照するリスク）
