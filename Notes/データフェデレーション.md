## 概要

データフェデレーション（Data Federation）は，物理的に分散した複数のデータソースを単一の仮想データベースとして扱う技術・アーキテクチャである．データを物理的に移動・コピーせず，クエリ実行時に各ソースから動的にデータを取得・統合する．

## 詳細

### 仕組み

データフェデレーションは各データソースの上に仮想層（フェデレーション層）を置き，物理的な複雑さを隠蔽する．

```
クライアント
    ↓ SQL クエリ
[フェデレーション層（仮想データベース）]
    ↓ 分散クエリ
┌──────────┬──────────┬──────────┐
DB-A      DB-B      API-C     ファイル-D
(RDBMS)   (NoSQL)   (REST)    (CSV/JSON)
```

処理フロー:
1. クライアントがフェデレーション層に単一クエリを送信
2. フェデレーション層がクエリを各ソース向けに分解
3. 各ソースからデータを並列取得
4. 標準データモデルで結合・変換して返却

### データ仮想化との関係

データフェデレーションと[[データ仮想化]]はほぼ同義で使われることが多いが，厳密には:
- **データフェデレーション**: 複数の異種データソースの統合アクセスに焦点
- **データ仮想化**: より広い概念でキャッシュ・最適化・ガバナンスも含む

### メリット

| メリット | 説明 |
|---------|------|
| データ移動不要 | 追加ストレージ・ETLコスト不要 |
| リアルタイムアクセス | 最新データをその場でクエリ可能 |
| 柔軟性 | 異種ソース（RDBMS，NoSQL，API，ファイル）を統合 |
| 迅速な実装 | ETLパイプライン構築よりも素早く統合できる |

### デメリット・制約

- ネットワーク遅延の影響を受けるため，大規模集計クエリは遅い
- ソース側のクエリ能力に依存する
- ソースが停止するとクエリが失敗する（可用性の連鎖依存）
- データ品質の担保がソース側に委ねられる

### ユースケース

- **レガシー統合**: 既存システムを変えずに新アプリから統合アクセス
- **規制対応**: データを元の場所に留めたままコンプライアンス要件を満たす
- **プロトタイプ**: ETL構築前にデータ統合の試験的実装
- **Lakehouse Federation**: Databricks Lakehouse Federation のような外部カタログ統合

## ポイント

- データを「コピー」ではなく「参照」することが本質的な特徴
- ETL/ELT の代替として，データ移動コストを削減できる
- 大量データの集計より，少量データの横断的な参照に向く
- [[データ仮想化]] と組み合わせてキャッシュ層を設けると性能改善できる
- クラウドでは Databricks Lakehouse Federation，Trino，Presto が代表的実装
