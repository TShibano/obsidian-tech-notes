## 概要

データアーキテクチャは、データ資産の収集・保存・配置・統合・利用を統制するモデル・ポリシー・ルール・標準の体系。DAMA-DMBOK の11知識領域の一つであり、TOGAF のエンタープライズアーキテクチャにおける4ドメイン（Business / Data / Application / Technology）の「Data」ドメインに相当する。

## 詳細

### 3層モデル

| 層 | 目的 | 内容 |
|---|---|---|
| **概念モデル（Conceptual）** | ビジネス視点の高レベル概要 | ビジネスゴールとの整合、主要エンティティ・関係の概略 |
| **論理モデル（Logical）** | データの抽象的構造 | エンティティ・属性・リレーション。DBMS 非依存 |
| **物理モデル（Physical）** | 実際の DB 実装 | テーブル定義・インデックス・パーティション・ストレージ方式 |

### 主要パターン

#### データウェアハウス（DWH）

構造化データの高品質な BI 向け。Inmon 方式（中央集権的 EDW）と Kimball 方式（データマート中心のスタースキーマ）が二大設計思想。

#### データレイク

生データを任意フォーマットで大量保存。柔軟性・コスト効率に優れるが、ガバナンス不足で「データスワンプ」化するリスク。

#### [[データレイクハウス]]

データレイクの柔軟性と DWH のデータ管理機能を統合。ACID トランザクション・スキーマ強制・BI〜ML の統合ワークロードを実現。Delta Lake / [[Apache Iceberg]] が主要実装。

#### [[メダリオンアーキテクチャ]]

Bronze → Silver → Gold の3層でデータ品質を段階的に向上。Databricks が普及させた。Lambda/Kappa どちらのパイプラインとも組み合わせ可能。

#### Lambda / Kappa アーキテクチャ

- **Lambda**: バッチ層（高スループット）＋スピード層（低遅延）＋サービング層。運用複雑性が高い
- **Kappa**: バッチ層を除去し、全データをストリームとして処理。シンプルだがリプレイコストが高い
- 2026年現在、Kappa 寄りの「ストリーミングファースト」設計への移行が加速

#### [[データメッシュ]]

ドメイン指向の分散オーナーシップ。各ビジネスドメインが自律的に「データプロダクト」を管理する組織的パターン。

#### [[データファブリック]]

分散する異種データ環境を統合的に接続・管理するメタデータ駆動のアーキテクチャ。AI 活用による自動化が特徴。データメッシュと補完関係。

### フレームワーク

| フレームワーク | 特性 | 用途 |
|---|---|---|
| **TOGAF ADM** | プロセス中心・適応性高い | EA 全体のガバナンス・開発プロセス |
| **Zachman** | 分類・文書化中心・6×6 マトリクス | ステークホルダー向けアーキテクチャ文書化 |
| **[[DMBOK]]** | データ管理特化・ロール・成果物を定義 | データガバナンス・データ管理標準化 |

成熟組織では TOGAF をガバナンス方法論、DMBOK をデータ固有の規律管理、Zachman をコミュニケーションツールとして組み合わせるハイブリッドアプローチが推奨される。

### 最新動向（2025-2026）

#### AI ネイティブデータアーキテクチャ

- ストリーミングファースト取込・自律的最適化・カタログ駆動ガバナンスが「ベースライン要件」化
- レイクハウスがリアルタイム分析・エージェント AI・エッジ推論まで拡張
- Agentic AI ワークフロー向けのデータ基盤設計が新たな設計課題

#### オープンテーブルフォーマットの収束

| フォーマット | 特徴 |
|---|---|
| **Apache Iceberg** | ベンダーニュートラル。AWS/GCP/Azure/主要エンジンから広くサポート。デファクト化が進行 |
| **Delta Lake** | Databricks エコシステムで支配的 |
| **Apache Hudi** | ストリーミングネイティブニッチ |
| **Apache Paimon** | ストリーミング特化の新興フォーマット |

Iceberg V3（2025-2026）: Variant 型（半構造化データ）、地理空間型、行レベル CDC、削除ベクター等を追加。Snowflake が 2026年4月に V3 サポートを発表。

#### ストリーミング統合

Apache Kafka + Apache Flink + Apache Iceberg の組み合わせがリアルタイム・バッチ統合基盤の標準スタックとして台頭。

#### ガバナンスの自動化

Polaris Catalog（Iceberg 向けオープンカタログ）がアクセス制御・リネージ・セマンティックコンテキストをエンジン非依存で管理。「ポリシーがデータと共に移動する」モデルを実現。

### 設計ステップ

1. **現状把握**: 既存データ資産・フロー・品質課題の棚卸し
2. **ビジネス要件の整合**: データアーキテクチャをビジネス戦略・KPI にマッピング
3. **フレームワーク選択**: TOGAF / DMBOK / Zachman のいずれか or ハイブリッド
4. **概念・論理・物理モデルの設計**: 3層で段階的に詳細化
5. **パターン選択**: ユースケースに応じて DWH / レイク / レイクハウス / メッシュを選択
6. **ガバナンス設計**: データカタログ・アクセス制御・リネージ・品質ルール
7. **組織設計**: データオーナーシップ・エンジニア/アナリスト/サイエンティストの役割定義

## ポイント

- 概念・論理・物理の3層で段階的に設計するのが基本
- パターン選択はユースケース次第。レイクハウスが 2026年の主流だが、単一パターンで全てを解決しない
- TOGAF + DMBOK + Zachman のハイブリッドが成熟組織の推奨構成
- オープンテーブルフォーマットは Apache Iceberg への収束が加速
- ストリーミングファースト設計（Kafka + Flink + Iceberg）が標準化
- 成熟したデータアーキテクチャを持つ組織は DX 成功率が 42% 高く、管理コストが 35% 低い
