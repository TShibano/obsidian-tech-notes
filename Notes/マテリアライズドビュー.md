## 概要

クエリの結果を物理的にディスクに保存するデータベースオブジェクト．通常のビューが実行のたびにクエリを再計算するのに対し，マテリアライズドビューは計算済みの結果をキャッシュし，複雑なクエリの高速化を実現する．

## 詳細

### 通常のビューとの違い

| | 通常のビュー | マテリアライズドビュー |
|---|---|---|
| データ保存 | なし（仮想） | あり（物理テーブルとして保存） |
| アクセス速度 | 毎回クエリ実行 | 事前計算済みのため高速 |
| データ鮮度 | 常に最新 | リフレッシュまで古いデータ |
| インデックス | 不可 | 可能 |
| 主キー | 不可（一般的に） | 定義可能 |

### 仕組み

1. 定義時に指定したクエリを実行し，結果をディスクに保存
2. 以降はキャッシュから高速に応答
3. 元テーブルの変更後，**リフレッシュ**処理で結果を更新

### リフレッシュ戦略

| 戦略 | 説明 |
|---|---|
| 手動リフレッシュ | `REFRESH MATERIALIZED VIEW` を明示的に実行 |
| スケジュール | 定期的に自動リフレッシュ |
| 増分リフレッシュ | 変更差分のみ更新（効率的だが DBMS によりサポート状況が異なる） |
| リアルタイム更新 | 元テーブルの変更に連動して即時更新 |

### クラウドデータウェアハウスでの実装

- **[[BigQuery]]**: マテリアライズドビューをサポート．増分更新が可能で，スキャンデータ量とコストを削減
- **Snowflake**: 自動的にリフレッシュするマテリアライズドビューをサポート
- **PostgreSQL**: `CONCURRENTLY` オプションでロックなしリフレッシュが可能

### データエンジニアリングでの活用

- **集計クエリの高速化**: 日次・週次集計の事前計算
- **コスト削減**: BigQuery 等では読み取りデータ量に応じた課金のため，マテビューでスキャン量を削減
- **変換処理**: [[ELT]] パターンでの変換結果をマテビューとして保持

## ポイント

- リフレッシュを怠ると古いデータを返すため，更新タイミングの設計が重要
- 増分リフレッシュはフルリフレッシュより効率的だが，元クエリに制約あり（JOIN や集計関数に制限）
- 集計・JOIN が複雑なクエリほどマテビューの効果が大きい
