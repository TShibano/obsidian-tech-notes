## 概要

ソースシステムからターゲット宛先へデータを移動・変換する一連の処理ステップ．データの取り込み（ingestion）から変換・格納・提供までの流れを自動化する仕組みで，データエンジニアリングライフサイクルの中核．

## 詳細

### データパイプラインとは

- データをソースから抽出し，変換を加え，特定の目的地に保存する一連のプロセス
- スケジュール実行またはリアルタイム（イベント駆動）で動作
- 信頼性・品質のためにモニタリングが必要

### 主なパターン

| パターン | 概要 |
|---|---|
| [[ETL]] | Extract → Transform → Load（変換してから格納） |
| [[ELT]] | Extract → Load → Transform（格納してから変換） |
| ストリーミング | リアルタイムに連続処理（[[Apache Kafka]] 等） |
| バッチ | 固定スケジュールで一括処理 |

### 設計の重要検討事項（データエンジニアリングの基礎）

データを取り込む際に検討すべき項目：
- データのユースケースは何か？
- データ量はどのくらいか？
- データのフォーマットは何か？
- データの頻度（バッチ or ストリーム）は？

### オーケストレーションツール

- **[[Apache Airflow]]**: DAG による依存関係管理が特徴
- **[[Dagster]]**: アセット志向のパイプライン管理
- **[[Prefect]]**: Pythonic なワークフロー定義

### Pipeline as Code

パイプライン定義をコードとして管理する手法（[[Pipeline as Code]]）．CI/CD と統合してテスト・バージョン管理を実現する．

## ポイント

- パイプラインの信頼性・スケーラビリティ・コストはトレードオフ
- データ品質の検証をパイプライン内に組み込む（データストロフィを防ぐ）
- [[データ契約]]によりソースシステムのスキーマ変更に備える
