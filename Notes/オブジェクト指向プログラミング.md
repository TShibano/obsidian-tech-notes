## 概要

オブジェクト指向プログラミング（OOP: Object-Oriented Programming）は、データとそれを操作する処理を「オブジェクト」という単位にまとめ、オブジェクト間のメッセージパッシングでシステムを構築するプログラミングパラダイム。カプセル化・継承・ポリモーフィズムを三大原則とし、C++・Java・Python・Ruby などの主要言語が採用している。

## 詳細

### 三大原則

#### カプセル化（Encapsulation）

データ（フィールド）とそのデータを操作するメソッドを一つのクラスにまとめ、外部から直接アクセスできないように隠蔽する。

```python
class BankAccount:
    def __init__(self, balance: float):
        self._balance = balance  # プライベート（慣習的）

    def deposit(self, amount: float):
        if amount > 0:
            self._balance += amount

    def get_balance(self) -> float:
        return self._balance

account = BankAccount(1000)
account.deposit(500)
print(account.get_balance())  # 1500.0
# account._balance = -999  # 直接変更は設計上避ける
```

**目的:** 実装の詳細を隠し、オブジェクトの状態を一貫したインターフェースのみで変更可能にする。

#### 継承（Inheritance）

既存クラス（親クラス/スーパークラス）の属性・メソッドを引き継いだ新しいクラス（子クラス/サブクラス）を定義する仕組み。

```python
class Animal:
    def __init__(self, name: str):
        self.name = name

    def speak(self) -> str:
        raise NotImplementedError

class Dog(Animal):
    def speak(self) -> str:
        return f"{self.name}: ワン！"

class Cat(Animal):
    def speak(self) -> str:
        return f"{self.name}: ニャン！"
```

**目的:** コードの再利用・「is-a」関係の表現。ただし継承の乱用は密結合を生むため、「コンポジション（has-a）を継承より優先する」という設計指針も重要。

#### ポリモーフィズム（Polymorphism）

同じインターフェースや基底クラスに対して、異なるクラスが異なる振る舞いを実装する。

```python
def make_sound(animal: Animal) -> str:
    return animal.speak()  # Dog でも Cat でも同じコードで呼べる

animals = [Dog("タロウ"), Cat("ミケ")]
for a in animals:
    print(make_sound(a))
# タロウ: ワン！
# ミケ: ニャン！
```

**目的:** 型に依存しない汎用的なコードを書けるようにし、新しいサブクラスを追加しても呼び出し側を変更不要にする。

### SOLID 原則

オブジェクト指向設計の5原則。Robert C. Martin（Uncle Bob）が提唱。

| 原則 | 説明 |
|------|------|
| **S** - 単一責任の原則 | クラスが変更される理由は1つだけであるべき |
| **O** - 開放閉鎖の原則 | 拡張に開き、変更に閉じる（継承・インターフェースで拡張） |
| **L** - リスコフ置換の原則 | サブクラスは親クラスと置き換え可能でなければならない |
| **I** - インターフェース分離の原則 | クライアントが不要なメソッドへの依存を強制されてはならない |
| **D** - 依存性逆転の原則 | 上位モジュールは下位モジュールに依存してはならない（両者は抽象に依存する） |

```python
# 依存性逆転の原則の例
from abc import ABC, abstractmethod

class DatabaseInterface(ABC):  # 抽象
    @abstractmethod
    def save(self, data: dict) -> None: ...

class PostgreSQLDB(DatabaseInterface):  # 具体的な実装
    def save(self, data: dict) -> None:
        print(f"PostgreSQL に保存: {data}")

class UserService:  # 上位モジュール
    def __init__(self, db: DatabaseInterface):  # 抽象に依存
        self.db = db

    def create_user(self, name: str):
        self.db.save({"name": name})
```

### デザインパターン

GoF（Gang of Four）が整理した代表的なOOPパターン：

| カテゴリ | パターン | 説明 |
|---------|---------|------|
| 生成 | Singleton | インスタンスを1つに制限 |
| 生成 | Factory Method | サブクラスでインスタンス生成を決める |
| 生成 | Builder | 複雑なオブジェクトを段階的に構築 |
| 構造 | Decorator | オブジェクトに機能を動的に追加 |
| 構造 | Adapter | 互換性のないインターフェースを変換 |
| 構造 | Composite | ツリー構造を表現 |
| 振る舞い | Observer | 状態変化を通知 |
| 振る舞い | Strategy | アルゴリズムを差し替え可能に |
| 振る舞い | Command | リクエストをオブジェクトとして扱う |

### OOP が適する場面

- 複雑なビジネスドメインをモデル化する（DDD: ドメイン駆動設計）
- 状態を持つ長命なオブジェクトが中心のシステム
- GUI アプリケーション（ウィジェットの階層）
- ゲームの「キャラクター」「アイテム」等のエンティティ

### OOP の批判と代替パラダイム

OOP の批判：
- **継承は密結合を生む**: 親クラスの変更が子クラスに波及する
- **状態管理が複雑**: ミュータブルな状態がバグを生みやすい
- **テストが難しい**: 状態を持つオブジェクトのモッキングが複雑

これを踏まえた代替・補完パラダイム：
- **[[関数型プログラミング]]**: 状態・副作用を排除、純粋関数で構成
- **[[データ指向プログラミング]]**: データとコードを分離し、パフォーマンス重視
- **コンポジション vs 継承**: 継承より委譲（has-a）を優先する設計

### 言語ごとのOOPサポート

| 言語 | OOP スタイル |
|------|------------|
| Java | クラスベース（すべてがクラス） |
| Python | クラスベース（マルチ継承あり） |
| Ruby | 純粋OOP（プリミティブもオブジェクト） |
| C++ | クラスベース（FP・手続き型も混在可） |
| [[Rust]] | トレイトベース（継承なし、コンポジション重視） |
| JavaScript | プロトタイプベース（+ クラス構文） |
| Go | インターフェースベース（クラス・継承なし） |

## ポイント

- カプセル化・継承・ポリモーフィズムがOOPの三大原則
- SOLID 原則は OOP 設計の質を高めるためのガイドライン
- 継承よりコンポジションを優先する現代的な設計思想
- FP・DOD など補完的なパラダイムと組み合わせることでより良い設計が可能
- Rust は継承なしのトレイトベースで OOP の良い部分を取り込んでいる
