## 概要

マイクロサービスアーキテクチャは、アプリケーションを小さく独立したサービスの集合として構築するアーキテクチャスタイル。各サービスはビジネスドメインに基づいて分割され、独立してデプロイ・スケーリング可能で、[[API]] を通じて通信する。

## 詳細

### 基本構造

```
┌─────────┐  ┌─────────┐  ┌─────────┐
│ 注文     │  │ 在庫     │  │ 決済     │
│サービス   │  │サービス   │  │サービス   │
│ (Python) │  │ (Go)    │  │ (Java)  │
│ [注文DB]  │  │ [在庫DB]  │  │ [決済DB]  │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │
─────┴─────────────┴─────────────┴───── API Gateway
```

各サービスが独自のデータベースを持ち（Database per Service）、サービス間は API やメッセージングで通信する。言語やフレームワークもサービスごとに自由に選択可能。

### 設計原則

- **ドメイン駆動設計（DDD）**: ビジネスドメインに基づいてサービス境界を決定する。Bounded Context がサービスの境界に対応
- **Database per Service**: 各サービスが専用のデータストアを持つ。他サービスのDBに直接アクセスしない
- **[[疎結合]]・高凝集**: サービス間の依存を最小化し、サービス内の関連機能は集約する
- **分散型ガバナンス**: 各チームが技術選択・デプロイ戦略を自律的に決定する

### サービス間通信

#### 同期通信
- **REST API**: 最も一般的。シンプルで理解しやすい
- **gRPC**: 高性能が求められる内部通信。Protocol Buffers によるバイナリシリアライズ
- **GraphQL**: API Gateway 層でのデータ集約に活用

#### 非同期通信
- **メッセージキュー**: Apache Kafka、RabbitMQ、Amazon SQS。[[疎結合]]な通信を実現
- **イベント駆動**: イベントソーシング + CQRS パターン。サービス間の結果整合性を実現

### 主要な設計パターン

| パターン | 目的 | 説明 |
|---------|------|------|
| API Gateway | 統一エントリポイント | クライアントからのリクエストを適切なサービスにルーティング |
| Saga | 分散トランザクション | 複数サービスにまたがるトランザクションを補償トランザクションで管理 |
| Circuit Breaker | 障害の連鎖防止 | 障害が発生したサービスへのリクエストを遮断し、システム全体の崩壊を防ぐ |
| Bulkhead | 障害の分離 | サービスごとにリソースプールを分離し、1サービスの障害が他に影響しない |
| Strangler Fig | 段階的移行 | [[モノリスアーキテクチャ]]から段階的にサービスを切り出す |
| Sidecar | 横断的関心事 | ログ・監視・認証等をサイドカーコンテナとして付与 |

### Saga パターンの詳細

マイクロサービスでは単一のACIDトランザクションが使えないため、Saga パターンで分散トランザクションを管理する:

```
注文作成 → 在庫確保 → 決済処理 → 配送手配
   ↓失敗時    ↓失敗時    ↓失敗時
注文取消 ← 在庫戻し ← 決済取消
         （補償トランザクション）
```

- **コレオグラフィ方式**: 各サービスがイベントを発行・購読して協調
- **オーケストレーション方式**: 中央のオーケストレータがフロー全体を制御

### メリット

- **独立デプロイ**: 各サービスを個別にデプロイ可能。リリース頻度の向上
- **技術的多様性**: サービスごとに最適な言語・フレームワーク・DBを選択可能
- **スケーラビリティ**: 負荷の高いサービスだけをスケールアウト可能
- **チームの自律性**: サービスごとにチームを割り当て、独立して開発可能（Conway の法則）
- **障害の分離**: 1サービスの障害がシステム全体に波及しにくい

### デメリットと課題

- **分散システムの複雑性**: ネットワーク障害・レイテンシ・データ整合性への対処が必須
- **運用コスト**: 多数のサービスの監視・ログ集約・デプロイパイプラインの管理
- **データの整合性**: 結果整合性（Eventual Consistency）の受け入れと補償処理の実装
- **デバッグの困難さ**: リクエストが複数サービスを横断するため、分散トレーシングが必要
- **テストの複雑性**: サービス間の結合テスト・契約テストの設計

### 市場動向

マイクロサービスの市場規模は2023年に37億米ドルに達し、2032年には118億米ドル（CAGR 13.2%）と予測されている。クラウドネイティブ開発と Kubernetes の普及が成長を牽引している。

## ポイント

- サービス境界は DDD の Bounded Context に基づいて決定する
- 分散トランザクションには Saga パターンで結果整合性を確保する
- Circuit Breaker で障害の連鎖を防ぎ、システム全体のレジリエンスを高める
- 「モノリスファースト」で始め、スケーラビリティやチーム規模の要件に応じて移行するアプローチが推奨される
- 分散トレーシング（Jaeger、Zipkin）・集中ログ管理が運用の前提条件
