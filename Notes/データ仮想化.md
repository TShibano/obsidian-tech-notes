## 概要

データ仮想化（Data Virtualization）はデータを物理的にコピーや移動することなく、複数の異なるデータソースを統合した仮想ビューを作成し、統一インターフェースでクエリを実行するデータ統合手法。DWH・データレイク・データベース・API・SaaS など分散した異種データソースを横断するフェデレーテッドクエリを可能にし、データ移動コストとリアルタイム性のトレードオフを解消する。

## 詳細

### 基本概念

```
利用者（BI ツール・アプリ・AI）
          ↓ 統一された SQL / API
仮想化レイヤー（クエリを分解・最適化）
  ↙         ↓         ↘
DWH    データレイク    SaaS（CRM 等）
（Snowflake）（S3）    （Salesforce）
```

データのコピーを作らず、クエリを各ソースに分解して実行し結果をマージする「フェデレーテッドクエリ（Query Federation）」が中核技術。

### フェデレーテッドクエリの仕組み

1. ユーザーが仮想テーブルに対して SQL を発行
2. データ仮想化エンジンがクエリを解析・最適化
3. 各サブクエリを適切なソースシステムに送信（プッシュダウン）
4. 各ソースの結果をマージして返す

**クエリプッシュダウン**: WHERE 句や JOIN 条件をソース側で実行させることで転送データ量を削減。

### データ仮想化 vs ETL/ELT

| 観点 | データ仮想化 | ETL/ELT |
|------|------------|---------|
| データ移動 | なし（クエリのみ） | あり（コピー） |
| リアルタイム性 | 高い（ソース直接参照） | 低い（バッチ依存） |
| クエリ性能 | ソース次第で不安定 | 事前最適化で高速 |
| ストレージコスト | 低い | 高い（複製分） |
| ガバナンス | 統一アクセス制御が容易 | ソースごとに管理 |
| 複雑な変換 | 難しい | 容易 |

### 主要ツール（2026）

| ツール | 特徴 |
|--------|------|
| **Denodo** | 純粋データ仮想化のリーダー。インテリジェントクエリ最適化・動的キャッシュ |
| **Databricks Lakehouse Federation** | Databricks 上でのフェデレーテッドクエリ基盤 |
| **IBM Data Virtualization（Watson Query）** | IBM クラウドエコシステム向け |
| **Domo** | 数百のデータソースへのフェデレーテッド接続 |
| **Starburst/Trino** | オープンソースベースの分散 SQL エンジン |

### データファブリックとの関係

[[データファブリック]] はデータ仮想化を主要な構成要素の一つとして採用する。データファブリックがメタデータ・ガバナンス・オーケストレーション全体を包含するのに対し、データ仮想化はその「統合・クエリ実行」レイヤーを担う。

### 2026 年の戦略パターン

実践的なアプローチとして「バーチャルファースト・マテリアライズオンデマンド」が台頭:

1. まず全データソースを仮想接続（コピーなし）で繋ぐ
2. クエリ頻度・パフォーマンスを監視
3. 複数チームが毎日参照するデータは Apache Iceberg テーブルとしてマテリアライズ（物理化）
4. アクセス頻度の低いデータは仮想のまま維持

これにより、コストとパフォーマンスの最適なバランスを段階的に実現できる。

## ポイント

- データをコピーせずに複数ソースを横断するフェデレーテッドクエリでリアルタイムアクセスを実現
- ストレージコスト削減とデータの鮮度維持が主なメリット、クエリ安定性がトレードオフ
- クエリプッシュダウンによりソース側で絞り込みを実行し、データ転送量を最小化
- 「仮想でまず繋ぎ、頻繁に使うものを物理化（Iceberg テーブル等）」がコスト最適化の実践戦略
- [[データファブリック]] の統合レイヤーとして、またデータ民主化のゲートウェイとして機能する
