## 概要

ベゾスAPI司令（Bezos API Mandate）は、2002年頃に Amazon CEO ジェフ・ベゾスが社内全チームに発した指令。すべてのチームがデータと機能をサービスインターフェース（API）経由で公開し、チーム間の通信は例外なく API 経由で行うことを義務化した。この指令は Amazon のサービス指向アーキテクチャ（SOA）への転換を促し、後の AWS 誕生の基盤となった。

## 詳細

### 指令の内容

2002年頃、ベゾスは以下のルールを全社に通達した:

1. **すべてのチームは、データと機能をサービスインターフェース経由で公開しなければならない**
2. **チーム間の通信はサービスインターフェース経由で行わなければならない**
3. **他のプロセス間通信は一切許可しない** — 直接リンク、他チームのデータストアへの直接読み取り、共有メモリモデル、バックドアなど、いかなる手段も禁止。ネットワーク経由のサービスインターフェース呼び出しのみが許可される
4. **使用するプロトコルは問わない**（HTTP、CORBA など何でもよい）
5. **すべてのサービスインターフェースは、例外なく、外部公開可能な設計でなければならない** — つまり、外部の開発者にそのまま公開できるように設計・計画する
6. **これに従わない者は解雇する**

### 公開の経緯

この指令の存在は約10年間、社外には知られていなかった。2011年、元 Amazon エンジニアの Steve Yegge が Google 社内向けに投稿したメモ（通称「Google Platforms Rant」）を誤って公開したことで、ベゾスの指令が広く知られることになった。

Yegge のメモは、Google がプラットフォーム思考を理解していないと批判する内容で、その対比として Amazon がベゾスの指令によっていかにプラットフォーム企業に変貌したかを詳述していた。

### Amazon への影響

ベゾスの指令により、Amazon は数年かけて全社的にサービス指向アーキテクチャ（SOA）へ移行した:

- **社内システムのAPI化**: すべての社内システムがサービスとして再設計された
- **AWS の誕生**: 社内で構築されたインフラサービス（ストレージ、コンピューティング等）が「外部公開可能」な品質で設計されていたため、そのまま外部サービスとして提供可能だった。これが AWS（2006年公開）の基盤となった
- **Fulfillment By Amazon（FBA）**: 物流サービスの API 化により、外部事業者への物流提供が可能に
- **Amazon Marketplace**: サードパーティ出品者向けの API プラットフォームの基盤
- **組織文化の変革**: 「Two-Pizza Team」（2枚のピザで足りる規模のチーム）が自律的にサービスを所有・運用する体制

### ソフトウェア業界への影響

ベゾスの指令は、以下の概念を業界に広めた先駆的事例として評価されている:

- **API ファースト設計**: システム設計の出発点をAPIとする思想
- **[[マイクロサービスアーキテクチャ]]**: サービス単位で独立したデプロイ・スケーリングを可能にするアーキテクチャ
- **[[疎結合]]の徹底**: チーム間・システム間の依存を API のみに制限
- **プラットフォーム思考**: 内部サービスを外部にも提供可能な品質で構築する考え方
- **DevOps / You Build It, You Run It**: チームがサービスの構築から運用まで責任を持つ文化

### 教訓

- **強制力の重要性**: 「これに従わない者は解雇する」という強いメッセージが組織全体の行動を変えた
- **外部公開品質の設計**: 内部向けであっても外部公開を前提に設計することで、API の品質が飛躍的に向上した
- **プラットフォームの副産物としての新事業**: AWS は当初外部提供を目的としていなかったが、高品質な内部サービスがそのまま巨大な新規事業になった

## ポイント

- 2002年の社内指令だが、その影響は現在のソフトウェアアーキテクチャ全体に及ぶ
- 「すべてのサービスインターフェースは外部公開可能でなければならない」が最も革新的なルール
- AWS は API ファースト設計の副産物として誕生した
- Steve Yegge の「Google Platforms Rant」（2011年）で世に広く知られた
- 現代の API ファースト設計・マイクロサービスの先駆的実践例
