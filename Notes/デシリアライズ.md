## 概要

デシリアライズ（Deserialization）は，バイト列やテキスト形式のデータを，メモリ上のデータ構造やオブジェクトに復元する処理．[[シリアライズ]] の逆変換にあたる．

## 詳細

### 処理の流れ

```
受信バイト列 / ファイル
  ↓ デシリアライズ
メモリ上のオブジェクト（構造体, クラスインスタンス等）
```

形式の例：
- JSON 文字列 → 言語のオブジェクト/マップ
- Protobuf バイト列 → 生成された構造体
- [[Apache Avro]] バイナリ → レコード

### セキュリティリスク

デシリアライズは信頼できないデータに対して適用すると深刻な脆弱性を生む：

| 脆弱性 | 説明 | 対策 |
|--------|------|------|
| **Java Deserialization Attack** | 悪意あるバイト列から任意コードを実行させる | ホワイトリスト, SerialKiller 等のライブラリ |
| **PHP Object Injection** | `__wakeup` 等のマジックメソッドを悪用 | `json_decode` を優先使用 |
| **YAML Bomb / Billion Laughs** | 再帰展開でメモリを枯渇させる | パーサの深さ制限 |
| **Pickle（Python）** | 任意コード実行が可能な設計 | 信頼できるソースのみに適用 |

### 型の安全なデシリアライズ

静的型付き言語では，スキーマを事前に定義してデシリアライズを型安全に行う：
- Go: `encoding/json` + 構造体タグ
- Rust: `serde`
- Java: Jackson / Gson
- Python: Pydantic

## ポイント

- 外部データのデシリアライズは必ずバリデーションを行う
- バイナリ形式（Protobuf, Avro）はスキーマが事前定義されるため型安全性が高い
- Python の `pickle` は内部データ専用で，外部受信データに使ってはいけない
