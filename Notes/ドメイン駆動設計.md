## 概要

ドメイン駆動設計（DDD: Domain-Driven Design）は，Eric Evans が2003年に著書「Domain-Driven Design: Tackling Complexity in the Heart of Software」で提唱したソフトウェア設計手法である．ビジネスドメイン（業務領域）の概念・ルール・プロセスをソフトウェアモデルに直接反映し，ドメインエキスパートと開発者が共通の言語（ユビキタス言語）で協働することで，複雑なビジネスロジックを持つシステムの設計品質を高める．

## 詳細

### DDDの2つの側面

DDD は「戦略的設計」と「戦術的設計」の2層で構成される：

```
戦略的設計（大きな設計判断）
  └─ 境界づけられたコンテキスト（Bounded Context）
  └─ コンテキストマップ
  └─ ユビキタス言語

戦術的設計（実装パターン）
  └─ エンティティ（Entity）
  └─ 値オブジェクト（Value Object）
  └─ 集約（Aggregate）
  └─ ドメインサービス（Domain Service）
  └─ リポジトリ（Repository）
  └─ ファクトリ（Factory）
  └─ ドメインイベント（Domain Event）
```

### 戦略的設計

#### ユビキタス言語（Ubiquitous Language）

ドメインエキスパートと開発者が共有する共通の言語・用語集．コードの変数名・クラス名もこの言語を使って命名する：

```python
# 悪い例（技術用語中心）
class DataRecord:
    def process(self, flag: bool): ...

# 良い例（ドメイン言語を使う）
class Order:
    def confirm(self, payment: Payment): ...
```

#### 境界づけられたコンテキスト（Bounded Context）

大規模なドメインを意味的に一貫したサブドメインに分割する単位．同じ概念でもコンテキストによって意味が異なる：

```
[注文コンテキスト]       [配送コンテキスト]     [請求コンテキスト]
  Order（注文）            Order（出荷依頼）       Order（請求対象）
  Customer（購入者）       Customer（届け先）      Customer（課金対象）
  Product（商品情報）      Product（梱包物）        Product（請求品目）
```

各コンテキストは独自のモデルを持ち，コンテキスト間の関係をコンテキストマップで管理する．

### 戦術的設計の主要パターン

#### エンティティ（Entity）

一意の識別子（ID）を持ち，時間とともに状態が変化するオブジェクト：

```python
class Order:  # 注文IDで同一性を判断
    def __init__(self, order_id: OrderId, ...): ...
    def add_item(self, product: Product, qty: int): ...
    def confirm(self): ...  # 状態が変化する
```

#### 値オブジェクト（Value Object）

識別子を持たず，値の等値性で判断するイミュータブルなオブジェクト：

```python
class Money:  # 金額 + 通貨の組で同一性を判断
    def __init__(self, amount: Decimal, currency: str): ...
    def add(self, other: 'Money') -> 'Money': ...
    # 変更時は新しいインスタンスを返す
```

#### 集約（Aggregate）

整合性の境界を持つエンティティ・値オブジェクトの集合．集約ルートを通じてのみ外部からアクセスできる：

```
[Order 集約]
  Order（集約ルート）
    ├─ OrderItem（エンティティ）
    │    └─ Money（値オブジェクト）
    └─ DeliveryAddress（値オブジェクト）
```

### マイクロサービスとの関係

[[マイクロサービスアーキテクチャ]] において，境界づけられたコンテキストがサービスの分割単位として使われる：

```
[注文コンテキスト]  →  注文マイクロサービス
[配送コンテキスト]  →  配送マイクロサービス
[請求コンテキスト]  →  請求マイクロサービス
```

**注意**: 1コンテキスト = 1サービスではなく，ビジネス要件・チーム構造によって複数コンテキストを1サービスに収めることも有効．

### データメッシュとの関係

[[データメッシュ]] のドメイン指向データオーナーシップは，DDD の境界づけられたコンテキストの思想を「データの所有権」に適用した発展形と見なせる：

- DDD のドメイン ≈ データメッシュのドメイン
- DDD の集約 ≈ データメッシュのデータプロダクト

## ポイント

- DDD の本質は「ビジネスドメインの概念を忠実にコードで表現する」こと
- 戦略的設計（コンテキスト分割）の方が戦術的設計（パターン実装）より重要
- ユビキタス言語を徹底することで，ビジネス担当者とエンジニアの認識ギャップを減らせる
- [[マイクロサービスアーキテクチャ]] のサービス境界を決定する重要な理論的基盤
- 過剰なDDD適用（過度のパターン適用）はコードを複雑にする「DDD地獄」に注意
