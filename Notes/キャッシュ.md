## 概要

キャッシュ（Cache）は，低速なデータソースへのアクセスを削減するために，高速なストレージに結果を一時保存する技術．CPU レジスタからブラウザキャッシュまで，コンピュータシステムの各層に存在する．

## 詳細

### キャッシュ戦略

| 戦略 | 説明 | 適用場面 |
|------|------|---------|
| **Cache-Aside（Lazy Loading）** | アプリがキャッシュミス時に DB から取得して格納 | 汎用；読み取り多め |
| **Write-Through** | 書き込み時にキャッシュと DB を同時更新 | 整合性重視 |
| **Write-Behind（Write-Back）** | 書き込みをキャッシュに行い，非同期で DB に反映 | 書き込み性能重視 |
| **Read-Through** | キャッシュが DB 取得を代理（アプリは透過的） | ORM 組み込み等 |

### 無効化（Invalidation）

キャッシュの難問は「いつ無効化するか」で，主なアプローチは：
- **TTL（Time To Live）**: 有効期限で自動失効
- **イベント駆動**: DB 変更時に明示的に削除・更新
- **バージョニング**: キーにバージョン番号を付与し古いデータを参照させない

### 適用レイヤ

| レイヤ | 具体例 |
|--------|--------|
| CPU 内蔵 | L1/L2/L3 キャッシュ（[[CPU]] 参照） |
| アプリケーション内 | インメモリキャッシュ，メモ化 |
| 分散キャッシュ | Redis, Memcached（[[KVストア]]） |
| HTTP | ブラウザキャッシュ，CDN |
| DB | クエリキャッシュ，バッファプール |

### 指標

- **ヒット率**（Hit Rate）= キャッシュから返せた割合，高いほど効果的
- **キャッシュスタンプ**（Thundering Herd）: TTL 切れが集中してバックエンドに殺到する問題

## ポイント

- 「キャッシュには2つの難問がある：無効化とネーミング」（Phil Karlton）
- TTL を短くしすぎると効果が薄く，長すぎると陳腐化リスクが高まる
- [[REST API]] の GET レスポンスは HTTP ヘッダ（Cache-Control, ETag）でキャッシュを制御できる
