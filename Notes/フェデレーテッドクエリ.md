## 概要

データを物理的に移動させることなく，複数の異なるデータソースにまたがってクエリを実行する機能．データをインポートせずに外部データソースに対して直接クエリを発行し，結果を統合して取得できる．

## 詳細

### 基本概念

- データをコピー・移動せずに外部ストアへクエリを送信
- クエリエンジンがフェデレーション層として各データソースに接続
- 結果を統合してユーザに返す

「データエンジニアリングの基礎」では，変換処理の手法の一つとして[[マテリアライズドビュー]]・[[データ仮想化]]とともに言及される．

### クラウドサービスでの実装

#### Google BigQuery

```sql
-- 外部データソースへのフェデレーテッドクエリ例
SELECT *
FROM EXTERNAL_QUERY(
  'project.region.connection',
  'SELECT id, name FROM cloud_sql_table'
);
```

- Cloud SQL，Spanner，AlloyDB，Bigtable に対応
- BigQuery Connection API で接続を管理
- データをロードせずに即時クエリ実行

#### Amazon Athena

- Athena Query Federation SDK でカスタムデータソースに対応
- DynamoDB，RDS，Redshift，S3 など多様なソースに接続可能

#### Amazon Redshift

- 横串検索（Federated Query）でオペレーションDB・データウェアハウス・データレイクを横断クエリ

#### Databricks（Lakehouse Federation）

- Unity Catalog を用いてクエリフェデレーションを管理
- Snowflake，BigQuery，MySQL 等との接続をサポート

### メリット・デメリット

| | メリット | デメリット |
|---|---|---|
| データ移動 | 不要（リアルタイムアクセス） | なし |
| 鮮度 | 常に最新データ | - |
| パフォーマンス | - | 外部ソースのネットワーク遅延の影響を受ける |
| コスト | ETL コストが不要 | クエリごとに外部ソースへの通信コスト |

### データ仮想化との違い

[[データ仮想化]]が論理的な統合レイヤを提供するのに対し，フェデレーテッドクエリは SQL クエリレベルでのアドホックな外部データアクセスを指す．概念的に重複する部分が多い．

## ポイント

- データを移動せずにリアルタイムアクセスできるためパイプラインの簡素化に有効
- 外部ソースの応答速度がボトルネックになりやすい
- 結果をキャッシュする際は [[マテリアライズドビュー]] との組み合わせが有効
