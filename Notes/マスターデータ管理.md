## 概要

マスターデータ管理（MDM: Master Data Management）は，組織内の「顧客・製品・従業員・拠点」などの核心的なビジネスエンティティ（マスターデータ）について，唯一の正式なレコード（Single Source of Truth）を作成・維持・管理するプロセスと技術の体系．複数のシステムに散在する重複・矛盾したデータを統合し，企業全体で一貫した高品質なマスターデータを確保することで，業務効率と意思決定の精度を向上させる．

## 詳細

### マスターデータとは

マスターデータとは，企業の業務プロセスを横断して共有される「変化の少ないコアデータ」．トランザクションデータ（受注・売上等）の文脈情報として機能する．

| マスターデータの種類 | 例 |
|---------------------|-----|
| **顧客マスター** | 顧客ID，名称，住所，連絡先，与信情報 |
| **製品マスター** | 商品コード，名称，カテゴリ，価格，仕様 |
| **従業員マスター** | 社員番号，部署，役職，スキル |
| **拠点マスター** | 店舗コード，住所，担当地域 |
| **勘定科目マスター** | 科目コード，名称，集計ルール |

### MDM が解決する問題

MDM がない状態では，システムをまたいでマスターデータが不整合を起こす:

```
営業システム: 顧客名「株式会社ABC」
会計システム: 顧客名「ABC(株)」
CRM:         顧客名「ABC株式会社」
→ 同一顧客なのに3つのレコードが存在し，統計・請求・分析がずれる
```
例えば，これでフィルターをかけて重複を判断しようとしても，判断されない問題がある(準重複という)．
84% の企業が「不正確・重複したデータ」による業務非効率に悩んでいるとされる．

### MDM の実装スタイル

| スタイル | 説明 | 特徴 |
|---------|------|------|
| **集中型（Registry）** | 各システムのマスターをリアルタイムで参照・突合 | 既存システムに変更少．即時反映は難しい |
| **共有型（Consolidation）** | 各システムから定期的にデータを集約してゴールデンレコードを作成 | バッチ方式．分析用途に適する |
| **統合型（Coexistence）** | MDM ハブでマスターを管理し，各システムへ配信 | ハブが唯一の真実のソース．高品質だが実装コスト高 |
| **中央集権型（Centralized）** | 全システムが MDM ハブのみを参照 | 最も厳格．移行コストが最大 |

### ゴールデンレコード

複数ソースの重複・矛盾データを統合・クレンジングして作成する「最高品質の単一マスターレコード」．作成プロセス:

1. **識別（Entity Resolution）**: 異なる表記の同一エンティティを突合
2. **重複排除（Deduplication）**: 重複レコードを検出・マージ
3. **クレンジング**: 表記統一・欠損補完・バリデーション
4. **ゴールデンレコード生成**: 最信頼性の高い属性値を選択してマージ

### 主要ツール（2026）

| ツール | 提供元 | 特徴 |
|--------|--------|------|
| **Informatica MDM** | Informatica | エンタープライズ向け．AI（CLAIRE GPT）による自動マッチング |
| **SAP Master Data Governance** | SAP | SAP エコシステムとの統合に強み |
| **Stibo Systems STEP** | Stibo | 製品マスター（PIM）特化 |
| **IBM InfoSphere MDM** | IBM | 大規模エンタープライズ向け |
| **Oracle MDM** | Oracle | Oracle エコシステム統合 |

### DMBOK における位置づけ

DAMA-DMBOK では MDM は「データ統合と相互運用性」知識領域に含まれる．[[データカタログ]] によるデータディスカバリー・[[データ品質]] 管理との連携が重要．

### 2026 年の動向

- AI/ML によるエンティティ解決・重複検出の自動化が標準機能化
- クラウドネイティブ MDM（SaaS）への移行が加速
- データメッシュ環境での「分散 MDM」（各ドメインが独自マスターを管理しつつ連邦ガバナンスで統合）のパターンが台頭
- デジタルツイン・Product 360 など製品マスターのリアルタイム管理ニーズ増加

## ポイント

- 顧客・製品・従業員等のコアデータについて「唯一の正式レコード」を組織全体で共有する
- 重複・矛盾した散在データを集約→クレンジング→ゴールデンレコード生成のプロセスで統合
- 集中型・共有型・統合型・中央集権型の4スタイルから業務要件・移行コストで選択
- 実装コストは高いが，データ不整合による業務非効率（配送ミス・請求エラー・分析ずれ）の解消効果は大きい
- 2026 年は AI 自動マッチング・クラウド MDM への移行・分散 MDM パターンが主要トレンド
