## 概要

イベントストリーミングは，システム内で発生するイベント（状態変化の記録）を継続的に生成・キャプチャし，リアルタイムで処理・配信するデータ処理パラダイムである．[[Apache Kafka]] が事実上の標準プラットフォームとして普及しており，Fortune 100企業の80%超が利用している．

## 詳細

### 基本概念

| 概念 | 説明 |
|------|------|
| **イベント** | 「何かが起きた」という不変の記録（例：ユーザーがログインした） |
| **トピック** | イベントを分類するカテゴリ（ログのカテゴリ名のようなもの） |
| **パーティション** | トピックを並列処理するための分割単位 |
| **プロデューサー** | イベントを書き込むアプリケーション |
| **コンシューマー** | イベントを読み取って処理するアプリケーション |
| **コンシューマーグループ** | 並列処理のためのコンシューマーの集合 |

### バッチ処理との違い

| 観点 | バッチ処理 | イベントストリーミング |
|------|------------|----------------------|
| データ取り込み | 定期的（時間・日次） | 継続的（ミリ秒〜秒単位） |
| レイテンシ | 高い（分〜時間） | 低い（ミリ秒〜秒） |
| 用途 | 定期レポート，ETL | リアルタイム監視，不正検知 |

### [[Apache Kafka]] のアーキテクチャ

```
プロデューサー群
  → [Kafka Cluster: トピック→パーティション（永続ログ）]
    → コンシューマー群（グループA: 分析）
    → コンシューマー群（グループB: 通知）
    → コンシューマー群（グループC: DWH取り込み）
```

- メッセージはディスクに永続化され，複数コンシューマーが独立して読める
- オフセットで「どこまで読んだか」を管理し，巻き戻し再処理も可能

### ストリーム処理フレームワーク

| 製品 | 特徴 |
|------|------|
| **Kafka Streams** | Kafkaと密結合したJavaライブラリ |
| **Apache Flink** | ステートフルな高度なストリーム処理 |
| **Apache Spark Structured Streaming** | バッチとストリームを統一したAPI |
| **Google Dataflow** | マネージドApache Beamランナー |

### 主な用途

- **リアルタイム不正検知**: 決済イベントをストリーム処理してパターン検出
- **ユーザー行動分析**: クリックストリームのリアルタイム集計
- **マイクロサービス連携**: サービス間の非同期イベント伝達
- **CDC パイプライン**: [[変更データキャプチャ]] と組み合わせてDBの変更をDWHに低遅延で反映

## ポイント

- イベントは不変（immutable）で，ログとして追記されるためトレーサビリティが高い
- メッセージの永続化と再処理能力は，パイプラインの障害復旧を容易にする
- ストリーミングとバッチを統合した「Lambda アーキテクチャ」や「Kappa アーキテクチャ」で使われる
- スキーマ管理（Schema Registry）を導入しないとスキーマ変更で下流が壊れるリスクがある
