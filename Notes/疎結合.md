## 概要

疎結合（Loose Coupling）とは、ソフトウェアシステムのコンポーネント間の依存関係を最小限に抑える設計原則。各コンポーネントが独立して変更・テスト・デプロイ可能となり、保守性・拡張性・テスト容易性を高める。対義語は密結合（Tight Coupling）。

## 詳細

### 疎結合と密結合の違い

**密結合（Tight Coupling）** では、あるクラスが別のクラスの具体的な実装に直接依存する。依存先を変更すると、依存元も修正が必要になる。

**疎結合（Loose Coupling）** では、コンポーネントがインターフェース（抽象）のみに依存し、具体的な実装に依存しない。これにより依存先の実装を自由に差し替えられる。

```python
# 密結合の例: 具体クラスに直接依存
class OrderService:
    def __init__(self):
        self.db = PostgreSQLDatabase()  # 具体実装に依存

# 疎結合の例: インターフェースに依存
class OrderService:
    def __init__(self, db: DatabaseInterface):  # 抽象に依存
        self.db = db
```

### 疎結合を実現する手法

#### 1. 依存性注入（Dependency Injection）

コンポーネントが必要とする依存オブジェクトを外部から注入する。コンストラクタ注入が最も一般的。[[オブジェクト指向プログラミング]]の SOLID 原則のうち「依存性逆転の原則（DIP）」を実現する中核的手法。

#### 2. インターフェースと抽象化

具体的な実装ではなくインターフェース（抽象クラス）を介して通信する。呼び出し側は実装の詳細を知る必要がない。

#### 3. イベント駆動アーキテクチャ

コンポーネント間を直接呼び出すのではなく、メッセージキュー（Kafka、RabbitMQ 等）やイベントバスを介して非同期通信する。発行側と購読側が互いの存在を知る必要がなく、最も疎結合な関係を構築できる。

#### 4. API による分離

サービス間通信を REST API や gRPC 等の明確なインターフェースで行う。[[マイクロサービスアーキテクチャ]]の基本原則であり、[[ベゾスAPI司令]]が Amazon で徹底させたアプローチ。

### 疎結合のレベル

疎結合はシステムの様々なレベルで適用される:

| レベル | 手法 | 例 |
|--------|------|-----|
| クラス間 | DI、インターフェース | Strategy パターン |
| モジュール間 | 公開 API の制限、パッケージ分離 | モジュラーモノリス |
| サービス間 | REST/gRPC、メッセージキュー | [[マイクロサービスアーキテクチャ]] |
| システム間 | イベント駆動、非同期メッセージング | Pub/Sub、Webhook |

### 関連する設計原則

- **単一責任の原則（SRP）**: 各モジュールの責任を1つに限定し、独立性を高める
- **関心の分離（Separation of Concerns）**: 異なる関心事を異なるモジュールに分離する
- **情報隠蔽（Information Hiding）**: 内部実装を隠蔽し、公開インターフェースのみ提供する
- **開放閉鎖の原則（OCP）**: 拡張に開き、修正に閉じる設計

### メリットとトレードオフ

**メリット:**
- 変更の影響範囲を局所化できる（保守性向上）
- コンポーネントを独立してテスト可能（テスト容易性向上）
- 新機能の追加時に既存コードの変更が最小限（拡張性向上）
- チームの独立した並行開発が可能

**トレードオフ:**
- 初期設計の複雑性が増す（適切な抽象化レベルの決定が難しい）
- 通信オーバーヘッドが発生する（特にサービス間通信）
- 間接参照が増えることでデバッグが難しくなる場合がある
- 過度な疎結合は不要な複雑性を生む（YAGNI 原則とのバランス）

## ポイント

- 疎結合の本質は「具体実装ではなく抽象に依存する」こと
- DI（依存性注入）は疎結合を実現する最も基本的な手法
- マイクロサービスは疎結合をサービスレベルで適用したアーキテクチャ
- イベント駆動アーキテクチャは最も疎結合な通信パターンを提供する
- 適切な粒度の疎結合が重要であり、過度な分離は逆に複雑性を増す
