## 概要

クエリオプティマイザ（Query Optimizer）は，SQL クエリを受け取り，最も効率的な実行計画（実行順序・アクセス方式・結合アルゴリズム等）を選択するデータベースエンジンのコンポーネント．同じ結果を返す複数の実行方法の中から，推定コストが最小のものを選ぶ．

## 詳細

### オプティマイザの2つのアプローチ

| 種類 | 説明 | 現在の主流 |
|------|------|----------|
| **RBO**（Rule Based Optimizer） | 事前定義のルールに従って実行計画を決定（例：インデックスがあれば常に使う） | 廃止済みが多い（Oracle 10g 以降非推奨） |
| **CBO**（Cost Based Optimizer） | 統計情報（テーブル行数，列のカーディナリティ，ヒストグラム等）に基づきコストを見積もり最小コスト計画を選択 | 現在の標準 |

### クエリ最適化のプロセス

```
SQL テキスト
  ↓ パース
構文ツリー（Abstract Syntax Tree）
  ↓ 論理計画（Logical Plan）生成
等価変換・述語プッシュダウン・結合順序の決定
  ↓ 物理計画（Physical Plan）生成
インデックス選択・結合アルゴリズム選択・並列化判断
  ↓ 実行計画（Execution Plan）
最小推定コストの計画を実行エンジンに渡す
```

### 主な最適化技術

| 技術 | 説明 |
|------|------|
| **述語プッシュダウン** | WHERE 条件を可能な限り早い段階（スキャン時）で適用してデータ量を削減 |
| **射影プッシュダウン** | SELECT で参照する列のみを読み込む |
| **結合順序の最適化** | 小さいテーブルを先にフィルタリングして結合コストを下げる |
| **インデックス選択** | フルスキャン vs インデックススキャン のコストを比較 |
| **結合アルゴリズム選択** | Nested Loop / Hash Join / Merge Join をコストで選択 |
| **パーティションプルーニング** | 条件に合致しないパーティションをスキャン対象から除外 |

### 統計情報の重要性

CBO は統計情報の精度に依存する：
- **PostgreSQL**: `ANALYZE` コマンドで `pg_statistic` を更新
- **MySQL**: `ANALYZE TABLE` でインデックス統計を更新
- 統計情報が古いと誤った実行計画が選ばれ，性能劣化の原因になる

### 実行計画の確認

```sql
-- PostgreSQL / MySQL
EXPLAIN SELECT ...;

-- 実際の実行時間・行数も計測
EXPLAIN ANALYZE SELECT ...;
```

EXPLAIN の出力でスキャン方式（Seq Scan / Index Scan / Bitmap Scan 等），結合方式，推定行数を確認できる．

### 適応的クエリ実行（AQE）

[[Apache Spark]] 3.0 で導入された **Adaptive Query Execution（AQE）**：
- シャッフル後の実際のデータサイズを見て，実行中に計画を再最適化
- 推定が外れた場合でも Broadcast Join への切り替え等を動的に実施

## ポイント

- 統計情報が陳腐化すると実行計画が劣化する → 定期的な `ANALYZE` が重要
- `EXPLAIN ANALYZE` でボトルネックを特定するのがチューニングの第一歩
- インデックスを作るだけでは不十分で，オプティマイザがそれを選ぶ条件（選択性）も重要
- [[OLAP]] では列指向ストレージ・パーティションプルーニング・ベクトル処理との組み合わせが最適化の要
