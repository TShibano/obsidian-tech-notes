## 概要

データマート（Data Mart）とは、企業に蓄積された膨大なデータから、特定の部門や業務目的に応じて必要な情報のみを抽出・加工した小規模なデータベースである。[[データウェアハウス]]が企業全体のデータを網羅的に管理するのに対し、データマートは部門ごとの分析ニーズに特化して最適化された「データの小売店」として機能する。

## 詳細

### データマートの3つの種類

| 種類 | 説明 | メリット | デメリット |
|------|------|----------|-----------|
| **従属型（Dependent）** | [[データウェアハウス]]を基盤として構築。DWH 内の統合データを抽出・加工して利用 | データの一貫性が保たれ、全社的なガバナンスを維持できる。重複・不整合を防止 | DWH が前提のため構築コストが高い。DWH の変更に影響を受ける |
| **独立型（Independent）** | DWH を使用せず、個別のソースシステムから直接データを抽出して構築 | 構築が迅速でコストが低い。特定部門のニーズに素早く対応可能 | データのサイロ化が起きやすい。部門間でのデータ不整合のリスク |
| **ハイブリッド型（Hybrid）** | DWH と外部ソースの両方からデータを収集。従属型と独立型の特徴を併せ持つ | 全社的なデータ活用と部門固有のニーズの両立が可能 | 設計・運用が複雑になりやすい |

### データマートと DWH の違い

| 観点 | データマート | [[データウェアハウス]] |
|------|-------------|---------------------|
| 範囲 | 特定部門・業務に限定 | 企業全体を横断 |
| データ量 | 小〜中規模 | 大規模 |
| 対象ユーザー | 部門の業務担当者・アナリスト | データエンジニア・全社アナリスト |
| 構築期間 | 短期間 | 長期間 |
| コスト | 比較的低コスト | 高コスト |
| 分析の柔軟性 | 特定目的に最適化 | 汎用的 |

### 構築の5ステップ

1. **設計**: ビジネス要件と技術要件を洗い出し、データ構造を設計する。現場ユーザーが必要なデータに素早くアクセスできるよう配慮する
2. **構築**: データベース環境を構築する。クラウド DWH を利用する場合はその上にビューやテーブルとして作成する
3. **移入**: ソースシステムや DWH からデータを収集し、品質チェック・クレンジングを行ってから移行する
4. **アクセス**: BI ツールや分析ツールからデータマートに接続し、ダッシュボードやレポートを構築する
5. **管理**: データの鮮度維持（定期的な更新）、アクセス権限管理、品質監視を継続的に行う

### よく用いられるツール

データマートは独立した製品というよりも、DWH やクラウドプラットフォーム上に構築されることが一般的である。

#### クラウド DWH 上での構築

| ツール | 特徴 |
|--------|------|
| **BigQuery**（Google Cloud） | ELT 機能が豊富。ビューやマテリアライズドビューでデータマートを効率的に構築可能。Looker Studio との連携 |
| **Snowflake** | Virtual Warehouse による部門別のコンピュートリソース分離。データシェアリングで部門横断の活用も容易 |
| **Amazon Redshift** | AWS 分析エコシステムとの統合。スキーマ単位での論理的なデータマート分離 |
| **dbt** | SQL ベースのデータ変換ツール。DWH 上でデータマートのモデリング・テスト・ドキュメント化を一元管理 |

#### BI ツール連携

| ツール | 特徴 |
|--------|------|
| **Looker** | LookML によるセマンティックレイヤーで論理的なデータマートを定義。BigQuery との緊密な連携 |
| **Tableau** | データ抽出機能（Hyper）によるローカルデータマートの構築。セルフサービス分析に強み |
| **Power BI** | DirectQuery や Import モードでデータマートに接続。DAX による柔軟な集計定義 |
| **Metabase** | OSS の BI ツール。シンプルなセットアップで小規模なデータマートの可視化に適する |

### 最新動向（2025〜2026年）

#### セルフサービス BI の普及

営業やマーケティングなど現場のビジネスユーザーが IT 部門に依存せず自らデータを分析できる「セルフサービス BI」が主流となり、データマートの重要性が増している。分析のスピードが飛躍的に向上している。

#### セマンティックレイヤーの台頭

物理的なテーブルとしてデータマートを作成するのではなく、Looker の LookML や dbt の Semantic Layer のような**セマンティックレイヤー**で論理的なデータマートを定義するアプローチが広まっている。物理テーブルの増殖を防ぎ、メトリクスの定義を一元管理できる。

#### DWH との境界の曖昧化

クラウド DWH の性能向上により、データマートとして物理的にデータをコピーせずとも、DWH 上のビューやマテリアライズドビューで十分なパフォーマンスが得られるケースが増えている。論理的なデータマートへの移行が進んでいる。

#### データの民主化とガバナンスの両立

データの民主化が進む一方で、データマートのサイロ化を防ぐために、データカタログやデータ品質ツールとの連携によるガバナンス強化が重要課題となっている。

## ポイント

- データマートは特定の部門・業務目的に特化した小規模なデータ集約基盤
- 従属型・独立型・ハイブリッド型の3種類があり、従属型が最もデータの一貫性を保てる
- クラウド DWH（BigQuery、Snowflake、Redshift）上にビューやテーブルとして構築するのが一般的
- dbt によるデータモデリングとセマンティックレイヤーによる論理的なデータマートがトレンド
- [[データ基盤]]の3層構造において最も利用者に近い層に位置し、BI ツールと直接接続される
