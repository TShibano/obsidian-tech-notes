## 概要

モダンデータウェアハウス（Modern Data Warehouse）とは、クラウドネイティブなアーキテクチャを基盤とし、ストレージとコンピュートの分離、ELT パイプライン、半構造化データ対応、AI/ML 統合などを特徴とする次世代のデータウェアハウス。従来の[[リレーショナルデータウェアハウス]]が持つスケーラビリティやデータ多様性の課題を解決し、クラウド上でのデータ分析基盤として主流となっている。

## 詳細

### 従来型 DWH との違い

| 観点 | 従来型（リレーショナル）DWH | モダン DWH |
|------|--------------------------|-----------|
| **インフラ** | オンプレミス RDBMS | クラウドネイティブ |
| **スケーリング** | ストレージ・コンピュート一体 | ストレージとコンピュートを独立にスケール |
| **データ取り込み** | ETL（変換してからロード） | ELT（ロードしてから変換） |
| **対応データ型** | 構造化データのみ | 構造化＋半構造化（JSON, Avro 等） |
| **課金モデル** | 固定ライセンス | 従量課金（使った分だけ） |
| **運用** | DBA によるチューニング・管理 | サーバーレス/自動最適化 |
| **AI/ML** | 外部ツールが必要 | SQL から直接 ML モデルを構築・推論 |

### コアアーキテクチャ

#### ストレージとコンピュートの分離

モダン DWH の最も重要なアーキテクチャ上の特徴。データの保存（ストレージ）と処理（コンピュート）を独立したリソースとして管理することで：

- **コスト最適化**: データ量の増加とクエリ負荷を独立にスケール可能
- **ワークロード分離**: ETL、BI クエリ、アドホック分析を別々のコンピュートリソースで実行し、互いに干渉しない
- **弾力的スケーリング**: 需要に応じてコンピュートリソースを自動的に増減

#### ELT パイプライン

従来の ETL（Extract → Transform → Load）から ELT（Extract → Load → Transform）へ移行。生データをまず DWH にロードし、DWH 自身のコンピュートパワーで変換処理を行う。

- DWH の MPP エンジンを活用した高速な変換処理
- [[dbt]] が ELT の変換層（T）として事実上の標準ツールに
- データの鮮度が向上し、変換ロジックの変更が容易

### 主要プラットフォーム（2026年）

#### Snowflake

- **アーキテクチャ**: 共有ディスク+共有ナッシングのハイブリッド
- **特徴**: Virtual Warehouse によるワークロード分離、データシェアリング/マーケットプレイス、Snowflake Cortex による SQL からの AI/ML 利用
- **戦略**: AI モデルをデータベース内で直接実行し、アナリストが SQL 環境から高度な分析を実行可能に

#### BigQuery（Google Cloud）

- **アーキテクチャ**: 完全サーバーレス。クラスタ管理不要
- **特徴**: ペタバイト規模のクエリを従量課金で実行。BigQuery ML で SQL から ML モデル構築。Looker との統合
- **差別化**: Google の Dremel エンジンによるカラムナーストレージとツリーベースの分散クエリ実行

#### Amazon Redshift（AWS）

- **アーキテクチャ**: RA3 インスタンスでストレージとコンピュートを分離。Redshift Serverless も提供
- **特徴**: AWS エコシステム（S3、Glue、Kinesis、SageMaker）との緊密な統合。Redshift ML による ML 推論
- **差別化**: S3 上のデータに対する Redshift Spectrum による直接クエリ

#### Azure Synapse Analytics（Microsoft）

- **アーキテクチャ**: サーバーレスプールと専用プールの両モード
- **特徴**: Power BI とのネイティブ連携。Azure ML との統合。Apache Spark プールによるビッグデータ処理
- **差別化**: Microsoft エコシステム全体との統合

#### Databricks SQL

- **アーキテクチャ**: [[データレイクハウス]]上の SQL 分析エンジン。[[Delta Lake]] ベース
- **特徴**: オープンフォーマット（Delta Lake、Apache Iceberg）。Apache Spark との統合。Unity Catalog によるガバナンス
- **差別化**: DWH とデータレイクの統合。ML/AI ワークロードとのシームレスな連携

### 2026年のトレンド

#### AI ネイティブ DWH

各プラットフォームが SQL から直接 AI/ML を実行する機能を強化。アナリストがデータサイエンティストに依存せずに予測分析・自然言語クエリを実行できる環境が整備されつつある。

#### オープンテーブルフォーマットへの収束

[[Apache Iceberg]] が業界標準として収束する傾向。Snowflake、BigQuery、Redshift がいずれも Iceberg をネイティブサポートし、ベンダーロックインの低減が進む。

#### DWH とレイクハウスの融合

DWH ベンダーがレイクハウス機能を取り込み、レイクハウスベンダーが SQL DWH 機能を強化。両者の境界がますます曖昧になっている。

#### リアルタイム分析

ストリーミングデータの取り込みとリアルタイムクエリへの対応が進み、バッチ処理とストリーム処理の統合が進展している。

## ポイント

- ストレージとコンピュートの分離がモダン DWH の最大の革新。コスト最適化と弾力的スケーリングを実現
- ETL から ELT へのシフトにより、[[dbt]] が変換層の標準ツールとして定着
- Snowflake / BigQuery / Redshift の3強体制だが、Databricks SQL がレイクハウスからのアプローチで追い上げ
- AI/ML 統合とオープンテーブルフォーマット対応が2026年の主要テーマ
- [[データレイクハウス]]との融合が進み、「DWH か レイクハウスか」という二者択一ではなくなりつつある
