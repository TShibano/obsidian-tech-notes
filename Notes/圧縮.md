## 概要

データ圧縮は，元のデータを損失なく（可逆圧縮）または許容範囲の損失で（非可逆圧縮）より少ないビット数で表現する技術．ストレージコスト削減とネットワーク転送時間の短縮が主目的．

## 詳細

### 可逆圧縮 vs 非可逆圧縮

| 種別 | 説明 | 用途 |
|------|------|------|
| 可逆（Lossless） | 元データを完全復元可能 | テキスト, DB, バイナリ |
| 非可逆（Lossy） | 一部情報を切り捨て | 画像（JPEG）, 音声（MP3）, 動画（H.264） |

### 主な可逆圧縮アルゴリズム

| アルゴリズム | 圧縮率 | 速度 | 特徴 |
|------------|--------|------|------|
| [[gzip]] | 高 | 中 | HTTP 転送の事実上の標準 |
| [[bzip2]] | 高め | 遅め | CPU コスト高，高圧縮率 |
| [[Snappy]] | 中 | 非常に高速 | Google 製，低レイテンシ優先 |
| LZ4 | 中 | 最速 | 高速解凍が必要な場面 |
| Zstandard（zstd） | 高 | 高速 | Meta 製，gzip 代替として普及中 |
| LZMA / xz | 最高 | 低速 | アーカイブ・配布用途 |

### データ処理での圧縮

- **列指向フォーマット**（[[Apache Parquet]] 等）は列単位の高圧縮率を実現
  - 同型データが連続するため Run-Length Encoding（RLE）や辞書圧縮が効果的
- **ストリーム処理**では Snappy / LZ4 が解凍速度重視で採用される
- **HTTP 転送**では gzip / Brotli が標準的（`Accept-Encoding` ヘッダ）

### 圧縮率と速度のトレードオフ

```
高圧縮率 ←──────────────────── 高速
LZMA  bzip2  gzip  zstd  Snappy  LZ4
```

## ポイント

- ストレージコストとデコード CPU コストのトレードオフを意識する
- 分析系ワークロード: Parquet + Snappy/zstd が定番
- リアルタイム通信: LZ4/Snappy（速度優先）
- 配布・バックアップ: zstd/xz（圧縮率優先）
