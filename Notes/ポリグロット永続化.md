## 概要

ポリグロット永続化（Polyglot Persistence）は，システム内のデータの特性・用途に応じて複数の異なるデータストア技術を使い分ける設計パターンである．単一のデータベースに全データを押し込むのではなく，適材適所のデータベースを選択することでパフォーマンス・スケーラビリティ・開発効率を最大化する．

## 詳細

### 背景と概念

従来のシステムではRDBMS（リレーショナルデータベース）に全データを集約するアーキテクチャが主流だったが，Webスケールのアプリケーション普及とNoSQLの台頭により，データの特性に合わせた複数DBの組み合わせが現実的な選択肢となった．

「Polyglot」は「複数言語を話せる」という意味で，「複数のデータストア言語（技術）を使いこなす」というコンセプトを表す．Martin Fowler と Pramod Sadalage が著書「NoSQL Distilled」（2012年）で提唱・普及させた．

### 主なデータストアタイプの使い分け

| データストア種別 | 得意なデータ | 代表製品 | 用途例 |
|---------------|------------|---------|-------|
| **RDBMS** | 構造化・トランザクション | PostgreSQL，MySQL | 注文・会計・マスターデータ |
| **ドキュメントDB** | 半構造化・スキーマ柔軟 | MongoDB，Firestore | 商品カタログ，ユーザープロファイル |
| **KV（キーバリュー）** | シンプルな高速アクセス | Redis，DynamoDB | セッション，キャッシュ，カート |
| **グラフDB** | エンティティ間の関係 | Neo4j，Amazon Neptune | SNS，レコメンド，知識グラフ |
| **カラム型** | 大量時系列・集計 | Cassandra，HBase | IoTデータ，ログ，タイムライン |
| **全文検索** | テキスト検索 | Elasticsearch，OpenSearch | 商品検索，ログ検索 |
| **時系列DB** | 時刻付き計測値 | InfluxDB，TimescaleDB | 監視メトリクス，センサーデータ |

### マイクロサービスとの親和性

[[マイクロサービスアーキテクチャ]] はポリグロット永続化の主要な適用場面である．各マイクロサービスが自分のデータストアを所有（「データの分権化」）することで，互いに独立してスケーリング・デプロイできる．

```
注文サービス → PostgreSQL（ACID トランザクション）
商品サービス → MongoDB（柔軟なスキーマ）
セッション管理 → Redis（高速 KV）
レコメンドエンジン → Neo4j（グラフ関係）
検索サービス → Elasticsearch（全文検索）
```

### メリット

- 各データの特性に最適化されたストレージを選択できる
- サービスごとに独立してスケーリング・バージョンアップが可能
- 単一DBの障害がシステム全体に波及するリスクを分散

### デメリット・課題

- 複数 DB の運用・監視・バックアップが複雑化
- サービス間をまたぐトランザクションの実現が困難（分散トランザクション，Saga パターンで対応）
- チームに複数の DB 技術スキルが求められる
- データ整合性の担保が各サービスの責任になる

## ポイント

- 「全データを1つの DB で管理」から「データの特性に合わせて DB を選ぶ」という発想の転換
- [[マイクロサービスアーキテクチャ]] と相性が良く，サービスごとに最適なストアを持つ
- 複雑化のコストと最適化のメリットのバランスを慎重に評価する必要がある
- [[ポリグロットデータストア]] は各ストアタイプの特性と選定基準に焦点を当てた概念
