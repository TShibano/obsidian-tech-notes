## 概要

データ品質（Data Quality）とは、データが正確・完全・一貫・適時であり、目的に合った形で利用できる状態を指す概念・プラクティス。データパイプラインの信頼性を担保する上で中核となる分野であり、検証フレームワーク・監視ツール・組織的なプロセスの整備が必要となる。

## 詳細

### データ品質の6次元

| 次元 | 説明 | 例 |
|------|------|---|
| **正確性（Accuracy）** | データが実態を正しく反映しているか | 顧客の住所が実際の住所と一致する |
| **完全性（Completeness）** | 必要なデータが欠けていないか | 必須カラムに NULL がない |
| **一貫性（Consistency）** | 複数のシステム間でデータが矛盾しないか | CRMとDWHで顧客数が一致する |
| **適時性（Timeliness）** | データが必要な時に最新状態か | レポートが前日データを反映している |
| **一意性（Uniqueness）** | 重複レコードがないか | 顧客IDが主キーとして一意である |
| **妥当性（Validity）** | 定義されたルール・形式に沿っているか | 日付が `YYYY-MM-DD` 形式で有効な日付 |

### 主要ツール

#### Great Expectations

2017年にリリースされたOSSの代表的なデータ検証フレームワーク。データに対する「期待値（Expectations）」を定義し、実際のデータと照合する。

```python
import great_expectations as gx

context = gx.get_context()
datasource = context.sources.add_pandas("my_datasource")
asset = datasource.add_dataframe_asset("orders")

# Expectation の定義
batch = asset.build_batch_request(dataframe=df)
validator = context.get_validator(batch_request=batch)

validator.expect_column_values_to_not_be_null("order_id")
validator.expect_column_values_to_be_unique("order_id")
validator.expect_column_values_to_be_in_set("status", ["placed", "shipped", "completed"])

# 検証実行
result = validator.validate()
```

**特徴:**
- 詳細で柔軟な設定が可能
- データドキュメント（Data Docs）を自動生成
- Airflow, dbt, Prefect との統合

#### Soda

YAML ベースの宣言的な記述でデータ品質チェックを定義。操作が簡単で、監視・オブザーバビリティに強み。

```yaml
# soda/checks.yml
checks for orders:
  - row_count > 0
  - missing_count(order_id) = 0
  - duplicate_count(order_id) = 0
  - invalid_percent(status) < 1%:
      valid values: [placed, shipped, completed]
```

**特徴:**
- YAML で直感的に記述
- アラート・通知との連携が容易
- Soda Core（OSS）と Soda Cloud（SaaS）が存在

#### dbt Tests

dbt に組み込まれたテスト機能。`schema.yml` で定義する組み込みテスト（not_null, unique 等）とカスタム SQL テストが使える。変換と品質テストを同一ツールで管理できる。

#### Deequ

Amazon が開発した Scala/Spark ベースの大規模データ向け品質検証ライブラリ。Spark DataFrameに直接適用でき、統計的検証が得意。

### データ品質パイプラインの構成例

```
[データ取り込み] → [プロファイリング] → [品質チェック] → [アラート/ブロック] → [監視ダッシュボード]
     ↓                                        ↓
  Airbyte                            Great Expectations / Soda
                                              ↓
                                    NG の場合: Airflow タスク失敗 + 通知
```

### データプロファイリング

品質チェックの前提として、データの統計的特性を把握するプロファイリングが重要：
- カラムの欠損率・ユニーク率
- 値の分布・最小/最大/平均
- スキーマの変化（スキーマドリフト）の検出

## ポイント

- **シフトレフト**: データの問題を下流（レポート・分析）ではなく上流（パイプライン入口）で検出する
- **品質ゲート**: CI/CD パイプラインにデータ品質チェックを組み込み、品質基準を満たさないデータの下流への流入をブロック
- **スキーマドリフトの監視**: ソースシステムのスキーマ変更を自動検出し、パイプライン破損を防ぐ
- **ツールの使い分け**: 組み込みテストは dbt、詳細な統計検証は Great Expectations、監視は Soda という使い分けが多い
