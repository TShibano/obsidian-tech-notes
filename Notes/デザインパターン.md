## 概要

デザインパターンとは、ソフトウェア設計に繰り返し現れる問題に対する実績ある解決策を、再利用可能な形でまとめた設計知識の集合体である。1994年に出版された書籍『オブジェクト指向における再利用のためのデザインパターン』（著者4名を「GoF: Gang of Four」と呼ぶ）が最も有名で、23種類のパターンを体系化した。

## 詳細

### GoF デザインパターンの分類

GoF の 23 パターンは 3 カテゴリに分類される:

#### 1. 生成パターン（Creational Patterns）— 5種

オブジェクトの生成プロセスを抽象化し、生成方法をコードから分離する。

| パターン | 概要 |
|---------|------|
| **Abstract Factory** | 関連するオブジェクト群を生成するインターフェースを提供。具体クラスを指定せず製品ファミリを生成 |
| **Builder** | 複雑なオブジェクトを段階的に構築。同じ構築プロセスで異なる表現を生成できる |
| **Factory Method** | オブジェクト生成をサブクラスに委譲。親クラスが生成インターフェースのみ定義 |
| **Prototype** | 既存オブジェクトのコピー（クローン）から新しいオブジェクトを生成 |
| **Singleton** | クラスのインスタンスを1つに制限し、そのグローバルアクセス点を提供 |

#### 2. 構造パターン（Structural Patterns）— 7種

クラス・オブジェクトを組み合わせてより大きな構造を形成する。

| パターン | 概要 |
|---------|------|
| **Adapter** | 互換性のないインターフェースを持つクラスを協調させるラッパー |
| **Bridge** | 抽象部分と実装部分を分離し、それぞれを独立して変更できるようにする |
| **Composite** | オブジェクトをツリー構造に組み合わせ、単一オブジェクトと複合オブジェクトを同一視 |
| **Decorator** | オブジェクトを動的にラップして機能を追加。継承の代替 |
| **Facade** | 複雑なサブシステムに対して簡略化されたインターフェースを提供 |
| **Flyweight** | 多数の細粒度オブジェクトを共有して記憶域を節約 |
| **Proxy** | オブジェクトの代理を提供し、アクセス制御・キャッシュ・遅延初期化などを実現 |

#### 3. 振る舞いパターン（Behavioral Patterns）— 11種

オブジェクト間の通信・責任の配分・アルゴリズムを定義する。

| パターン | 概要 |
|---------|------|
| **Chain of Responsibility** | リクエストをオブジェクトのチェーンに沿って渡す。処理できるオブジェクトが受け取る |
| **Command** | リクエストをオブジェクトとしてカプセル化。undo/redo・キューイングが可能 |
| **Interpreter** | 言語の文法を定義し、インタープリタを提供 |
| **Iterator** | コレクションの要素に順番にアクセスする方法を提供。内部表現を露出しない |
| **Mediator** | オブジェクト間の複雑な通信を仲介オブジェクトに集約。疎結合を実現 |
| **Memento** | オブジェクトの内部状態を保存・復元する（undo 機能など） |
| **Observer** | オブジェクトの状態変化を、依存する全オブジェクトに自動通知（イベント駆動の基本） |
| **State** | オブジェクトの内部状態が変わると振る舞いが変わるように見せる |
| **Strategy** | アルゴリズムのファミリーをカプセル化し、実行時に切り替え可能にする |
| **Template Method** | アルゴリズムの骨格を親クラスで定義し、一部の手順をサブクラスに委譲 |
| **Visitor** | オブジェクト構造を変えずに、新しい操作を追加できるようにする |

### デザインパターンのメリット

- **共通語彙の提供**: チームで「Observerパターンを使おう」と言えば設計意図が即座に伝わる
- **変更容易性の向上**: パターンが示す疎結合・単一責任の原則を守ることで変更コストが下がる
- **再利用性の向上**: 実績ある解決策を再利用し、車輪の再発明を避けられる

### よく使われるパターン

**フロントエンド・UI**:
- Observer（イベントハンドラ、React の状態管理）
- Composite（ツリー状の UI コンポーネント）
- Decorator（HOC、ミドルウェア）

**バックエンド・アーキテクチャ**:
- Factory Method（DI コンテナ）
- Strategy（支払い方法・認証方式の切り替え）
- Facade（API ゲートウェイ）
- Proxy（キャッシュ、アクセス制御）

**データ処理**:
- Iterator（コレクションの走査）
- Template Method（ETL パイプラインの骨格）
- Command（Undo/Redo 機能）

### GoF パターンを超えて

GoF パターン以外にも以下の設計パターン群がある:

| 分類 | 例 |
|------|----|
| **アーキテクチャパターン** | MVC、MVVM、Clean Architecture、Hexagonal Architecture |
| **エンタープライズパターン** | Repository、Unit of Work、CQRS（P of EAA） |
| **分散システムパターン** | Circuit Breaker、Saga、Event Sourcing |
| **関数型パターン** | Monad、Functor（[[関数型プログラミング]]） |

## ポイント

- GoF の 23 パターンは「生成（5）・構造（7）・振る舞い（11）」の3カテゴリに分類
- パターンは「解法の名前と構造を共有する語彙」であり、適用すること自体が目的ではない
- 使いすぎ（オーバーエンジニアリング）に注意。シンプルな解法がある場合はパターンを使わない
- Observer と Strategy は特にシステム設計で頻出。Observer はイベント駆動・リアクティブシステムの基盤
- [[関数型プログラミング]] の普及で、Strategy パターンを関数・高階関数で表現するケースが増えている
