## 概要

責任共有セキュリティモデル（Shared Responsibility Model）は、クラウド環境におけるセキュリティ責任をクラウドプロバイダーと顧客の間で分担するフレームワーク。プロバイダーはクラウドインフラ自体のセキュリティ（Security of the Cloud）を担い、顧客はクラウド上で構築・運用するシステムのセキュリティ（Security in the Cloud）に責任を持つ。

## 詳細

### 基本概念

クラウドサービスでは、セキュリティの全責任がプロバイダーまたは顧客のどちらか一方にあるわけではない。サービスモデル（[[XaaS]]）に応じて責任の境界が変化する。

```
       プロバイダーの責任 ←──────────────────→ 顧客の責任

IaaS:  物理/ネットワーク/仮想化 │ OS, ミドルウェア, アプリ, データ
PaaS:  + OS, ランタイム         │ アプリ, データ
SaaS:  + アプリケーション        │ データ, アクセス管理
```

### サービスモデル別の責任分担

#### IaaS（Infrastructure as a Service）

顧客の責任が最も大きいモデル。

| 責任領域 | 担当 |
|---------|------|
| 物理インフラ（データセンター、ハードウェア） | プロバイダー |
| ネットワークインフラ | プロバイダー |
| ハイパーバイザー・仮想化層 | プロバイダー |
| OS のパッチ適用・設定 | **顧客** |
| ミドルウェア・ランタイム | **顧客** |
| アプリケーション | **顧客** |
| データの暗号化・分類 | **顧客** |
| ネットワーク設定（セキュリティグループ、NACL） | **顧客** |
| アイデンティティ・アクセス管理 | **顧客** |

#### PaaS（Platform as a Service）

プロバイダーの責任範囲が拡大し、顧客はアプリケーションとデータに集中。

| 責任領域 | 担当 |
|---------|------|
| インフラ + OS + ランタイム | プロバイダー |
| アプリケーションのセキュリティ | **顧客** |
| アクセスポリシー | **顧客** |
| データ暗号化 | **顧客** |

#### SaaS（Software as a Service）

プロバイダーがほぼすべてを管理するが、データとアクセスは常に顧客の責任。

| 責任領域 | 担当 |
|---------|------|
| インフラ + アプリケーション全体 | プロバイダー |
| アイデンティティ・アクセスガバナンス | **顧客** |
| データ分類・保護 | **顧客** |
| ユーザー管理 | **顧客** |

### 不変のルール: データは常に顧客の責任

IaaS・PaaS・SaaS いずれのモデルでも、**データの分類・暗号化・アクセス制御はプロバイダーに委任されない**。これが責任共有モデルにおける最も重要な定数。

### プロバイダー別の特徴

#### AWS

AWS は責任を明確に2つに分割:

- **Security of the Cloud**（AWS の責任）: データセンターの物理セキュリティ、ネットワーク、ハイパーバイザー、マネージドサービスの基盤
- **Security in the Cloud**（顧客の責任）: OS、ネットワーク設定、IAM、データ暗号化、アプリケーション

AWS は IAM（Identity and Access Management）を提供し、顧客が最小権限のアクセスポリシーを設定できるようにしている。

#### Azure

Microsoft は責任を3カテゴリに分類:

- **顧客が完全に管理する領域**: データ、エンドポイント、アカウント管理
- **Microsoft が管理する領域**: 物理インフラ、ネットワーク
- **共同で管理する領域**: サービスモデルに応じて変動（OS パッチ、ネットワーク制御等）

#### GCP

Google は「**Shared Fate（運命共同体）**」モデルを提唱:

- 従来の「責任分担」よりも協調的なアプローチ
- **セキュリティブループリント**: Google が規範的なセキュリティ設計を提供
- **デフォルトセキュア設定**: プロジェクト作成時からセキュリティが有効
- **リファレンスアーキテクチャ**: 推奨構成に従えば Google も共同でセキュリティに責任を持つ

### よくある誤解と事故

- **「クラウドにデータを置けば安全」**: プロバイダーはインフラを守るが、データのアクセス制御・暗号化は顧客の責任
- **S3 バケットの公開設定ミス**: 何度も大規模な情報漏洩事故が発生。IAM ポリシーとバケットポリシーの設定は顧客の責任
- **セキュリティグループの設定漏れ**: デフォルトで全ポート開放のまま運用し、不正アクセスを受けるケース
- **IAM 権限の過剰付与**: 最小権限の原則に反した広範なアクセス権付与

### ベストプラクティス

1. **最小権限の原則**: IAM ポリシーで必要最小限の権限のみ付与
2. **暗号化の徹底**: 保存時（at rest）と転送時（in transit）の両方で暗号化
3. **MFA の有効化**: 管理コンソール・API アクセスに多要素認証を必須化
4. **ログ・監査**: CloudTrail（AWS）、Cloud Audit Logs（GCP）等で操作ログを記録
5. **定期的なセキュリティ評価**: CIS ベンチマーク、Security Hub 等で自動チェック
6. **[[ゼロトラストセキュリティ]]の適用**: クラウド環境でもネットワーク位置を信頼しない設計

## ポイント

- クラウドのセキュリティは「プロバイダーに任せて終わり」ではない
- サービスモデル（IaaS/PaaS/SaaS）によって責任の境界が大きく変わる
- **データの保護は常に顧客の責任** — これが最も重要な原則
- GCP は「Shared Fate」として、従来の責任分担より協調的なモデルを提唱
- S3 バケット公開ミスなど、顧客側の設定ミスがクラウドセキュリティ事故の主要原因
