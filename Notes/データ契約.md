## 概要

データ契約（Data Contract）は，データの提供者（プロデューサー）と利用者（コンシューマー）の間で，データのスキーマ・品質・SLA・オーナーシップを明示的に定義した合意文書である．GoCardlessのAndrew Jonesが提唱した概念で，データ品質問題の根本原因である「上流の無断スキーマ変更」を防ぐことを目的とする．

## 詳細

### 何を定義するか

```yaml
# ODCS（Open Data Contract Standard）形式の例
apiVersion: v3.0.0
kind: DataContract
id: orders-v1
name: orders
version: 1.0.0
owner: data-engineering-team
description: 注文テーブルのデータ契約

schema:
  - name: order_id
    type: string
    required: true
    unique: true
  - name: amount
    type: number
    minimum: 0

quality:
  - type: sql
    query: "SELECT COUNT(*) FROM orders WHERE amount < 0"
    mustBe: 0

slaProperties:
  - property: latency
    value: 1h
```

### データ契約の構成要素

| 要素 | 内容 |
|------|------|
| **スキーマ** | カラム名，型，NULL可否，ユニーク制約など |
| **品質ルール** | 欠損率，重複率，値域チェックなどのSQLテスト |
| **SLA** | 更新頻度，遅延許容時間，可用性 |
| **オーナー** | データの責任者・問い合わせ先 |
| **バージョン** | 後方互換性の管理 |
| **セマンティクス** | カラムの意味・定義の説明 |

### ツール

| ツール | 説明 |
|--------|------|
| **datacontract-cli** | ODCS 形式のデータ契約を検証・テスト・ドキュメント生成する OSS CLI |
| **dbt** | データ契約YAMLから dbt の schema.yml を生成し，CI/CDに組み込める |
| **Soda** | データ品質テストを自動実行するツール |
| **Great Expectations** | Pythonベースのデータ品質テストフレームワーク |

### データ契約駆動のワークフロー

1. プロデューサーがデータ契約をYAMLで定義・コミットする
2. CI/CDでデータ契約の検証テストが自動実行される
3. コンシューマーは契約に基づいてパイプラインを構築する
4. スキーマ変更時は契約のバージョンアップが必要になる（破壊的変更を防ぐ）

### ODCS（Open Data Contract Standard）

PayPalがデータ契約仕様をオープンソース化し，Linux Foundation のプロジェクト「Bitol」として標準化が進んでいる．業界横断の互換性を高めることが目的．

## ポイント

- データ契約はコードと同様 Git で管理し，レビュー・バージョン管理する
- 「データメッシュ」のドメインオーナーシップと親和性が高い（ドメインが自ドメインのデータ契約を管理する）
- 契約なしのパイプラインは上流の変更で静かに壊れる；契約があれば CI で即座に検出できる
- 生成AIにデータ契約YAMLを渡すことで，パイプラインコードの自動生成が可能になる（AI駆動データエンジニアリング）
