## 概要

データ基盤とは、企業が保有するあらゆる種類のデータを収集・蓄積・管理・保護するための IT インフラストラクチャ全般を指す。データベース、ストレージ、データ連携ツール、セキュリティ技術などで構成され、データを安全かつ効率的に保管し、必要な時にアクセスできるようにすることを目的とする。特に分析用途に特化したものは「データ分析基盤」と呼ばれ、データを分析しやすい形に加工・整理し、BI ツールや機械学習モデルで活用できるようにする。

## 詳細

### 従来型のデータ基盤

#### ETL と DWH の誕生（1970〜1990年代）

1970年代に大型の集中化されたデータベースへの移行が始まり、1980年代後半にはリレーショナルデータベースが普及した。業務システムの増加に伴いデータの置き場所が分散し、データの集約・統合が課題となった。この課題を解決するために**データウェアハウス（DWH）**という概念が生まれ、データを整理・統合する **ETL（Extract, Transform, Load）**ツールが登場した。

従来の ETL プロセスはプログラミングにより実現されていたが、膨大な工数を要した。1990年代にはダウンサイジングとオープン化の流れで業務システムが増加し、データの分散がさらに進んだ。

#### 典型的な3層構造

従来のデータ分析基盤は、以下の3層構造が主流であった:

1. **[[データレイク]]**: 異なるソースから得られるデータをそのままの形（生データ）で一元的に保持する層
2. **[[データウェアハウス]]**: ビジネス上の意思決定に利用できるよう、データを構造化・変換して保持する層
3. **[[データマート]]**: 特定の部門や用途向けにデータを集約・加工した層

### モダンデータスタック（MDS）

2020年代に入り、クラウドネイティブなツールを組み合わせた**モダンデータスタック（Modern Data Stack）**が主流となった。従来型と比較して以下の特徴がある:

- **ELT 方式の標準化**: 従来の ETL（変換してからロード）ではなく、ELT（ロードしてから変換）方式が主流に。データをまずクラウド DWH に格納してから変換する
- **クラウドネイティブ**: 各コンポーネントが SaaS として提供され、スケーラビリティとコスト効率に優れる
- **ツールの専門化と分離**: データ統合、変換、可視化などの各機能が専門ツールとして独立し、ベスト・オブ・ブリードで組み合わせ可能

#### MDS の代表的な構成要素

| レイヤー | 役割 | 代表的ツール |
|----------|------|-------------|
| データ統合（Ingestion） | データソースからの収集・取り込み | Fivetran, Airbyte, TROCCO |
| ストレージ / DWH | データの蓄積・保管 | Snowflake, BigQuery, Redshift |
| データ変換 | データの加工・モデリング | dbt |
| データオーケストレーション | ワークフロー管理・スケジューリング | Apache Airflow, Dagster, Prefect |
| BI / 可視化 | データの分析・ダッシュボード | Looker, Tableau, Metabase |
| データカタログ | メタデータ管理・データ検索 | DataHub, Amundsen |
| データオブザーバビリティ | データ品質の監視 | Monte Carlo, Elementary |

### 最新トレンド（2025〜2026年）

#### レイクハウスアーキテクチャ

[[データレイク]]のスケーラビリティと[[データウェアハウス]]のトランザクション管理を統合した**レイクハウス**が台頭している。Apache Iceberg などのオープンテーブルフォーマットにより、データレイク上で DWH 相当の ACID トランザクションやスキーマ管理が可能になった。

#### データメッシュ

分散型データアーキテクチャとして**データメッシュ**が注目されている。各ドメイン（事業部門）がデータの所有権を持ち、「データをプロダクトとして扱う」という考え方に基づく。4つの原則で構成される:

1. **ドメインオーナーシップ**: 各部門が自身のデータに責任を持つ
2. **セルフサービスデータプラットフォーム**: 専門部門に依存せずデータにアクセスできる環境
3. **データ・アズ・ア・プロダクト**: データを製品として品質管理する
4. **フェデレーテッドガバナンス**: 全社共通のルールと各部門の自律性を両立

#### データファブリック

企業内外に散在するデータを場所や形式に関係なく統合・管理・活用するためのアーキテクチャ。メタデータの自動管理と AI による最適化が特徴で、データメッシュが組織論的アプローチであるのに対し、データファブリックは技術的アプローチと位置づけられる。

#### ストリーミングと CDC の普及

データ取り込み層が「バッチ + API」からストリーミング + CDC（Change Data Capture）へシフトしている。Debezium で RDB の変更を Kafka へ送信し、リアルタイムにデータを同期する構成が広まっている。

#### SQL と Python のハイブリッド変換

dbt の Python モデルと Polars の組み合わせにより、SQL と Python を用途に応じて使い分けるハイブリッド変換が一般化した。

#### AI 時代のデータ基盤

生成 AI の普及に伴い、データ基盤は「データを出すパイプ」から「価値を生む装置」へと進化が求められている。AI アプリケーションのトレーニングに必要なデータの品質管理やメタデータ統合が重要課題となっている。

## ポイント

- データ基盤はデータの収集・蓄積・管理のための IT インフラ全般を指し、分析特化型は「データ分析基盤」と呼ばれる
- 従来型は[[データレイク]] → [[データウェアハウス]] → [[データマート]]の3層構造が主流であった
- モダンデータスタックにより ELT 方式・クラウドネイティブ・ツール専門化が標準となった
- 最新トレンドとしてレイクハウス、データメッシュ、データファブリック、ストリーミング/CDC が注目されている
- 構築のベストプラクティスはスモールスタートと段階的拡張、ユースケース起点の設計
