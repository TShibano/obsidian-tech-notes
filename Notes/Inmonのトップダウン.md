## 概要

Inmon のトップダウンアプローチは Bill Inmon（「データウェアハウスの父」）が 1990 年代に提唱した DWH 設計手法．企業全体を対象とした中央集権的なエンタープライズデータウェアハウス（EDW）を**先に設計・構築**し，そこからデータマートを派生させる「トップダウン（上から下）」の設計思想．EDW は第三正規形（3NF）で設計され，「単一の真実のソース（Single Source of Truth）」を実現する．

## 詳細

### 設計思想

```
エンタープライズデータモデル（全社の分析）
         ↓ 設計
EDW（Enterprise Data Warehouse）
 ・3NF（正規化）でデータ整合性を保証
 ・Single Source of Truth
 ・ODS（Operational Data Store）経由でソース統合
         ↓ 派生
データマート A   データマート B   データマート C
 （売上分析）     （在庫管理）     （顧客分析）
 ※ EDW のサブセット
```

[[Kimballのボトムアップ]] が「データマートを先に作り後から統合」するのに対し，Inmon は「全社データモデルを先に設計してから部門向けマートを派生」する．

### 3NF（第三正規形）

Inmon の EDW は 3NF で設計されるため，データの重複がなく整合性が高い．

```
[正規化の例]
- 非正規形: 注文テーブルに顧客名・顧客住所を重複格納
- 3NF: 顧客テーブルを分離し，注文テーブルから外部キーで参照
```

| 3NF の利点 | 3NF の欠点 |
|-----------|-----------|
| データ整合性・冗長性なし | ジョイン数が多くクエリが複雑 |
| 更新・挿入が効率的 | BI ツールでの使いにくさ |
| データの単一真実ソース | 初期設計コストが高い |

### Corporate Information Factory (CIF)

Inmon が提唱した全体フレームワーク．以下のレイヤーで構成される:

| レイヤー | 役割 |
|---------|------|
| **ODS（Operational Data Store）** | ソースシステムからの近リアルタイム統合．オペレーショナルな用途 |
| **EDW** | 全社統合データ．3NF．分析・履歴管理の中核 |
| **データマート** | EDW のサブセット．部門・業務特化の非正規化ビュー |
| **Exploration Warehouse** | アドホック分析・データマイニング用 |

### Kimball との比較

| 観点 | Inmon（トップダウン） | [[Kimballのボトムアップ]] |
|------|---------------------|----------------------|
| 設計順序 | EDW → データマート | データマート → DWH |
| データモデル | 3NF（正規化） | スタースキーマ（非正規化） |
| 初期コスト | 高（全社モデルが先） | 低（部分的に開始可能） |
| 初期デリバリー | 遅い（全社設計完了まで） | 速い |
| クエリ性能 | 中程度（ジョイン多） | 高い（ジョイン少） |
| データ整合性 | 3NF で保証（高い） | 共有ディメンションで管理 |
| 適する規模 | 大規模エンタープライズ | 中小規模から段階的に |
| 変化への対応 | 難しい（EDW の変更が波及） | 比較的容易 |

### 適用シナリオ

Inmon アプローチが適するケース:
- データ整合性・コンプライアンスが最重要の金融・医療等
- 全社横断の複雑な分析要件がある大規模組織
- データガバナンスを中央集権的に管理したい場合
- 長期的な EDW 投資が可能な組織

## ポイント

- 「全社を1つの真実のソースで統合」するために，EDW を 3NF で先に設計する
- 3NF により整合性・冗長性排除は達成できるが，クエリ複雑性とデリバリー速度がトレードオフ
- CIF（Corporate Information Factory）は ODS → EDW → データマートの3層フレームワーク
- 大規模エンタープライズ・コンプライアンス重視の組織に向く
- [[Kimballのボトムアップ]] のスタースキーマが主流化した現在も，金融・医療領域では Inmon 方式が採用される
