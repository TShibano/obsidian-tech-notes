## 概要

プロセス設計とは、システム開発においてシステム化の対象となる業務や作業を明確にした上で、最適な作業の流れ（プロセス）を定義する活動である。要件定義の前工程として位置づけられ、「何をシステム化するか」を決める前に「業務がどう動いているか・どう動くべきか」を整理することを目的とする。

## 詳細

### プロセス設計の位置づけ

システム開発の上流工程において、プロセス設計は以下のように位置づけられる：

```
[経営戦略・業務課題の把握]
         ↓
[業務設計 / プロセス設計]  ← ここ
         ↓
[要件定義（機能・非機能要件の洗い出し）]
         ↓
[基本設計 → 詳細設計 → 実装 → テスト]
```

プロセス設計を要件定義の前に行うことで、「本当に必要な機能は何か」を業務フローから導き出せる。逆にプロセス設計を省略すると、現場の実際の業務と乖離したシステムができたり、仕様変更が頻発する原因になる。

### プロセス設計の内容

プロセス設計では以下を明確にする：

1. **成果物の定義**: どのようなアウトプット（成果・データ・書類）を出すか
2. **必要な作業の洗い出し**: 成果を出すために必要な業務・作業を列挙する
3. **作業の順序・依存関係**: 誰が・いつ・何をするかを整理し、図示する
4. **例外・エラー処理の考慮**: イレギュラー発生時のフローを定める

### PFD（プロセス・フロー・ダイアグラム）

PFD は、インプット・作業・アウトプット（成果物）の3要素を使ってプロセスを表現する図式手法。「ある作業にどのようなインプットが必要で、どんな成果物が生まれるか」という繋がりを可視化する。

- 派生開発の現場で広く活用される
- 開発プロセスの形骸化を防ぎ、属人化を排除する効果がある
- 複雑なプロセスも構造化して表現できる

### プロセス設計の手法

| 手法 | 概要 |
|------|------|
| **業務フロー図** | 業務の流れを図示する（スイムレーン図が代表的） |
| **PFD（プロセスフローダイアグラム）** | インプット・作業・成果物の繋がりを図示 |
| **BPMN（Business Process Model Notation）** | 国際標準のビジネスプロセス記法 |
| **ユースケース** | アクターとシステムのインタラクションを記述。業務フローからユースケースを抽出する手法（RDRA 等）もある |

### プロセス設計とシステム要件への橋渡し

業務フローを分析することで、システム化が必要な箇所（自動化・データ連携・画面設計のトリガー）が明確になる：

1. 業務フロー上でシステムが関与するアクションを特定
2. それらのアクションからユースケースを抽出
3. ユースケースを機能要件として要件定義書に反映

## ポイント

- プロセス設計は「要件定義のその前に」行うべき工程。省略すると要件漏れや仕様変更の遠因になる
- 「出すべき成果を定める→必要な仕事を考える→図示する」の3ステップが基本
- PFD はシンプルながら開発プロセスの可視化・形骸化防止に有効
- 業務フロー図からユースケースを抽出することで、要件定義への自然な橋渡しができる
- 現場の担当者へのヒアリングが不可欠。管理者には見えていない細かい作業を把握することが重要
