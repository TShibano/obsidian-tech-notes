## 概要

データ指向プログラミング（DOP/DOD: Data-Oriented Programming / Design）は、「データ」を設計の中心に据えるプログラミングパラダイムの総称。**DOP**（Yehonathan Sharvit 提唱）はコードとデータの分離・不変データ構造によるソフトウェア複雑性の削減を目標とし、**DOD**（Mike Acton らが提唱するゲーム開発文脈のデータ指向設計）はCPUキャッシュ効率を最大化するためにデータレイアウトを最適化する。両者は「データを主役にする」という方向性を共有しつつ、目的と文脈が異なる。

## 詳細

### 2種類の「データ指向」

| 概念 | 提唱者 | 目的 | 主な対象 |
|------|--------|------|----------|
| **DOP** (Data-Oriented Programming) | Yehonathan Sharvit | ソフトウェア複雑性の削減 | 汎用アプリ・バックエンド |
| **DOD** (Data-Oriented Design) | Mike Acton / Noel Llopis | CPU キャッシュ効率・性能最適化 | ゲーム開発・高性能システム |

### DOP（Data-Oriented Programming）

#### 概要

Yehonathan Sharvit が著書 *Data-Oriented Programming*（Manning, 2022）で体系化したパラダイム。Clojure の設計思想をベースに、OOP の複雑さ（ミュータブルな状態・継承階層・密結合）を解消することを目的とする。

#### 4 原則

##### 原則 1: コードとデータを分離する（Separate code from data）

OOP ではデータ（フィールド）とロジック（メソッド）をクラスにまとめるが、DOP ではデータを保持する構造体とロジックを担う関数を明確に分離する。

```javascript
// OOP スタイル（コードとデータが混在）
class User {
  constructor(name, age) {
    this.name = name;
    this.age = age;
  }
  isAdult() { return this.age >= 18; }
}

// DOP スタイル（コードとデータを分離）
const user = { name: "Alice", age: 20 };  // データ（純粋）

const isAdult = (user) => user.age >= 18;  // コード（純粋関数）
```

##### 原則 2: 汎用データ構造でデータを表現する（Represent data with generic data structures）

カスタムクラスの代わりに、マップ（連想配列）やリストなどの汎用データ構造を使用する。データの形を特定クラスに縛らないため、柔軟性・相互運用性が高まる。

```clojure
;; Clojure での例（マップで表現）
(def user {:name "Alice" :age 20 :email "alice@example.com"})

;; フィールドの追加も自由
(def user-with-role (assoc user :role "admin"))
```

Java 21 の `record` はこの考えに近い（不変・コードとデータの分離）：

```java
// Java 21 record
record User(String name, int age) {}

// パターンマッチングと組み合わせ
if (obj instanceof User(var name, var age) && age >= 18) {
    System.out.println(name + " is adult");
}
```

##### 原則 3: データは不変（Data is immutable）

データを変更する代わりに、新しいデータを作成する。並行処理時のロックが不要になり、バグの追跡も容易になる。

```javascript
// ミュータブル（DOP では避ける）
user.age = 21;

// イミュータブル（DOP 推奨）
const updatedUser = { ...user, age: 21 };  // スプレッド構文で新オブジェクト
```

##### 原則 4: データスキーマとデータ表現を分離する（Separate data schema from data representation）

データの「形（スキーマ）」と「中身（データ）」を別々に管理する。JSON Schema や Clojure の Spec・Malli などで実現する。

```javascript
// JSON Schema でスキーマを別定義
const userSchema = {
  type: "object",
  properties: {
    name: { type: "string" },
    age:  { type: "integer", minimum: 0 }
  },
  required: ["name", "age"]
};

// データはシンプルな plain object
const user = { name: "Alice", age: 20 };
```

#### DOP のメリット・デメリット

| 観点 | メリット | デメリット |
|------|----------|-----------|
| テスト性 | 純粋関数・不変データで容易 | スキーマ検証の実装コスト |
| 並行処理 | ロック不要（不変データ） | パフォーマンスへの注意が必要 |
| 柔軟性 | 汎用データ構造で拡張が容易 | 型安全性の欠如（TypeScript等で補完可能） |
| 学習コスト | 4 原則はシンプル | OOP 的直感との乖離 |

### DOD（Data-Oriented Design）

#### 概要

Mike Acton（Insomniac Games）が 2014 年 CppCon で発表した、ゲームエンジン・高性能計算向けの設計手法。「問題はデータにある」を基本哲学に、CPU キャッシュミスを最小化するためにデータのメモリレイアウトを最適化する。

#### キャッシュミスの問題

現代の CPU はキャッシュから高速にデータを読み出すが、バラバラに配置されたデータ（OOP のオブジェクト参照など）はキャッシュミスを多発させ、性能を大幅に低下させる。

```
OOP のメモリレイアウト（キャッシュミス多発）:
[Ref→Entity1] [Ref→Entity2] [Ref→Entity3]
       ↓               ↓               ↓
[pos][vel][hp][pos][vel][hp][pos][vel][hp]  ← 不連続

DOD のメモリレイアウト（キャッシュフレンドリー）:
positions[] = [pos1, pos2, pos3, ...]  ← 連続
velocities[] = [vel1, vel2, vel3, ...] ← 連続
```

位置データだけを処理する場合、DOD では連続したメモリ領域から読み出せるため、キャッシュヒット率が大幅に向上する。

#### ECS（Entity Component System）

DOD の代表的なアーキテクチャパターン。

- **Entity**: 一意の ID（データの入れ物ではない）
- **Component**: 特定の種類のデータ（位置・速度・HP など）の純粋なデータ構造
- **System**: Component を操作するロジック（ループ処理）

```rust
// Bevy（Rust の ECS ゲームエンジン）の例
#[derive(Component)]
struct Position { x: f32, y: f32 }

#[derive(Component)]
struct Velocity { dx: f32, dy: f32 }

// System: Position を持つすべての Entity を更新
fn movement_system(mut query: Query<(&mut Position, &Velocity)>) {
    for (mut pos, vel) in &mut query {
        pos.x += vel.dx;
        pos.y += vel.dy;
    }
}
```

| プロジェクト | 説明 |

| Unity DOTS | Unity の Data-Oriented Technology Stack（ECS + Job System + Burst Compiler） |
| Bevy | Rust 製のゲームエンジン（ECS ファースト設計） |

| Insomniac Games | Spider-Man シリーズなど（Mike Acton の所属先） |

### DOP vs DOD vs OOP vs FP の比較

| 観点 | OOP | FP | DOP | DOD |
|------|----|-----|-----|-----|
| 中心概念 | オブジェクト | 関数 | データ（汎用構造） | データレイアウト |
| 状態管理 | ミュータブル | イミュータブル | イミュータブル | データ分離 |
| 主な目的 | コード再利用・モデル化 | 副作用の排除 | 複雑性の削減 | キャッシュ効率・性能 |
| 主な用途 | ビジネスアプリ | データ変換・並行処理 | バックエンドAPI | ゲーム・高性能計算 |
| 型安全性 | クラス型 | 代数的データ型 | スキーマ分離 | 低レベル制御 |

## ポイント

- **DOP（Sharvit）**: コードとデータを分離し、不変の汎用データ構造でソフトウェアの複雑性を削減するパラダイム
- **DOD（Acton）**: CPU キャッシュ効率を最大化するためにデータレイアウトを設計するアプローチ（ゲーム開発が源流）
- DOP の 4 原則は Clojure の設計思想に由来し、Java・JavaScript などの主流言語でも適用できる
- Java 21 の `record` とパターンマッチングは DOP を実践しやすい機能
- DOD の ECS パターンは Unity DOTS や Rust の Bevy エンジンで採用されている実用的なアーキテクチャ
- FP（不変データ）と DOP（コード・データ分離・不変）は思想的に近く、補完的な関係
