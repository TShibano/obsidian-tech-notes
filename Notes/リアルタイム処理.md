## 概要

リアルタイム処理（Real-time Processing / ストリーミング処理）は，データが発生した瞬間またはそれに近いタイミングで即座に処理するデータ処理方式である．センサーデータ・ログ・金融取引・ユーザー行動など，継続的に発生するストリームデータを対象とし，低レイテンシなインサイトとアクションを実現する．

## 詳細

### バッチ処理との比較

| 観点 | リアルタイム処理 | バッチ処理 |
|------|---------------|----------|
| 処理タイミング | データ到着次第，即時 | 定時（日次，時次など） |
| レイテンシ | ミリ秒〜秒 | 分〜時間 |
| スループット | 低〜中（逐次） | 高い（大量一括） |
| 実装複雑さ | 高い（状態管理等） | 低い |
| 向いているデータ | センサー，ログ，取引 | 大量履歴データ，集計 |
| 障害復旧 | 複雑（チェックポイント） | 容易（再実行） |

### ストリーミング処理の仕組み

```
データソース（IoT/ログ/取引）
        ↓ イベント生成
メッセージブローカー（Apache Kafka 等）
        ↓ ストリーム配信
ストリーミング処理エンジン（Flink/Spark Streaming）
  ├─ ウィンドウ集計（時間窓/カウント窓）
  ├─ フィルタリング
  ├─ エンリッチメント（静的データとの結合）
  └─ 状態管理（ステートフル処理）
        ↓ 処理済みイベント
出力先（DB，DWH，アラート，ダッシュボード）
```

### ウィンドウ処理

ストリーミングデータの集計は「ウィンドウ」（時間範囲）で行う:

| ウィンドウ種別 | 説明 | 例 |
|-------------|------|-----|
| **タンブリング** | 重複のない固定幅の時間窓 | 1分ごとの集計 |
| **スライディング** | 一定間隔でずらしながら集計 | 5分ウィンドウを1分ごとに更新 |
| **セッション** | アクティビティの間隔でグループ化 | ユーザーセッション単位の集計 |

### 主要フレームワーク（2026）

| ツール | 特徴 |
|-------|------|
| **Apache Kafka** | 分散メッセージブローカー．高スループット・低レイテンシ |
| **Apache Flink** | ステートフルストリーミング処理のデファクト．正確な1回処理保証 |
| **Apache Spark Streaming** | マイクロバッチ方式のストリーミング．バッチ処理と同じコードで書ける |
| **Apache Kafka Streams** | Kafka 内で処理が完結する軽量ストリーミングライブラリ |
| **AWS Kinesis** | AWS ネイティブのストリーミングサービス |
| **Google Pub/Sub + Dataflow** | GCP ネイティブのストリーミングパイプライン |

### [[Kappaアーキテクチャ]] での位置づけ

[[Kappaアーキテクチャ]] はすべてのデータ処理をストリーミングで統一することで [[Lambdaアーキテクチャ]] の複雑性（バッチ＋ストリーミングの二重管理）を解消する．2026年現在，「ストリーミングファースト設計」への移行が加速している．

## ポイント

- 低レイテンシなインサイト・アラート・オペレーションに必須の処理方式
- Kafka + Flink の組み合わせが2026年の標準スタックとして定着
- バッチと比べて実装・運用コストが高いため，リアルタイム性の要否を慎重に判断する
- ウィンドウ処理と状態管理の設計がストリーミング品質の鍵
- [[バッチ処理]] と組み合わせた [[Lambdaアーキテクチャ]]，または完全ストリーミングの [[Kappaアーキテクチャ]] で設計する
