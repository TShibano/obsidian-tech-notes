## 概要

業務設計とは、企業・組織の業務プロセスを分析・可視化し、効率的かつ品質の高い業務フローを設計・最適化する活動である。システム開発文脈では「要件定義の前段工程」として位置づけられ、システム化すべき業務の範囲・流れ・ルールを明確にすることで、要件定義・システム設計の精度を高める役割を担う。

## 詳細

### 業務設計の目的

業務設計に取り組む主な目的：

1. **業務の可視化**: 「誰が・何を・いつ・どのように」行うかを整理し、暗黙知・属人化を排除
2. **効率化（3M の排除）**: ムリ（過負荷）・ムダ（不要な作業）・ムラ（品質のばらつき）を特定・改善
3. **システム化範囲の明確化**: どの業務をシステムで自動化・支援するかを決定
4. **業務標準化**: 組織全体で一貫した業務遂行を可能にする

### 業務設計のプロセス

一般的な業務設計は以下の流れで進める：

#### As-Is 分析（現状把握）

1. 現行業務に関わる情報を収集
2. 現場担当者へのヒアリング（管理者には見えない細かい作業を把握）
3. 現状の業務フロー（As-Is フロー）を作成
4. 問題点・課題・改善の余地を特定

#### To-Be 設計（あるべき姿の設計）

1. 理想的な業務フロー（To-Be フロー）を設計
2. ECRS フレームワークで改善案を検討
3. システム化する箇所・人手で行う箇所を切り分け
4. 新しい業務フローをドキュメント化・承認

### ECRS フレームワーク

業務改善案を検討する際に活用するフレームワーク：

| 頭文字 | 意味 | 例 |
|--------|------|-----|
| **E**liminate（排除） | 不要な作業・ステップを廃止 | 重複した承認フローの廃止 |
| **C**ombine（統合） | 複数の作業を1つにまとめる | 2つの入力フォームを統合 |
| **R**earrange（再配置） | 作業の順序・担当者を変更 | 上流で品質チェックを行い手戻りを削減 |
| **S**implify（簡素化） | 作業の複雑さを減らす | 入力項目の削減・自動化 |

### CAPDo サイクル

業務設計では、新しい取り組み向けの PDCA ではなく、既存業務が前提の **CAPDo** サイクルを用いる：

```
Check（現状把握）→ Analyze（分析）→ Plan（計画）→ Do（実行）
         ↑___________________________________|
```

1. **Check**: 現状の業務を数値・データで評価
2. **Analyze**: 問題の原因を分析
3. **Plan**: 改善計画を立案
4. **Do**: 改善を実施（そして次の Check へ）

### 業務フロー図の種類

業務を可視化する主な図式手法：

| 図 | 特徴 |
|----|------|
| **スイムレーン図** | 担当者・部署ごとに役割を「レーン」で分割して表現 |
| **フローチャート** | 処理の流れを記号で表現（汎用性が高い） |
| **BPMN** | ビジネスプロセスの国際標準記法（詳細かつ形式的） |
| **業務フロー（和文）** | 日本企業で慣習的に使われる帳票・業務フロー形式 |

### スイムレーン図の例（システム開発文脈）

```
┌─────────────┬─────────────┬─────────────┐
│  ユーザー    │   営業担当   │  システム   │
├─────────────┼─────────────┼─────────────┤
│ 注文入力    │             │             │
│      ↓      │             │             │
│    ────────────────→      │             │
│             │ 受注確認     │             │
│             │      ↓      │             │
│             │    ─────────────────→    │
│             │             │ 在庫確認    │
│             │             │      ↓      │
│             │    ←─────────────────    │
│             │ 発注指示     │             │
└─────────────┴─────────────┴─────────────┘
```

### システム開発における業務設計の位置づけ

```
[経営戦略・業務課題]
       ↓
[業務設計（As-Is 分析 → To-Be 設計）]
       ↓
[プロセス設計（業務フロー・PFD の作成）]
       ↓
[要件定義（機能要件・非機能要件の洗い出し）]
       ↓
[基本設計 → 詳細設計 → 実装]
```

業務設計とプロセス設計は密接に関連し、業務設計が「業務全体の最適化」を担うのに対し、プロセス設計はシステム開発の文脈で「システム化する処理の流れの定義」に重点を置く。

## ポイント

- 業務設計の出発点は「現状把握（As-Is）」であり、現場担当者へのヒアリングが不可欠
- ECRS フレームワークで「廃止→統合→再配置→簡素化」の順に改善策を検討する
- 業務設計なしに要件定義を行うと、現場の実態と乖離したシステムができる原因になる
- To-Be フロー設計で「人が行う業務」と「システムが行う業務」の境界を明確にすることが要件定義の精度を上げる
- 業務の可視化（フロー図の作成）だけで問題が明確になることも多い
