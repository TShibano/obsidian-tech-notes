# Vault Lint（健全性チェック）

Notes/ を中心に Vault 全体の整合性を検査し、問題を報告・修正します。
LLM Wiki のメンテナンス（Lint）パターンに基づいた定期実行を想定しています。

## 使用方法

```
/lint [モード] [カテゴリ]
```

- `モード`: `report`（デフォルト）または `fix`
- `カテゴリ`: 特定のチェックのみ実行（省略時は全チェック）

**例:**
- `/lint` → 全チェックを実行し、レポートのみ出力
- `/lint fix` → 全チェックを実行し、自動修正可能なものを修正
- `/lint links` → リンク関連のチェックのみ実行
- `/lint fix frontmatter` → frontmatter のチェック＋自動修正

## チェックカテゴリ

### 1. `links` — リンク整合性

#### 1.1 related の非対称検出

Notes/ 内の全ノートを走査し、`related` frontmatter の双方向リンクが対称であるかを検査する。

- A の `related` に `[[B]]` があるが、B の `related` に `[[A]]` がない → **非対称**
- B のファイル自体が存在しない場合は「壊れたリンク」（1.2）として扱う

**fix モード**: B の `related` に `[[A]]` を追加し、「関連項目」セクションにも追記する。

#### 1.2 壊れたリンク検出

Notes/ 内の全ノートの本文と `related` から `[[ノート名]]` を抽出し、リンク先が Notes/ に存在するかを検査する。

- Notes/ に存在しない → **壊れたリンク**として報告
- ただし、Books/ や他フォルダのノートへのリンクは除外する

**fix モード**: 自動修正しない（ファントムリンクの可能性があるため）。レポートに候補リストを出力し、ユーザーに判断を委ねる。

#### 1.3 孤立ノート検出

Notes/ 内で、他のどのノート（Notes/ および Books/）からも `[[ノート名]]` でリンクされていないノートを検出する。

**fix モード**: 自動修正しない。孤立ノートの一覧と、内容に基づくリンク候補を提示する。

### 2. `frontmatter` — Frontmatter 検査

#### 2.1 必須フィールドの欠落

Notes/ 内の全ノートの frontmatter に以下の必須フィールドがあるかを検査:

- `title`: 存在するか
- `date`: 存在するか、`YYYY-MM-DD` 形式か
- `tags`: 存在するか、1つ以上のタグがあるか
- `related`: 存在するか

**fix モード**:
- `title` 欠落 → ファイル名から補完
- `date` 欠落 → ファイルの git 初回コミット日、または本日の日付で補完
- `tags` / `related` 欠落 → 空配列を追加（`tags:\n  -`）

#### 2.2 タグ体系の逸脱

`tags` の値が以下のタグ体系に含まれるかを検査:

| 親タグ | サブタグ例 |
|--------|-----------|
| `AI` | `LLM`, `ML`, `Agent`, `NLP`, `ComputerVision` |
| `Web` | `Frontend`, `Backend`, `API`, `Framework` |
| `Cloud` | `AWS`, `GCP`, `Azure`, `K8s`, `Serverless` |
| `DevOps` | `CI/CD`, `IaC`, `Monitoring` |
| `Security` | `Auth`, `Encryption`, `ZeroTrust` |
| `DB` | `SQL`, `NoSQL`, `VectorDB` |
| `Language` | `Python`, `TypeScript`, `Go`, `Rust` |
| `Tool` | `Git`, `Editor`, `CLI` |
| `DataEngineering` | `ETL`, `DataWarehouse`, `DataLake`, `Pipeline` |
| `Architecture` | `Design`, `Pattern`, `Microservices` |
| `Management` | `Agile`, `Scrum`, `ProjectManagement` |

- 上記に含まれないタグ → **未知のタグ**として報告

**fix モード**: 自動修正しない。未知タグの一覧と、推奨される代替タグを提示する。

### 3. `content` — コンテンツ品質

#### 3.1 空セクション検出

Notes/ 内のノートで、テンプレートの placeholder がそのまま残っているセクションを検出:

- `（このトピックの簡潔な説明を1-3文で記述する）` が残っている
- `（トピックの詳細な解説を記述する）` が残っている
- セクション見出しの直後に内容がない（見出しの次の行が空行で、その次が別の見出しまたはファイル末尾）

**fix モード**: 自動修正しない。空セクションの一覧を報告する。

#### 3.2 Books 未展開トピック検出

Books/ 内の全ファイルから `[[ノート名]]` リンクを抽出し、Notes/ に対応するノートが存在しないものを検出する。
これは `/compile-book` で展開すべき候補の発見に使える。

また、Books/ ファイルに「コンパイル実施」記録がない場合、未コンパイルの書籍として報告する。

**fix モード**: 自動修正しない。未展開トピックと未コンパイル書籍の一覧を報告する。

## 処理手順

### 1. 引数の解析

`$ARGUMENTS` を解析:

- `fix` が含まれる → fix モード（修正を実行）
- `report` が含まれる、または指定なし → report モード（レポートのみ）
- `links`, `frontmatter`, `content` が含まれる → そのカテゴリのみ実行
- カテゴリ指定なし → 全カテゴリを実行

**例:**
- `(空)` → report モード、全カテゴリ
- `fix` → fix モード、全カテゴリ
- `links` → report モード、links のみ
- `fix frontmatter` → fix モード、frontmatter のみ

### 2. Notes/ の全ファイル走査

Notes/ 配下の全 `.md` ファイルを読み取り、以下を解析する:

- frontmatter（YAML）
- 本文中の `[[ノート名]]` リンク
- セクション構造

### 3. 指定カテゴリのチェック実行

各チェック項目を実行し、問題を収集する。

### 4. レポート出力

検出された問題をカテゴリ別に整理して出力する:

```
## Vault Lint レポート — YYYY-MM-DD

### サマリー
- 検査ノート数: N 件
- 問題検出: M 件（うち自動修正可能: K 件）

### 🔗 links
#### related の非対称: X 件
| ノート | 欠落している逆リンク | 修正 |
|--------|---------------------|------|
| [[A]] | → [[B]] の related に [[A]] がない | ✅ 修正済 / ⚠️ 要修正 |

#### 壊れたリンク: X 件
| ノート | リンク先 | 備考 |
|--------|---------|------|
| [[A]] | [[存在しないノート]] | ファントムリンクの可能性 |

#### 孤立ノート: X 件
| ノート | リンク候補 |
|--------|-----------|
| [[A]] | 内容から [[B]], [[C]] が関連しそう |

### 📋 frontmatter
#### 必須フィールド欠落: X 件
| ノート | 欠落フィールド | 修正 |
|--------|---------------|------|
| [[A]] | title, date | ✅ 修正済 / ⚠️ 要修正 |

#### 未知のタグ: X 件
| ノート | 未知タグ | 推奨タグ |
|--------|---------|---------|
| [[A]] | `unknown_tag` | `AI` ? |

### 📝 content
#### 空セクション: X 件
| ノート | 空セクション |
|--------|-------------|
| [[A]] | 概要, 詳細 |

#### Books 未展開トピック: X 件
| 書籍 | 未展開トピック |
|------|--------------|
| データエンジニアリングの基礎 | [[トピック1]], [[トピック2]] |

#### 未コンパイル書籍: X 件
| 書籍 | 備考 |
|------|------|
| 書籍名.md | コンパイル記録なし |
```

### 5. fix モードの場合: 修正の実行

自動修正可能な項目を修正する:

- related の非対称 → 逆リンクを追加
- frontmatter 必須フィールド欠落 → デフォルト値で補完

修正後、レポートの「修正」列を「✅ 修正済」に更新する。

### 6. fix モードの場合: コミットとプッシュ

修正が1件以上ある場合:

1. 修正したファイルを `git add` でステージング
2. 以下の形式でコミットメッセージを作成し `git commit` する:
   - 形式: `lint修正(YYYY/MM/DD): 修正内容の要約`
   - 例: `lint修正(2026/04/15): related非対称5件修正, frontmatter補完3件`
3. `git push` でリモートにプッシュ

## エラーハンドリング

- Notes/ が空の場合: 「ノートがありません」と報告して終了
- frontmatter のパースに失敗した場合: そのファイルを「frontmatter 不正」として報告し、他のファイルの処理を継続
- 個別チェックの失敗: エラーをログに記録し、他のチェックを継続
