# Book コンパイル

Books/ の読書メモを解析し、トピック単位で Notes/ に知識を展開します。
LLM Wiki の「コンパイル」パターンに基づき、1冊の書籍メモから複数の技術ノートを横断的に作成・更新します。

## 使用方法

```
/compile-book <Books/のファイルパス> [対象章の指定]
```

- `ファイルパス`: Books/ 配下のマークダウンファイル（例: `Books/データエンジニアリングの基礎.md`）
- `対象章の指定`: 省略時は全章を処理。指定時はその章のみ処理（例: `3章`, `1-3章`）

**例:**
- `/compile-book Books/データエンジニアリングの基礎.md` → 全章を処理
- `/compile-book Books/データエンジニアリングの基礎.md 3章` → 3章のみ処理
- `/compile-book Books/データエンジニアリングの基礎.md 1-3章` → 1〜3章を処理

## 処理手順

### 1. 引数の解析

`$ARGUMENTS` を解析:

- 第1引数: Books/ 配下のファイルパス（必須）
- 第2引数以降: 対象章の指定（省略可）

ファイルが存在しない場合はエラーメッセージを表示して終了する。

### 2. 書籍メモの読み取りと解析

Books/ のマークダウンファイルを読み取り、以下を抽出する:

- **書籍メタ情報**: URL、タイトル（ファイル名から推定）
- **章構造**: 各章の見出しと内容
- **既存リンク**: 本文中の `[[ノート名]]` リンク
- **キーコンセプト**: 各章で扱われている主要トピック

### 3. トピックマッピングの作成

各章の内容を分析し、Notes/ のトピックにマッピングする:

#### 3.1 マッピングルール

各章から以下の基準でトピックを抽出する:

- 書籍メモ内の `[[ノート名]]` リンク → そのままトピック候補
- 章の主題となっている概念 → トピック候補
- 繰り返し言及される重要な技術用語 → トピック候補

#### 3.2 既存ノートの確認

各トピック候補について:

- `Notes/{トピック名}.md` が存在する → **更新対象**（既存内容に書籍の知見を統合）
- 存在しない → **新規作成対象**

#### 3.3 マッピング結果の提示

処理を開始する前に、以下のマッピング結果をユーザーに提示して確認を取る:

```
## コンパイル計画: {書籍名}

### 更新対象（既存ノート）: X 件
- [[トピック1]] ← 3章: {該当する内容の要約}
- [[トピック2]] ← 1章, 5章: {該当する内容の要約}

### 新規作成: Y 件
- トピック3 ← 2章: {該当する内容の要約}
- トピック4 ← 4章: {該当する内容の要約}

### スキップ（情報量が少ない）: Z 件
- トピック5（言及のみで詳細なし）

実行しますか？（修正があれば指示してください）
```

ユーザーの承認を得てから次のステップに進む。

### 4. ノートの作成・更新

承認されたマッピングに従い、各トピックを順次処理する:

#### 4.1 既存ノートの更新

既存ノートに書籍の知見を統合する:

1. 既存ノートの内容を読み取る
2. 書籍メモから該当トピックに関連する情報を抽出する
3. 既存の記述と重複しない新しい知見を特定する
4. 適切なセクションに情報を追記する（既存構造を壊さない）
5. frontmatter の `related` に書籍内の関連トピックへのリンクを追加する（未存在の場合）
6. 「参考」セクションに書籍情報を追加する:
   - 形式: `- 書籍名, 章名`

**重要**: 既存の記述を削除・上書きしない。あくまで補強・追記を行う。

#### 4.2 新規ノートの作成

`Templates/tech-note.md` のテンプレート構造に従って作成する:

```yaml
---
title: "トピック名"
date: YYYY-MM-DD
tags:
  - 親タグ
  - サブタグ
related:
  - "[[関連ノート]]"
---
```

**セクション構成:**
- **概要**: 書籍の記述を基にトピックを簡潔に説明（1-3文）
- **詳細**: 書籍メモの内容を再構成して記述。必要に応じて WebSearch で補足調査する
- **ポイント**: 重要な点を箇条書きで整理
- **関連項目**: 書籍内で関連するトピックへの `[[ノート名]]` リンク
- **参考**: 書籍情報 + WebSearch で得た一次情報源

**WebSearch の利用基準:**
- 書籍メモの情報だけでは概要・詳細が薄い場合に補足調査を行う
- 書籍の出版後に大きな変化があったトピック（最新動向の追記）
- 書籍メモに含まれる情報だけで十分な場合は WebSearch を省略してよい

#### 4.3 タグの選定

以下のタグ体系に基づいて付与する:

| 親タグ | サブタグ例 |
|--------|-----------|
| `AI` | `LLM`, `ML`, `Agent`, `NLP`, `ComputerVision` |
| `Web` | `Frontend`, `Backend`, `API`, `Framework` |
| `Cloud` | `AWS`, `GCP`, `Azure`, `K8s`, `Serverless` |
| `DevOps` | `CI/CD`, `IaC`, `Monitoring` |
| `Security` | `Auth`, `Encryption`, `ZeroTrust` |
| `DB` | `SQL`, `NoSQL`, `VectorDB` |
| `Language` | `Python`, `TypeScript`, `Go`, `Rust` |
| `Tool` | `Git`, `Editor`, `CLI` |

### 5. 双方向リンクの整合性維持

作成・更新した全ノートについて:

1. `related` frontmatter でリンクした既存ノートに逆リンクを追加
2. リンク先ノートの「関連項目」セクションにも `[[ノート名]]` を追加
3. 既に逆リンクが存在する場合は追加しない（重複を避ける）

### 6. Books ファイルへのコンパイル記録

Books/ の元ファイルの先頭（URL の下）に、コンパイル実施記録を追記する:

```markdown
- コンパイル実施: YYYY-MM-DD
  - 展開先: [[トピック1]], [[トピック2]], [[トピック3]], ...
```

これにより、どの書籍がコンパイル済みか、どのノートに展開されたかを追跡できる。

### 7. コミットとプッシュ

1. 新規作成・更新したノートファイル、逆リンクを追加した既存ノート、Books ファイルを `git add` でステージング
2. 以下の形式でコミットメッセージを作成し `git commit` する:
   - 形式: `コンパイル(YYYY/MM/DD): 書籍名 (新規X件, 更新Y件)`
   - 例: `コンパイル(2026/04/15): データエンジニアリングの基礎 (新規5件, 更新3件)`
3. `git push` でリモートにプッシュ

### 8. 結果報告

```
## コンパイル結果: {書籍名}

**新規作成**: X 件
- [[トピック1]] — 概要の1行要約
- [[トピック2]] — 概要の1行要約

**既存更新**: Y 件
- [[トピック3]] — 追記内容の要約
- [[トピック4]] — 追記内容の要約

**スキップ**: Z 件
- トピック5（ユーザー指示 / 情報不足）
```

## エラーハンドリング

- ファイルが存在しない場合: エラーメッセージを表示して終了
- ファイルが Books/ 配下でない場合: 警告を表示し、続行するか確認
- 章の指定が不正な場合: エラーメッセージを表示して終了
- 個別トピックの処理失敗: エラーをログに記録し、次のトピックへ進む
