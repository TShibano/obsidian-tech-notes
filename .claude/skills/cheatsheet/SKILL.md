# チートシート作成

指定されたツールのチートシートを `CheatSheets/` に作成してください。

## 1. 引数の解析

`$ARGUMENTS` を解析してツール名とフォーカス指示を分離する:

- 最初のトークンがツール名（= ファイル名）になる
- ダブルクォートで囲まれている場合はクォート内全体がツール名（例: `"GitHub CLI"` → `GitHub CLI`）
- ツール名以降のテキストはフォーカス指示として扱う（特定サブコマンドや用途に絞る等）
- フォーカス指示がなければ主要コマンド全般を網羅する

**例:**
- `git` → ツール: git, フォーカス: なし
- `docker compose の使い方中心` → ツール: docker, フォーカス: compose の使い方
- `"GitHub CLI" PR操作` → ツール: GitHub CLI, フォーカス: PR操作

## 2. 既存チートシートの確認

- `CheatSheets/{ツール名}.md` が既に存在するか確認する
- 存在する場合はユーザーに **上書き / マージ / キャンセル** を提示する

## 3. 調査

優先順位:

1. **公式ドキュメント / 公式 CLI ヘルプ** （最優先）
2. context7 MCP でライブラリ/CLI のドキュメントを取得（該当する場合）
3. WebSearch で補完（主要コマンド・典型ユースケース・最新バージョン）

収集する情報:

- ツールの一行説明、公式ドキュメントURL、最新安定版バージョン、ライセンス
- カテゴリ別の主要コマンド（サブコマンド・オプション・キーバインド等）
- よく使うワークフロー / レシピ
- 落とし穴・Tips

## 4. チートシートの作成

`Templates/cheatsheet.md` の構造に従って作成する。日本語で記述するが、コマンド・キー名・オプション名は原文（英語）のまま残す。

### 構成のガイドライン

- セクションは **ツールの性質に合わせて柔軟に** 設定する
  - CLI ツール: `Basic`, `Branching`, `Remote` 等の機能カテゴリ別
  - エディタ: `Movement`, `Editing`, `Search` 等のキーバインド別
  - ライブラリ/フレームワーク: `Setup`, `Routing`, `State` 等
- **テーブル形式** を基本とし、左カラムにコマンド/キー、右カラムに簡潔な説明を書く
- セクションが多い場合は冒頭に概要を blockquote で添える
- `Workflows / Recipes` には実際のユースケースを `<details>` で折りたたみ、コメント付きコマンド列を載せる
- `See Also` には Vault 内の関連ノート（`[[ノート名]]`、ファントムリンク可）と外部リンクを混在させてよい

### Frontmatter

```yaml
---
title: "{ツール名} Cheatsheet"
date: YYYY-MM-DD
tags:
  - Cheatsheet
  - {親タグ}
  - {サブタグ}
related:
  - "[[関連ノート]]"
---
```

タグは `Cheatsheet` を必ず含め、加えて `CLAUDE.md` のタグ体系（`Tool`, `Language`, `Web`, `DevOps` 等）から親+サブを1〜2個付与する。

## 5. 双方向リンクの整合性維持

`See Also` で Vault 内の既存ノート（`Notes/` や他の `CheatSheets/`）にリンクした場合、リンク先の `related` frontmatter と関連項目セクションに逆リンクを追加する。重複追加は避ける。

## 6. ファイルの保存と報告

- `CheatSheets/{ツール名}.md` として保存する
- 保存完了後、作成したチートシートの構成（セクション名一覧）を簡潔に報告する

## 7. コミットとプッシュ

1. 対象ファイル（`CheatSheets/` 配下と逆リンク追加した既存ノート）を `git add`
2. コミットメッセージ形式: `チートシート: ツール名`
3. `git push` でリモートにプッシュ
