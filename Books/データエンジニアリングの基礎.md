# メモ

## 1. データエンジニアリング概説
- データエンジニアリングの定義
	- 生データを取り込み，下流の分析や機械学習などで利用しやすい高品質で一貫性のある情報を生成するシステムとプロセスの開発，実装，維持管理を意味する
- データエンジニアリングライフサイクル      
	- 生成 -> 保存 -> 取込 -> 変換 -> 提供
	- 底流(undercurrents)
		- セキュリティ，データ管理，DataOps，データアーキテクチャ，オーケストレーション，ソフトウェアエンジニアリング
- データエンジニアの歴史
	- データウェアハウスからWebへ(1980~2000)
		- RDB(リレーショナルデータベース)にデータを溜めてBIツールで活用
		- データエンジニアリングの先駆け
		- 徐々にWebでデータが生成されるようになる
	- 現代的なデータエンジニアリングの誕生(2000年代初頭)
		- データの爆発的な増加(ビッグデータ; データの3V; Velocity, Variety, Volume)により，従来のモノシリックサービスの分解と非集中化
		- GoogleによるMapReduce(スケーラブルなデータ処理)やパブリッククラウドの発展により，今日のデータエコシステムの基礎が成立
	- ビッグデータエンジニアリング(2000~2010年代)
		- [[Apache Hadoop]]や[[Apache Hive]]，[[Apache Spark]]などの低レベルのツールを駆使してデータの管理を行なった
	- データライフサイクルのエンジニアリング(2020年代)
		- ツールの方向性が変わり，非集中化・モジュール化・マネージド・高度に抽象化されたツールへと移行しつつある
			- https://www.mattturck.com/mad2025
		- 高度な抽象化と単純化によって，細部にまで細心の注意を払う必要が減り，バリューチェーンのより上位に焦点を当てるようになった
- データエンジニアリングとデータサイエンス
	- データサイエンティストは，AIモデルの構築や活用に重きを置きたいが，基盤がない状態では，データ収集や前処理などに非常に時間がかかってしまう．
	- データサイエンティストがすぐに価値を出せるように，データエンジニアがデータ基盤を整える必要がある．
	- https://medium.com/@tfreundp/evolving-the-ai-hierarchy-of-needs-for-modern-systems-4ed52be952dc

## 2.データエンジニアリングライフサイクル

### 2.1 データエンジニアリングライフサイクルとは何か？？
- 本章では，重要な概念を紹介し，詳細は別の章で述べられている
データエンジニアリングライフサイクル
- 以下の5つからなる．ただし，保存は，取込・変換・提供にまたがって行われる
	- 生成
	- 保存
		- 取込
		- 変換
		- 提供
- この全ての工程で，セキュリティ・データ管理・DataOps・データアーキテクチャ・オーケストレーション・ソフトウェアエンジニアリングが底流として機能している
- 生成: ソースシステム
	- データが生成されるシステム
	- ソースシステムを評価する(システムの動作，生成速度・頻度，スキーマなど)ことは非常に重要である
	- なお，直接制御できることが少ない(突然のシステムダウンなど)ため，知らぬ間にデータの品質が低下することがある
- 保存: ストレージ
	- データをストレージに保存すること．
	- データエンジニアリングライフサイクルすべてに影響を与えるので，慎重に検討すべきである
- 取込
	- ソースのデータをストレージに保存する
	- ボトルネックになりやすく，ソースとストレージから十分に考慮する必要がある
- 変換
	- データを適切に変換する
	- 正しく意味のある変換をすることで，下流のデータ活用時に価値を生む
	- 提供するユーザによって変換方法は変わる
- 提供
	- ユーザにデータを提供し，活用してもらう
	- ユーザがデータを使って価値を生み出して初めて意味がある．

### 2.2 データエンジニアリングにおける主要な底流
- セキュリティ
	- 何よりも重視すべき項目
	- データ・システムへのアクセス権
- データ管理
	- [[DMBOK]]にもデータ管理の項目が記載されており，詳細はそちらを参照すること
	- データガバナンス，発見可能性，メタデータ，データの説明責任，データ品質，データモデリングと設計，データリネージ，データ統合と相互運用性，データライフサイクル管理，倫理とプライバシー
- DataOps
	- 技術的なプラクティス，ワークフロー，文化的な慣習，アーキテクチャのパターンの集合により，より高速に顧客に洞察を提供したり，高品質かつ低エラー率のデータを持ち，様々なコラボレーションが可能になり，得られる価値の測定・監視・透明性がある
	- 自動化・観測と監視・インシデント対応という3つの技術的側面がある
	- DataOpshはまだ発展途上である
	- 初期のうちにコストをかけてでも取り組むことで長期的に大きな価値が得られる．
- データアーキテクチャ
	- 組織の長期的なデータ需要と戦略をサポートするデータシステムの現在と将来の状態を示す
- オーケストレーション
	- 多数のジョブを事前に決められた感覚で可能な限り素早く効率的に動作するように調整すること
	- [[Apache Airflow]]などが代表ツールである
- ソフトウェアエンジニアリング
	- データエンジニアリング業務はかなり抽象化が進んでいるが，まだまだ低レイヤの実装等が必要になることがあり，ソフトウェアエンジニアリングは必要である
	- コアデータ処理コード，オープンソースフレームワークの開発，ストリーミング，コードとしてのインフラストラクチャ([[IaC]], Infrastructure as Code)，コードとしてのパイプライン([[Pipekine as Code]])

## 3. 適切なデータアーキテクチャの設計
### 3.1 データアーキテクチャとは何か？
- [[データアーキテクチャ]]とは，企業の進化するデータへの要求をサポートするシステムの設計であり，トレードオフを慎重に評価した上での柔軟で可逆な決定を通じて実現される
	- データアーキテクチャは，[[エンタープライズアーキテクチャ]]の一つである．
		- 企業内での変化に対応しうるシステム設計であり，慎重なトレードオフの評価を伴う柔軟で可逆な決定によって実現される
### 3.2 良いデータアーキテクチャの原則
- [AWS Well Architected Framework](https://docs.aws.amazon.com/ja_jp/wellarchitected/latest/framework/definitions.html)は6つの柱
	- オペレーショナルエクセレンス
	- セキュリティ
	- 信頼性
	- パフォーマンス効率
	- コスト最適化
	- 持続可能性
- [Google Cloud 5principles](https://cloud.google.com/blog/products/application-development/5-principles-for-cloud-native-architecture-what-it-is-and-how-to-master-it?hl=en)
	- 自動化を前提に設計する
	- 状態をスマートに扱う
	- マネージドサービスを優先する
	- 深い防御を実践する
	- 常に設計し続ける
- 本書の9つの原則
	- 共通コンポーネントを賢く選択する
		- 組織内で広く適用できるものを選ぶ
		- 共通コンポーネントとは，オブジェクトストレージやバージョン管理システム，処理エンジンなど
	- 障害に備える
		- 可用性(availability)
		- 信頼性(reliability)
		- 目標復旧時間(RTO: Recovery Time Objective)
		- 目標復旧時点(RPO: Recovery Point Objective)
	- スケーラビリティ設計
		- 扱うデータ量に応じてスケールアップとスケールダウンができることが重要
	- アーキテクチャはリーダーシップだ
		- データアーキテクトは，深い技術力とリーダーシップを持ち，エンジニアのメンターとなって専門知識を普及しつつ，会社資源を最大限活用して目標を達成する
	- 常に設計し続ける
		- データアーキテクトは，現在の状態(ベースラインアーキテクチャ)を把握しながら，目標の状態(ターゲットアーキテクチャ)を開発し，優先順位と順序を決定する必要がある
		- 現在のアーキテクチャは，時間とともに変化するのが当たり前であり，アジャイル的な開発が必要である．
	- 疎結合システムを構築する
		- [[ベゾスAPI司令]]
		- 技術と人的システムを疎結合することで，部署をまたいでより効率的に協業できるようになる
		- 疎結合システムの特徴
			- 1. システムを多くの小さな部品に分解する
			- 2. 抽象化レイヤを通じて他のサービスと通信する．データベースバックエンドや内部クラス，メソッド呼び出しなどの内部は保護されている
			- 3. 2により，システムコンポーネントの内部的な変更は他の部分の変更を必要としない．コードの更新は安定したAPIの背後に隠されて，各パーツを個別に進化させることができる
				- 4. 3により，システム全体のリリースはなく，各コンポーネントが個別に更新される
	- 可逆な決定をする
		- 変更可能で，可逆で双方向に行き来できる選択を行う
	- セキュリティを優先する
		- データエンジニアはセキュリティエンジニアでもあるべきである．
		- [[ゼロトラストセキュリティ]]
		- [[責任共有セキュリティモデル]]
	- [[FinOps]]を活用する
		- エンジニアリング，財務，テクノロジ，ビジネスの各チームが協力してデータ駆動で支出決定することで，組織が最大のビジネス価値を得ることを可能にする
		- オンプレミス環境では数年に一回，スペックと予算のバランスを考えていた
		- クラウド環境では，従量課金になり支出が動的に変化するため，予算・優先順位・効率を管理する必要がある．
